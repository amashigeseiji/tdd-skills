# /tdd-vocab plan — 初期語彙定義

## 目的

フィーチャー着手前に、このコンテキストのアプリケーションドメイン語彙を `plans/<プラン名>/dictionary.json` に作る。
語彙が先にあることで、tdd-run はこの空間の中でツリーを構成できる。

既存の `docs/dictionary.json` がある場合は、新しいコンテキストを既存 BC の語彙ネットワークに位置づけることが主な目的になる。
同じ言葉が複数の BC に現れる場合、この時点で関係を明確にしておく。

## 手順

problem.md はこのセッション開始時に tdd-run が読み込んでいるはず。手順1・3では
再度 Read/cat せず、既にコンテキストにある内容を使う。

**1. コンテキスト名とテストディレクトリを確定する**

problem.md の `コンテキスト:` フィールドを確認する。なければユーザーに問う。
コンテキスト名は「どの範囲の問題について語るか」を示す。

次の二択をユーザーに確認する:

- **新設**: このコンテキストは新しい領域 → 英語 dir 名を決め、`tests/<dir>/` ディレクトリを新規作成する
- **既存への追加**: このコンテキストは既存 BC に属する → その BC の dir 名と `tests/<dir>/` を使う

**dir 名:** コンテキスト名の英語 kebab-case。例: `グリッドキャンバス` → `grid-canvas`。
ここで決めた dir 名を辞書エントリの `**dir:**` フィールドに書き、tdd-run のテストファイル置き場になる。
promote でコンテキストが吸収される場合はディレクトリ移動が必要になるため、この判断が後工程のコストを左右する。

**2. 既存 BC との関係を探る**

`dict-search.js` でキーワード検索し、新しいコンテキストと既存コンテキストの関係を確認する:

```bash
# 複数キーワードを一括スキャン（1行サマリーで全体を把握）
node "$(realpath "${CLAUDE_SKILL_DIR}")/../bin/dict-search.js" -s <キーワード1> [<キーワード2> ...]
```

- このコンテキストで使う言葉が、既存のどの BC にも現れているか
- 同じ言葉が異なる意味で使われているなら、それは意図的か
- 新しいコンテキストは既存コンテキストのデータを受け取る（下流）か、提供する（上流）か

発見した関係はエントリの **関係** フィールドに「〇〇コンテキストの××の反映」「〇〇コンテキストへデータを提供」などの形で書く。

**3. problem.md からアプリドメイン概念を拾い上げる**

まず「## /tdd-run への申し送り」の「語彙定義の種になるもの」「新規語彙の候補」を候補リストの
起点にする（/tdd-problem の調査手順で既に拾われている）。ここに挙がっていない概念だけ、本文
（やりたいこと・障害・制約・観察された症状）から「何が」「何を」「どうする」の主語・目的語を
拾って補う。
既存 BC の概念と重なる場合は新しいエントリを作らず、**関係** フィールドで参照する。

**4. 概念リストをユーザーに提示し、対話して確定する**

提示は概念名の列挙（「新規4件: 概念A, 概念B, ...」）や、名前だけを選択肢に並べる形で
済ませない。**必ず各概念の内容を表で本文に示してから**対話に入る:

```
| 概念名 | コンテキスト/ドメイン | 定義（案） | 関係 |
|--------|----------------------|------------|------|
| <name> | <context>/<domain> | <定義の案> | <既存BC・他概念との関係> |
```

ユーザーが判断するのは名前の採否ではなく定義と関係の中身。表を見た上で確認する:

- 名前は適切か（ユーザーが使う言葉か）
- 定義はユーザーの理解と一致するか
- 既存 BC の語彙との関係は正しく書かれているか
- 見落としている概念はないか

**5. 概念間の関係を明示する**

定義の中で他の概念を参照することが、リンクの実体です。
どの概念からも参照されない概念は「これは本当に独立した原始概念か」をユーザーと確認する。
確認には `dict-search.js -o`（孤立概念チェック）を使う（`dictionary.json` を cat して目視で確認しない）:

```bash
node "$(realpath "${CLAUDE_SKILL_DIR}")/../bin/dict-search.js" -o <plans_dir>
```

**6. plans/<プラン名>/dictionary.json に書く**

書き込みには `dict-write.js` を使う（Edit/Write や python で JSON を直接操作しない）。
エントリを書く前に `${CLAUDE_SKILL_DIR}/format.json` を Read してフォーマットを確認する。

```bash
node "$(realpath "${CLAUDE_SKILL_DIR}")/../bin/dict-write.js" add \
  --to <plans_dir>/dictionary.json --discovered "tdd-vocab plan" <<'EOF'
{
  "contexts": [ <コンテキスト定義（新設の場合のみ）> ],
  "entries": [ <アプリドメインエントリ> ]
}
EOF
```

wip フィールドは自動付与される。検証エラー（フォーマット違反・重複・未定義 context）が出たら
入力を直して再実行する。警告（参照未解決・孤立）は意図的なものか確認し、そうでなければ修正する。
ソリューションドメインエントリは tdd-run の手順4で追加される。
