# /tdd-vocab init — コンテキスト発見と初期辞書の生成

## 目的

既存のプロジェクトに初めてスキルを導入するとき、または BC が増えたと感じたときに、
コードベースを俯瞰してコンテキストを発見し、各コンテキストのアプリドメイン語彙を
コードから抽出して `docs/dictionary.json` を作る。

`/tdd-vocab plan` はフィーチャー着手時に一つのコンテキストを詳細定義する手順だが、
`init` はシステム全体を一度見渡してコンテキストの地図を描き、初期語彙まで定義する手順。

## 手順

**1. コードベースを読む**

```bash
find . -maxdepth 3 -type d | grep -v node_modules | grep -v .git
ls packages/ 2>/dev/null || true
```

加えて README、CLAUDE.md、既存のドキュメントを読む。
「このシステムは何ができるか」「どんなユーザーが何をするか」を把握する。

**2. コンテキスト候補を挙げる**

「どの範囲の問題について語るか」でグループを作る。
シグナルとなるもの:

- ディレクトリ・パッケージの境界（`packages/editor`, `packages/ssg` など）
- 異なるユーザー・役割が登場する領域
- 同じ言葉が異なる意味で使われている場所
- 明らかに独立して語れる機能領域

**インフラは除外する。** ファイルシステム・データベース・ビルドツールは
どのコンテキストにも属さない物理的事実であり、ドメイン概念ではない。
各コンテキストがインフラを自分の文脈でモデル化したものが、そのコンテキストの語彙になる。

**3. 候補をユーザーに提示し確定する**

各候補について:
- 名前は適切か（「どの範囲の問題について語るか」を一言で示しているか）
- 境界は明確か（このコンテキストの外にあるものが分かるか）
- 複数のコンテキストに同じ概念が現れるなら、それは意図的か

**4. 各コンテキストのアプリケーションドメイン語彙をコードから抽出する**

コンテキストごとに、そのコンテキストに属するコードを読んで概念を拾い上げる。

読む対象:
- README.md・docs/ 以下のドキュメント・CLAUDE.md（システムの説明・用語・設計方針）
- そのコンテキストのディレクトリ・ファイルの変数名・関数名・クラス名
- コメント・テストの `describe()` 文字列
- ユーザーに見える UI のラベル・メッセージ

拾い上げる候補:
- ユーザーが「何を」「どうする」という文の主語・目的語になるもの
- 複数の場所で繰り返し現れる概念名
- インフラ名（ファイルシステム・DB・フレームワーク）ではなく、それをモデル化した名前

コンテキストごとに候補リストをユーザーに提示し、対話して確定する:
- 名前は適切か（ユーザーが使う言葉か）
- 定義は実感と合っているか
- 見落としている概念はないか
- 複数のコンテキストに同じ言葉が現れるなら、それは意図的か

**5. コンテキスト間の関係を確認する**

複数のコンテキストにまたがる概念（同じ言葉が複数の BC に現れるもの）について:
- 同じインフラを別々にモデル化しているのか（それぞれ独自の概念として定義する）
- 一方が他方のデータを使う上流/下流の関係にあるのか（関係フィールドに明記する）

**6. docs/dictionary.json に書く**

確定した内容を一コンテキストずつ書く。ソリューションドメインは空エントリのまま。

survey は既存システムの知識を定義するため、`plans/` 経由ではなく `docs/dictionary.json` に直接書く。

エントリのフォーマットは skill.md の「語彙エントリのフォーマット」を参照する。
