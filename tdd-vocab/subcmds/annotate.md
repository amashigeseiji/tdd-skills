# /tdd-vocab annotate — 実装への @vocab アノテーション付与

## 目的

既存の実装ファイルを読み、辞書と照合して `@vocab` アノテーションを付与する。
`@test` の候補は `suggest-annotations.js` が機械的に出す。両方を同時に挿入する。

新規プロジェクトへの導入時や、アノテーションなしで書き進めた後の一括付与に使う。

## 手順

**1. @test 候補を出力する**

```bash
node ${CLAUDE_SKILL_DIR}/scripts/suggest-annotations.js [test-dir]
```

出力されたファイルと候補を確認し、手元に置いておく。

**2. 辞書を検索する**

アノテーション対象の関数名・クラス名をキーワードに `dict-search.js` を使って照合する:

```bash
# -n: 名前フィールドのみ検索（定義文のノイズを除く）、複数関数名を一括スキャン可
node "$(realpath "${CLAUDE_SKILL_DIR}")/../bin/dict-search.js" -n -s <関数名> [<関数名2> ...] [<plans_dir>]
```

**3. 対象ファイルを読んで照合する**

`@vocab` も `@test` もない実装ファイルを対象にする。

各関数・クラスについて、辞書エントリと照合する:

- 関数名・クラス名が辞書エントリ名（または英訳・略称）と対応しているか
- 関数の責務（何をするか）が辞書の「定義」と一致するか
- 複数の概念を実現している場合は複数の `@vocab` を付ける（セミラティスの共有ノードはその典型）
- 概念名のみを記載する（`// @vocab: 概念名`）。同名概念が複数コンテキストに存在する場合はコンテキストを明示する（`// @vocab: 概念名[context]`）

**4. アノテーションを挿入する**

関数・クラスの直前に `@vocab` と `@test` を書く:

```javascript
// @vocab: グリッドビルダー
// @test: tests/grid-canvas/grid-builder.test.js
function buildGrid(rows, cols) { ... }

// 同名概念が複数コンテキストに存在する場合はコンテキストを明示する:
// @vocab: ページ[ssg-core]
function renderPage(page) { ... }
```

対応する辞書概念が見つからない関数はスキップし、一覧に記録する。

**5. 結果を報告する**

付与したアノテーションの一覧と、スキップした関数の一覧を提示する。
スキップした関数は「辞書に概念がない」か「用途が不明」かを区別して報告する。
