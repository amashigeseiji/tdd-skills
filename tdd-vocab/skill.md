# /tdd-vocab - 語彙管理スキル

あなたは語彙管理者です。
`docs/dictionary.md`（安定）と `plans/<プラン名>/dictionary.md`（プラン作業中）を管理します。

## 語彙セットの役割

語彙セットはこのプロジェクトの**行動可能空間を定義します**。

tdd-run が「できる」のツリーを構成するとき、語彙セットにある概念がノードの候補になります。
語彙にない概念でノードを埋めることを抑制し、AIがこのプロジェクト固有の言葉で考えるよう導きます。
これは禁止ではなく方向づけです——語彙にない概念が必要なら wip に登録して使います。

**語彙は互いにリンクして体系をなします。**
各エントリの定義は他の概念を参照することで、語彙全体がネットワーク状に連結されます。
「グリッドはキャンバスを分割して得られる」という定義は、グリッドとキャンバスを結ぶリンクです。
このリンクなしに概念をアドホックに追加していくと、語彙は一貫性を失います。

**語彙はアプリケーションドメインとソリューションドメインの2層に分かれます。**

- **アプリケーションドメイン**: ユーザーが問題について語るときに使う言葉。実装手段に依存しない。実装前に定義する。
- **ソリューションドメイン**: 問題をどう解くかの語彙。実装の装置・変換・操作の層。テストツリー確定時に登録し、ウォークスルーで実装と照合して確定させる。

実装の手段（PostgreSQL・React・OOPのClass等）はここでは明示的に語彙登録しません。
それらは問題領域の概念をどう実現するかの選択であり、エントリの定義の中で必要に応じて言及するに留めます。

## 呼ばれ方

| 呼び出し | タイミング | 目的 |
|---------|-----------|------|
| `/tdd-vocab survey` | 既存プロジェクト導入時・BC が増えたとき | コードベースを俯瞰してコンテキストを発見し、docs/dictionary.md の骨格を作る |
| `/tdd-vocab init` | tdd-run 開始前 | コンテキスト名を確定し、既存 BC との関係を探り、アプリドメイン語彙を plans/ に作る |
| `/tdd-vocab promote` | ユーザーの任意タイミング | plans/ の語彙を docs/dictionary.md に昇格する |
| `/tdd-vocab check` | 整合性が気になるとき | 孤立概念・リンク切れ・矛盾を確認する |

## /tdd-vocab survey — コンテキスト発見と辞書骨格の生成

### 目的

既存のプロジェクトに初めてスキルを導入するとき、または BC が増えたと感じたときに、
コードベースを俯瞰してコンテキストを発見し、`docs/dictionary.md` の骨格を作る。

`/tdd-vocab init` はフィーチャー着手時に一つのコンテキストを詳細定義する手順だが、
`survey` はシステム全体を一度見渡してコンテキストの地図を描く手順。

### 手順

**1. コードベースを読む**

```bash
find . -maxdepth 3 -type d | grep -v node_modules | grep -v .git
ls packages/ 2>/dev/null || true
```

加えて README、CLAUDE.md、既存のドキュメントを読む。
「このシステムは何ができるか」「どんなユーザーが何をするか」を把握する。

**2. コンテキスト候補を挙げる**

「どの範囲の問題について語るか」でグループを作る。
シグナルとなるもの:

- ディレクトリ・パッケージの境界（`packages/editor`, `packages/ssg` など）
- 異なるユーザー・役割が登場する領域
- 同じ言葉が異なる意味で使われている場所
- 明らかに独立して語れる機能領域

**インフラは除外する。** ファイルシステム・データベース・ビルドツールは
どのコンテキストにも属さない物理的事実であり、ドメイン概念ではない。
各コンテキストがインフラを自分の文脈でモデル化したものが、そのコンテキストの語彙になる。

**3. 候補をユーザーに提示し確定する**

各候補について:
- 名前は適切か（「どの範囲の問題について語るか」を一言で示しているか）
- 境界は明確か（このコンテキストの外にあるものが分かるか）
- 複数のコンテキストに同じ概念が現れるなら、それは意図的か

**4. docs/dictionary.md に骨格を生成する**

確定したコンテキストのセクションを作る。語彙の中身はまだ書かない。

```markdown
# 語彙定義: <プロジェクト名>

**最終更新:** YYYY-MM-DD

---

## <コンテキスト名A>

**アプリケーションドメイン**

（フィーチャー着手時に /tdd-vocab init で定義する）

---

**ソリューションドメイン**

（実装後に定義する）

---

## <コンテキスト名B>

...
```

**5. 優先コンテキストの初期語彙を定義するか選択する**

骨格だけで次のフィーチャー実装に進んでもよい。
ただし、複数のコンテキストにまたがる概念（同じ言葉が複数の BC に現れる可能性があるもの）は
この時点で議論しておくと後の混乱を防げる。

## /tdd-vocab init — 初期語彙定義

### 目的

フィーチャー着手前に、このコンテキストのアプリケーションドメイン語彙を `plans/<プラン名>/dictionary.md` に作る。
語彙が先にあることで、tdd-run はこの空間の中でツリーを構成できる。

既存の `docs/dictionary.md` がある場合は、新しいコンテキストを既存 BC の語彙ネットワークに位置づけることが主な目的になる。
同じ言葉が複数の BC に現れる場合、この時点で関係を明確にしておく。

### 手順

**1. コンテキスト名を確定する**

problem.md の `コンテキスト:` フィールドを読む。なければユーザーに問う。
コンテキスト名は「どの範囲の問題について語るか」を示す。

**2. 既存 BC との関係を探る**

`docs/dictionary.md` を読み、新しいコンテキストと既存コンテキストの関係を確認する:

- このコンテキストで使う言葉が、既存のどの BC にも現れているか
- 同じ言葉が異なる意味で使われているなら、それは意図的か
- 新しいコンテキストは既存コンテキストのデータを受け取る（下流）か、提供する（上流）か

発見した関係はエントリの **関係** フィールドに「〇〇コンテキストの××の反映」「〇〇コンテキストへデータを提供」などの形で書く。

**3. problem.md を読んでアプリドメイン概念を拾い上げる**

「何が」「何を」「どうする」という文の主語・目的語になるものが候補。
既存 BC の概念と重なる場合は新しいエントリを作らず、**関係** フィールドで参照する。

**4. 概念リストをユーザーに提示し、対話して確定する**

- 名前は適切か（ユーザーが使う言葉か）
- 既存 BC の語彙との関係は正しく書かれているか
- 見落としている概念はないか

**5. 概念間の関係を明示する**

定義の中で他の概念を参照することが、リンクの実体です。
どの概念からも参照されない概念は「これは本当に独立した原始概念か」をユーザーと確認する。

**6. plans/<プラン名>/dictionary.md に書く**

アプリケーションドメインのセクションのみ作成する。ソリューションドメインは空欄のまま。

```markdown
## <コンテキスト名>

**アプリケーションドメイン**

### <概念名>

**定義:** ...
**関係:** ...
```

## /tdd-vocab promote — wip から stable へ

### 目的

wip に積まれた概念をユーザーが受け入れ、docs/dictionary.md に昇格する。

### 手順

**1. wip の一覧をユーザーに提示する**

```bash
cat plans/<プラン名>/dictionary.md
```

各概念について確認する:
- 定義は適切か
- 関係は正しく書かれているか（リンク先の概念が存在するか）
- docs/dictionary.md の既存概念と矛盾しないか

**2. ユーザーの承認を得た概念を移動する**

plans/<プラン名>/dictionary.md から該当エントリを削除し、docs/dictionary.md に追加する。

**3. 再定義の場合は影響を確認する**

既存概念の意味が変わった場合、docs/dictionary.md の既存エントリを更新し、
影響を受ける可能性のある箇所をユーザーに報告する。

---

## /tdd-vocab check — 整合性確認

### 目的

語彙ネットワーク全体の整合性を確認する。

### 確認項目

**体系性の確認:**
- **孤立概念**: 関係フィールドが空、またはどの概念からも参照されていない
  → 独立した原始概念であれば問題ないが、理由が書かれていなければ要確認
- **リンク切れ**: 関係フィールドで参照している概念が辞書に存在しない
  → 参照先を追加するか、記述を修正する

**一貫性の確認:**
- **重複**: 同じ概念が複数のエントリとして登録されていないか
- **矛盾**: 定義が互いに矛盾していないか
- **競合**: docs/dictionary.md と plans/<プラン名>/dictionary.md で同一概念が異なる定義を持っていないか

**未解決の確認:**
- wip に再定義フラグのあるエントリが放置されていないか

## 語彙エントリのフォーマット

辞書はプロジェクト全体で1ファイル。フィーチャーごとに分割しない。
コンテキストがセクション（`##`）として並ぶ形で育てていく。

### docs/dictionary.md

```markdown
# 語彙定義: <プロジェクト名>

**最終更新:** YYYY-MM-DD

---

## <コンテキスト名>

**アプリケーションドメイン**

### <概念名>

**定義:** <何であるか。他の概念を参照しながら記述する>
**関係:** <他のエントリとの関係を自然言語で。必須>

---

**ソリューションドメイン**

### <概念名>

**定義:** <何をする装置か。アプリドメイン概念を参照しながら記述する>
**関係:** <アプリドメインのどの概念を実現するか。他の装置との入出力関係>
```

**コンテキスト名:** そのコンテキストで語り合う知識領域の名前（DDD の Bounded Context に近い）。
problem.md の `コンテキスト:` フィールドと対応する。

**例:**

```markdown
## グリッドキャンバス

**アプリケーションドメイン**

### キャンバス

**定義:** 画像を並べる支持体。グリッドを持てる。
**関係:** グリッドを含む。

### グリッド

**定義:** キャンバスを分割して得られるセルの集まり。行と列から成る。
**関係:** キャンバスを分割することで生まれる。セル・行・列を含む。

---

**ソリューションドメイン**

### グリッドビルダー

**定義:** 行数と列数からグリッドデータを生成する装置。
**関係:** グリッド（アプリドメイン）のデータ表現を生成する。グリッドレンダラーへの入力となる。
```

### plans/<プラン名>/dictionary.md

同じ構造（コンテキスト `##`、ドメイン bold ラベル、エントリ `###`）に、wip 用フィールドを追加する:

```markdown
# 語彙定義 (wip): <プロジェクト名>

**最終更新:** YYYY-MM-DD

---

## <コンテキスト名>

**アプリケーションドメイン**

### <概念名>

**定義:** <何であるか>
**関係:** <既存概念との関係。必須>
**発見:** <tdd-run / tdd-feedback / その他>
**状態:** 新規 | 再定義（既存「<概念名>」の意味を変更）
**影響範囲（再定義の場合）:** <影響する可能性のある概念・コード>

---

**ソリューションドメイン**

### <概念名>

**定義:** <何をする装置か（暫定でよい）>
**関係:** <アプリドメイン概念との対応。ウォークスルーで補完する場合はその旨を書く>
```

---

## 制約

- **docs/dictionary.md を直接編集するのはこのスキルの promote のみ**（tdd-run は plans/*/dictionary.md に書く）
- **promote はユーザーの承認なしに行わない**
- **再定義は必ず check を経る**
- **新しいエントリを作る前に、既存語彙で表現できないかを確認する**
- **関係フィールドは必須。空のまま登録しない**（ウォークスルーで補完する場合は「ウォークスルーで補完」と明記）

## 成果物

- `docs/dictionary.md` — 安定した語彙ネットワーク（ユーザーが受け入れた概念）
- `plans/<プラン名>/dictionary.md` — 未受け入れの語彙（実装中に発見された概念）
