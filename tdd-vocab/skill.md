# /tdd-vocab - 語彙管理スキル

あなたは語彙管理者です。
`docs/dictionary.md`（安定）と `docs/wip/dictionary.md`（未受け入れ）を管理します。

## 語彙セットの役割

語彙セットはこのプロジェクトの**行動可能空間を定義します**。

tdd-run が「できる」のツリーを構成するとき、語彙セットにある概念がノードの候補になります。
語彙にない概念でノードを埋めることを抑制し、AIがこのプロジェクト固有の言葉で考えるよう導きます。
これは禁止ではなく方向づけです——語彙にない概念が必要なら wip に登録して使います。

**語彙は互いにリンクして体系をなします。**
各エントリの定義は他の概念を参照することで、語彙全体がネットワーク状に連結されます。
「グリッドはキャンバスを分割して得られる」という定義は、グリッドとキャンバスを結ぶリンクです。
このリンクなしに概念をアドホックに追加していくと、語彙は一貫性を失います。

**ここで管理するのは問題領域の語彙です。**
実装の手段（PostgreSQL・React・OOPのClass等）はここでは明示的に語彙登録しません。
それらは問題領域の概念をどう実現するかの選択であり、エントリの定義の中で必要に応じて言及するに留めます。

## 呼ばれ方

| 呼び出し | タイミング | 目的 |
|---------|-----------|------|
| `/tdd-vocab init` | プロジェクト開始時 | 問題領域の語彙ネットワークを一式作る |
| `/tdd-vocab add` | tdd-run・tdd-feedback から、または直接 | 新しい概念を wip に登録する |
| `/tdd-vocab promote` | ユーザーの任意タイミング | wip → stable に昇格する |
| `/tdd-vocab check` | 整合性が気になるとき | 孤立概念・リンク切れ・矛盾を確認する |

## /tdd-vocab init — 初期語彙定義

### 目的

実装を始める前に、この問題領域を語る語彙ネットワークを一式作る。
語彙が先にあることで、tdd-run はこの空間の中でツリーを構成できる。

### 手順

**1. problem.md を読む**

問題領域に登場する概念を拾い上げる。
「何が」「何を」「どうする」という文の主語・目的語になるものが候補。

**2. 概念のリストをユーザーに提示し、対話して確定する**

- 名前は適切か（ユーザーが使う言葉か）
- 定義は実感と合っているか
- 見落としている概念はないか
- 既存の docs/dictionary.md があれば、追加・変更が必要な点はないか

**3. 概念間の関係を明示する**

各概念について、他の概念とどう関係するかを自然言語で書き出す。
定義の中で他の概念を参照することが、リンクの実体です。

```
例：
キャンバス → 「グリッドを持てる」「画像を並べる支持体」
グリッド   → 「キャンバスを分割して得られる」「セルの集まり。行と列から成る」
セル       → 「行と列の交点にある単位」「画像を配置できる」
行         → 「グリッドを横に走る単位」「複数のセルを含む」
列         → 「グリッドを縦に走る単位」「複数のセルを含む」
```

どの概念からも参照されない概念、何も参照しない概念は孤立しています。
孤立している場合は「これは本当に独立した原始概念か」をユーザーと確認する。

**4. 辞書に書く**

- **tdd-run から呼ばれた場合（docs/dictionary.md が存在しない）:**
  `docs/wip/dictionary.md` に書く。作業仮説として機能し、実装後に promote する。
- **直接呼ばれた場合（既存プロジェクトへの追加など）:**
  ユーザーと確認の上、`docs/dictionary.md` または `docs/wip/dictionary.md` に書く。

## /tdd-vocab add — 概念の登録

### 目的

tdd-run や tdd-feedback が発見した概念を wip に登録する。

### 手順

**1. 既存語彙で表現できないか確認する**

docs/dictionary.md と docs/wip/dictionary.md を参照する。
既存の概念の組み合わせで表現できるなら、新しいエントリは作らない。

**2. 新規か再定義かを判断する**

- **新規**: この概念はまだ辞書にない
- **再定義**: 既存概念の意味が変質した（→ check を実行する）

**3. 既存概念との関係を確立する**

新しい概念が既存の何と関係するかを明示する。
どの既存概念とも関係しない場合は「独立した原始概念」として理由を書く。

**4. docs/wip/dictionary.md に書く**

登録したら呼び出し元（tdd-run など）に概念名と定義を返す。

---

## /tdd-vocab promote — wip から stable へ

### 目的

wip に積まれた概念をユーザーが受け入れ、docs/dictionary.md に昇格する。

### 手順

**1. wip の一覧をユーザーに提示する**

```bash
cat docs/wip/dictionary.md
```

各概念について確認する:
- 定義は適切か
- 関係は正しく書かれているか（リンク先の概念が存在するか）
- docs/dictionary.md の既存概念と矛盾しないか

**2. ユーザーの承認を得た概念を移動する**

docs/wip/dictionary.md から該当エントリを削除し、docs/dictionary.md に追加する。

**3. 再定義の場合は影響を確認する**

既存概念の意味が変わった場合、docs/dictionary.md の既存エントリを更新し、
影響を受ける可能性のある箇所をユーザーに報告する。

---

## /tdd-vocab check — 整合性確認

### 目的

語彙ネットワーク全体の整合性を確認する。

### 確認項目

**体系性の確認:**
- **孤立概念**: 関係フィールドが空、またはどの概念からも参照されていない
  → 独立した原始概念であれば問題ないが、理由が書かれていなければ要確認
- **リンク切れ**: 関係フィールドで参照している概念が辞書に存在しない
  → 参照先を追加するか、記述を修正する

**一貫性の確認:**
- **重複**: 同じ概念が複数のエントリとして登録されていないか
- **矛盾**: 定義が互いに矛盾していないか
- **競合**: docs/dictionary.md と docs/wip/dictionary.md で同一概念が異なる定義を持っていないか

**未解決の確認:**
- wip に再定義フラグのあるエントリが放置されていないか

## 語彙エントリのフォーマット

### docs/dictionary.md

```markdown
# 語彙定義: <プロジェクト名>

**最終更新:** YYYY-MM-DD

---

## <概念名>

**定義:** <何であるか。他の概念を参照しながら記述する>
**関係:** <他のエントリとの関係を自然言語で。必須>
```

**例:**

```markdown
## グリッド

**定義:** キャンバスを分割して得られるセルの集まり。行と列から成る。
**関係:** キャンバスを分割することで生まれる。セル・行・列を含む。
```

### docs/wip/dictionary.md

```markdown
# 語彙定義 (wip): <プロジェクト名>

**最終更新:** YYYY-MM-DD

---

## <概念名>

**定義:** <何であるか>
**関係:** <既存概念との関係。必須。どの概念とも関係しない場合はその理由>
**発見:** <tdd-run / tdd-feedback / その他>
**状態:** 新規 | 再定義（既存「<概念名>」の意味を変更）
**影響範囲（再定義の場合）:** <影響する可能性のある概念・コード>
```

---

## 制約

- **docs/dictionary.md を直接編集するのはこのスキルだけ**
- **promote はユーザーの承認なしに行わない**
- **再定義は必ず check を経る**
- **新しいエントリを作る前に、既存語彙で表現できないかを確認する**
- **関係フィールドは必須。空のまま登録しない**

## 成果物

- `docs/dictionary.md` — 安定した語彙ネットワーク（ユーザーが受け入れた概念）
- `docs/wip/dictionary.md` — 未受け入れの語彙（実装中に発見された概念）
