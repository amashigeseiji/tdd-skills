# パターン辞書の設計メモ

## 問題意識

現在のワークフローでは、ステートマシンやパイプラインのような構造パターンが採用されにくい。
AIはこれらのパターンを知っているが、「このテストツリーにパターンを当ててみる」という手順が存在しない。
一般的なパターンカタログの欠如が問題なのではなく、パターン照合を行うフェーズが欠けていることが問題。

---

## 設計方針

### domain: "pattern" を dictionary に統合する

`docs/dictionary.json` に `domain: "pattern"` エントリを追加する。
既存の `"application" | "solution"` に `"pattern"` を加えた三値。

別ファイル（`docs/patterns.json`）は作らない。
既存の `#概念名` 参照と dict-search によるグラフ探索がそのまま使えるため、
ネットワーク構造の中に置くほうが整合する。

### パターンエントリの役割

- カタログ（一般的なパターンの定義）ではなく、**このシステムが採用したパターンの採用記録**
- ドメイン語彙がそのパターンを `references` で参照することで、構造的決定が明示される
- dictionary が意味的決定を安定させるように、pattern エントリは構造的決定を安定させる

---

## 具体例：記事公開状態

```
application: 記事公開状態    → references: ステートマシン
solution:    公開状態遷移イベント → relations: belongs_to 記事公開ステートマシン
solution:    公開状態トランジション → relations: belongs_to 記事公開ステートマシン
pattern:     ステートマシン   ← これが参照先として存在する
```

パターンエントリ例：

```json
{
  "name": "ステートマシン",
  "en": "StateMachine",
  "context": null,
  "domain": "pattern",
  "definition": "有限個の状態とイベント駆動のトランジションで振る舞いを管理する構造パターン。#状態 ・#イベント ・#トランジション の3概念を含む。"
}
```

`context: null` はコンテキスト横断（クロスカッティング）であることを示す。

パターンを採用したことで `solution` 語彙がパターンの語彙（状態・イベント・トランジション）に揃う。
採用前は名前のない手続きとして散在していたものが、体系として立ち上がる。

---

## ワークフローへの介入点

**テストツリー初稿の後に、パターン照合ステップを追加する。**

```
問題 → テストツリー初稿 → [パターン照合] → テストツリー確定 → 実装
```

パターン照合では汎用的な問い（「何かパターンがあるか？」）ではなく、
ツリーの形から発火する具体的な観察をする：

- ノード群が特定の状態を持ち、条件によって遷移している → ステートマシン候補
- ノードが一方向に連なり、前の出力が次の入力になっている → パイプライン候補
- 一つのイベントに複数のノードが反応している → オブザーバー候補

当てはまるパターンがあれば、そのパターンの語彙でツリーを再構成してみる。
見通しが改善するか人間が判断し、採用するなら `plans/*/dictionary.json` に wip として記録する。
フィードバック後に `docs/dictionary.json` へ昇格——dictionary の二層管理と同じ構造。
