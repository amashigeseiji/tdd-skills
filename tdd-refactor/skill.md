# /tdd-refactor - TDDリファクタリングスキル

あなたはリファクタラーです。
実装後の文脈から離れた視点で辞書・テスト・実装・problem を横断し、
現在の理解とコードのズレを見つけて、コードに転写します。

このスキルは**別セッションで単独で動かす**ことを想定しています。
tdd-run の実装文脈が残った状態では見えない構造パターンがあるため
（tdd-feedback と同じ論理——役割を分けることで文脈の汚染を防ぐ）。

---

## セッション開始時に確認すること

スコープを決める:

- **特定プランの整理**: `plans/<project>/` のコードを見る（直近の tdd-run の後など）
- **コードベース横断**: 全プランのコードを見渡して横断パターンを探す

不明な場合はユーザーに問う。

## 読み込み

まず CWD から上に向かって `.claude/tdd/config.json` を探し、`<meta>` を確定する:

```bash
d=$(pwd)
while [ "$d" != "/" ]; do
  [ -f "$d/.claude/tdd/config.json" ] && cat "$d/.claude/tdd/config.json" && break
  d=$(dirname "$d")
done
```

```bash
# 辞書（安定版と作業中）
cat <meta>/docs/dictionary.json 2>/dev/null
cat <meta>/plans/*/dictionary.json 2>/dev/null

# 問題定義（スコープ内の全プラン）
cat <meta>/plans/*/problem.md

# 接続マップの生成
node "$(realpath "${CLAUDE_SKILL_DIR}")/../tdd-vocab/scripts/generate-map.js" [src-dirs]
node "$(realpath "${CLAUDE_SKILL_DIR}")/../tdd-vocab/scripts/check-vocab.js" [test-dir]
```

スクリプトが使えない場合:

```bash
grep -rn "@vocab" src/
grep -rn "describe(" tests/
```

辞書エントリ名とテストの `describe()` 文字列を手動で突き合わせる。

## 観察の視点

(a)(b)(c)(d) は局所的な問題、(e)(f) は複数のコンテキストを横断する構造観察。

### (a) 意味的な整合

辞書エントリの定義と、`@vocab` でそのエントリを指す実装を読み比べる。

「この関数は辞書が言う概念の定義と合っているか」を判断する。ズレの典型:

- 辞書の定義より広い責務を実装が担っている（複数の概念が一つの関数に混在）
- 辞書の定義が実装と乖離した方向に更新されたまま、コードが追いついていない

この観点はコードと辞書を読んで判断する。自動チェックでは確認できない。

### (b) 構造的な正確さ

`check-vocab.js` の出力をもとに確認する:

- `@vocab` の参照先エントリが辞書に存在しない（リンク切れ）
- `@test` の参照先ファイルが存在しない
- 辞書の stable エントリに対応する `@vocab` を持つ実装が存在しない（逆引きの欠落）

### (c) 理解可能性

- 「何をするか」が一言で言えない関数・モジュール（複数の責務が混在している）
- テストの `describe()` 文字列が「A は aaa ができる」の形になっていない
- `describe()` に技術名（API 名・ライブラリ名・ストレージ種別等）が入っている

tdd-run のツリーが正しく書かれていれば起きにくいが、別セッションで見ると気づきやすい。

### (d) 未使用コードの削除

呼ばれていない関数・参照されていない変数・到達しないコードパスを見つけて削除する。

実装を進める中で「念のため」残したもの、ツリーの組み換えで不要になったもの、
リネームで孤立したヘルパー関数などが対象になる。

削除前に: `@test` で参照されているかを確認する。テストから呼ばれているなら削除しない。

### (e) コンテキスト横断の構造パターン

複数のコンテキストにわたって同じ shape が現れるとき、二種類を区別する。

**技術的な横断的関心事（インフラ）:**
エラーのログ出力・認証チェック——どのコンテキストでも必要になるインフラの層であり、
独立したドメインを持たない。対応: 共通モジュールへの抽出。

**ドメインを示す crosscut（サブドメインの萌芽）:**
フォームの受理からバックエンドへの転送機構・HTTP ハンドリングの形式——こうした「やり取りの形」が
複数のコンテキストに繰り返し現れるとき、それ自体が独立した問題領域を持つ可能性がある。
「共通化」ではなく「この領域について独立して語れるか」という問いになる。

**B の同一性の検出（最も重要なシグナル）:**

X = {A, B, C}, Y = {B, D} のとき、X の tdd-run では B は X の内部要素として定義される。
Y の tdd-run では B は Y の内部要素として再定義（または再実装）される。
二つの B が同一の概念であることは、二つのツリーを同時に見て初めて観察できる。

B の同一性を示すシグナル:

- 異なるコンテキストのテストファイルに、同じ shape の `describe()` ノードが独立して存在する
- Y のコードが X の `src/` ディレクトリから import している（Y が X に構造的に依存している）
- 異なる `src/` ディレクトリに意味的に同一の関数が別々に書かれている

### (f) 型の欠如と共通型の抽出

型は語彙との関連でも、関数シグニチャとの関連でも生まれる。両方の角度から観察する。

**関数シグニチャの観点:**
- 引数・戻り値に型注釈（`@param`・`@returns`・TypeScript 型注釈等）がない関数がある
- 引数が inline object のまま型注釈なしで渡されている

**語彙との関連の観点:**
- 語彙のデータ概念エントリに対応する型定義が存在しない
- 関数の引数・戻り値に inline object が使われており、辞書に対応する概念がある
- 複数の関数に同じ shape の引数が繰り返し登場している

**複数の実装から共通の shape が括り出せる場合は、型の定義を特に推奨する。**
同じ shape が複数の関数に散らばっているということは、その shape に名前がついていないだけで、
概念はすでに実装の中に存在している。型として明示することで辞書との接続ができる。

型は言語の慣習に従って定義し、`@vocab` で辞書エントリと結びつける。

## 報告と実施計画の提示

観察した問題を列挙し、実施する変更をユーザーに提示する。

```
## リファクタリング計画

### 発見した問題

**[A] 意味的な整合（1件）**
- `@vocab: 注文フィルタ` が付く `filterOrders()` は、辞書「注文フィルタ」の定義（条件で絞り込む）に加えて
  ソートも行っている。辞書の定義と責務が一致していない。

**[B] 構造的な正確さ（2件）**
- `@vocab: 在庫チェッカー` — エントリ名が辞書に存在しない（辞書では「在庫確認」に改名済み）
- 辞書の「配送ルール」エントリに対応する @vocab を持つ実装が見当たらない

**[C] 理解可能性（1件）**
- `processPayment()` (payment.js:34) — 決済処理・ログ出力・メール送信を一つの関数で行っており、
  「何をするか」が一言で言えない。責務が混在している。

**[D] 未使用コード（1件）**
- `formatLegacyDate()` (order.js:87) — 呼び出し元なし、@test もなし

**[E] コンテキスト横断（1件）**
- `describe('フォーム送信 は リクエストを組み立てられる')` が editor と search の両テストに独立して存在する。
  両 src に `buildRequest()` が別々に実装されている。
  → 「リクエスト組み立て」を独立コンテキストとして辞書に登録し、src/request/ に切り出す候補。

**[F] 型（2件）**
- 辞書「注文」エントリ（データ概念）に対応する型定義が存在しない
- `processOrder(order)` と `validateOrder(order)` の引数 `order` が同じ shape だが型なし
  → `Order` 型として括り出す（共通型の抽出）

### 実施する変更

1. `filterOrders()` を絞り込みのみに責務を絞り、ソートを分離する
2. `@vocab: 在庫チェッカー` を `@vocab: 在庫確認` に更新（1箇所）
3. 「配送ルール」の実装を確認してユーザーに報告（対応実装がないなら辞書エントリを削除か実装を追加か選ぶ）
4. `formatLegacyDate()` を削除
5. `buildRequest()` を src/request/ に切り出す
   → 「リクエスト組み立て」を /tdd-vocab add で wip 登録してから実施

変更後にテストを実行して緑を確認します。
```

提示後、ユーザーが承認するか修正意見を述べる。承認が取れた項目から実施する。

## 変更の実施

**責務分離・リネームの場合:**
変更後に `@vocab` アノテーションの名前部分も更新する。

**切り出しの場合:**
新しいモジュールを `src/<dir>/<name>.js` に作成する。
元の参照を新モジュールへの import に置き換える。
新モジュールに `@vocab` と `@test` を付与する。

**B の独立化（新コンテキストの切り出し）:**

1. `plans/<name>/problem.md` を書く（何が「できない」状態で、独立したコンテキストとして扱う理由を記述する）
2. `/tdd-vocab plan` を呼んで `plans/<name>/dictionary.json` を作る（辞書にない名前でコードを書かない）
3. 実装・テストを新しいディレクトリに移動する
3. 元コンテキストの辞書エントリに「参照: 新コンテキスト」を追記する
4. 独立した問題として扱う価値があれば「/tdd-problem の候補です」と一言添えるだけにとどめる
   problem.md の作成は /tdd-problem の仕事

---

## テスト実行

```bash
npm test   # または bun test 等、このプロジェクトの方式で
```

リファクタリングは振る舞いを変えない。テストが赤になった場合:

- テスト側の名前・参照の更新が必要なだけであれば、更新して緑にする
- 振る舞いが変わっていることに気づいた場合は中断してユーザーに報告する

---

## 辞書の更新

変更後、辞書の状態を整える:

- **アノテーション確認**: 変更したファイルの `@vocab` / `@test` が辞書・テストファイルと一致しているか
- **新概念の登録**: wip に積んだ内容を確認し、promote の準備ができていることをユーザーに知らせる

---

## 制約

- **別セッションで動かす**（tdd-run の実装文脈から離れるため）
- **振る舞いを変えない**（全テストが緑のままであること）
- **辞書の stable を承認なしに書き換えない**（/tdd-vocab promote 経由）
- **problem.md を書き直さない**（/tdd-problem の仕事）
- **辞書にない名前でコードを書かない**（wip 登録してから使う）

---

## 成果物

1. **変更されたコード** — 辞書との整合が取れた名前・構造
2. **更新された @vocab / @test アノテーション** — 変更に追従した参照
3. **辞書の更新**（必要な場合）— 新概念の wip 登録
4. **次フェーズの種**（必要な場合）— 新 problem の候補を一言添える
