# TDD支援スキル — リポジトリ作業ガイド

このリポジトリは Claude Code 用の TDD 支援スキル一式（**problem → run → feedback + vocab**）。

**このファイルが読まれる場面:** このリポジトリを開いて**スキルを再設計・更新する**とき
（CLAUDE.md は作業ディレクトリで自動ロードされる）。スキルを*使う*ときは各 `tdd-*/skill.md`
だけが注入され、このファイルは読まれない。だからここには「使い方」ではなく
「**なぜこの形か＝再設計のときに要る前提**」を置く。

## まず読む

- `docs/rationale.md` — なぜこの形なのか（思考枠組み／深いモデルの本体）。
  **スキルの構造を変える前に必ず読む。**
- `docs/notes/` — そこに至った思考・対話の記録（探索の沈殿、日付つき）。思想的・哲学的探求。
- `docs/references/` — 具体的な構想・設計の検討メモ（日付つき）。スキルへの組み込みが完了したものは `docs/references/archives/` に移す。
- `README.md` — 使う人向けの全体像（ワークフロー・運用）。

## 再設計するときの心得

「スキルの語彙・体系を変える」ことは「思考枠組みを変える」こと。問題が変わったのではなく
*枠組みが更新された*ために再設計することがある。その更新は `docs/rationale.md` を起点にし、
結論だけでなく*なぜ*を更新する。更新したら `docs/notes/` に新しい日付のメモを足す。

判断の軸（詳細は rationale）:

- **フェーズの責務を混ぜない** — 問題/解決、事実/解釈、コミット/保留。
- **分解は実装者の責務** — 分解を事前フェーズに固定しない。実装しながら合成の形が見えてくる。
  深いモデルは再編成から生まれ、手続き化できない。できるのは*再編成を試みる契機*を用意すること。
- **語彙は体系をなす** — 概念は互いにリンクしてネットワーク状に連結される。孤立した概念は
  アドホックな追加の兆候。語彙セットはAIの行動可能空間を定義するガードレール。
- **接地は人間と実行から取る** — AI は差異の*源*ではなく*引き出し役*。
- **緑 ≠ 完了** — コードは使用で清算される仮説。run は利用可能な状態まで持っていき、
  feedback が実使用と照合する。

## CHANGELOG の更新ルール

スキルを変更したとき、以下の基準で `CHANGELOG.md` に記述する:

- **書く**: スキルの動作・手順・出力が変わるもの。新スキル追加・廃止。ユーザー側で対応が必要な変更（マイグレーション等）。
- **書かなくてよい**: typo 修正、表現の微調整、ドキュメントのみの変更。

エントリは日付見出し（`## YYYY-MM-DD`）の下に追記する。
PR 単位でまとめて書くのが書きやすい。直 push の場合も大幅な改訂なら忘れずに書く。

## 文体ルール（このリポジトリの文書を書くとき）

- 平易に。初見のエンジニアが読んで分かること。
- 比喩・造語・過度な抽象ラベルを避ける。概念には役割の伝わる名前を与える。
- 「力・条件（forces）」は問題側（problem.md）の記述に限る。
- 受け入れ条件は `P-01` 形式、利用仮説は「使ったらこうなるはず／こうなったら外れ」。
- **他リポジトリの固有名をこのリポジトリの資料類に書かない**（CHANGELOG・skill.md・README・
  `docs/` 以下すべて。スキルの利用プロジェクトの名前や、そのプロジェクト内の findings 番号など）。
  症状・原因は一般的な言葉で記述する。
