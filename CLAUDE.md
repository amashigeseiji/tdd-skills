# TDD支援スキル — リポジトリ作業ガイド

このリポジトリは Claude Code 用の TDD 支援スキル一式（**problem → design → run → feedback** の4フェーズ）。

**このファイルが読まれる場面:** このリポジトリを開いて**スキルを再設計・更新する**とき
（CLAUDE.md は作業ディレクトリで自動ロードされる）。スキルを*使う*ときは各 `tdd-*/skill.md`
だけが注入され、このファイルは読まれない。だからここには「使い方」ではなく
「**なぜこの形か＝再設計のときに要る前提**」を置く。

## まず読む

- `docs/rationale.md` — なぜこの形なのか（思考枠組み／深いモデルの本体）。
  **スキルの構造を変える前に必ず読む。**
- `docs/notes/` — そこに至った思考・対話の記録（探索の沈殿、日付つき）。
- `README.md` — 使う人向けの全体像（4フェーズ・運用）。

## 再設計するときの心得

「スキルの語彙・体系を変える」ことは「思考枠組みを変える」こと。問題が変わったのではなく
*枠組みが更新された*ために再設計することがある。その更新は `docs/rationale.md` を起点にし、
結論だけでなく*なぜ*を更新する。更新したら `docs/notes/` に新しい日付のメモを足す。

判断の軸（詳細は rationale）:

- **フェーズの責務を混ぜない** — 問題/解決、事実/解釈、コミット/保留。
- **設計文書は平易な自然言語で** — 比喩・造語・記号ラベル（「力A」等）を成果物の*解決側*に
  持ち込まない。「力・条件（forces）」は*問題側（problem.md）*に限る。
- **分解は診断であって、解の生成ではない** — 深いモデルは再編成から生まれ、手続き化できない。
  できるのは*再編成を試みる契機*を用意すること。
- **接地は人間と実行から取る** — AI は差異の*源*ではなく*引き出し役*。
- **緑 ≠ 完了** — コードは使用で清算される仮説。run は利用可能な状態まで持っていき、
  feedback が実使用と照合する。

## 文体ルール（このリポジトリの文書を書くとき）

- 平易に。初見のエンジニアが読んで分かること。
- 比喩・造語・過度な抽象ラベルを避ける。概念には役割の伝わる名前を与える。
- 受け入れ条件は `P-01` 形式、利用仮説は「使ったらこうなるはず／こうなったら外れ」。
