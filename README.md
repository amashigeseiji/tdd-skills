# TDD支援スキル

Claude Code で、問題定義・設計・実装を分離して開発を進めるためのスキルセットです。

## 概要

開発を **問題定義 → 設計 → 実装（TDD）→ 利用フィードバック** の 4 フェーズに分離します。
セッションを分割することでトークン消費を抑えつつ、各フェーズの責務を明確にして
「問題」と「解決」、「設計」と「テスト」、「事実」と「解釈」の混同を防ぎます。
最後の利用フィードバックは、使ってみて生じたズレを次のループに戻す入口で、全体は
直線ではなく再帰的に回ります。

### なぜフェーズを分けるか

問題定義と解決案の策定を同じフェーズで行うと、解決手段を問題の記述に紛れ込ませてしまう
（「正規化が必要」「パディングが足りない」はすでに解決領域の言葉で、問題そのものではない）。
さらに設計フェーズでテストまで書くと、設計が「どう測るか」まで先回りして、テスト仕様を
書きすぎる・テストが実装の写経になる、といった症状が出る。

そこで責務を 3 つに分ける:

- **問題（発見・定義）** — 何が適切か。問題の言葉だけで書く
- **設計** — どう解くか。問題を小問題へ分解し、解決の道具立て（関数・オブジェクト・サービス・
  データ構造）と、その設計が満たすべき性質（受け入れ条件）を示す。テストは書かない
- **実装（TDD）** — 受け入れ条件を検証するテストを設計・実装し、テスト駆動で実装する。
  変更を利用可能な状態にする
- **利用フィードバック** — 実際に使い、何が起きたかを集めて、設計の利用仮説と照合する。
  ズレが次のループを開く

本来の TDD（テストファースト）は実装フェーズに一本化される。

（構造としては problem=発見・定義、design=展開、run=提供、というダブルダイヤモンド的な分割に符合する）

## スキル構成

### `/tdd-problem` - 問題定義フェーズ

**目的:** 問題領域を問題の言葉で確定する

- 観察された現象から問題を言語化（解決手段を持ち込まない）
- 「何が適切か」「解決したと言える状態」を定性的に定義
- **成果物:** `plans/<project>/problem.md`
- **やらないこと:** 解決手段・テスト・実装を書かない

### `/tdd-design` - 設計フェーズ

**目的:** 問題をどう解くかを設計する

- problem.md を基に、問題を小問題へ分解する
- 各小問題を解く道具立て（概念）と、それらをつなぐ全体の組み立てを示す
- 設計が満たすべき性質（受け入れ条件）を記述
- 必要な量的パラメータは暫定値を提示（最終判断は run）
- **成果物:** `plans/<project>/design.md`
- **やらないこと:** テストの設計・実装、実装コードを書かない

### `/tdd-run` - 実装フェーズ（TDD）

**目的:** 設計の受け入れ条件を検証しつつテスト駆動で実装する

- design.md の受け入れ条件を検証するテストを**設計・実装**
- red → green → refactor のループ
- 受け入れ条件・暫定値の妥当性を検証
- 変更を利用可能な状態にする（入口へ組み込み、実アプリで動かして確認）
- **完了の定義:** テスト緑だけでは完了でなく、利用可能（実アプリで動かせる）になって完了
- **成果物:** テスト、利用可能になった実装、`findings.md`（オプション）、docs（オプション）

### `/tdd-feedback` - 利用フィードバックフェーズ

**目的:** 実際に使い、利用の事実と印象を集めて、設計の利用仮説と照合する

- 開発に肩入れしない聞き役として、**別セッション**でインタビューする
- まず操作の事実を集め、利用仮説（使ったらこうなるはず／外れ）と照合
- 粗い印象（「デカい」等）を、ユーザー自身の言葉で**解像度を上げる**（解釈は代行しない）
- **成果物:** `plans/<project>/findings.md`（利用の事実と、解釈・原因の仮説を節で分けて一枚に）
- **やらないこと:** 設計の擁護・誘導質問・コード修正

## ワークフロー

```
セッション1: 問題定義
  $ /tdd-problem
  → plans/<project>/problem.md

セッション2: 設計
  $ /tdd-design
  → plans/<project>/design.md

セッション3: 実装（TDD）
  $ /tdd-run
  → テスト + 実装（利用可能な状態）

セッション4: 利用フィードバック（使ってみる → インタビュー）
  $ /tdd-feedback
  → plans/<project>/findings.md（利用の事実＋解釈の仮説を節で分けて一枚に）
  → ズレがあれば problem / design へ戻り、次のループへ
```

### フィードバックの戻し先（重要）

実装中・利用中の発見（findings.md）は、性質によって戻し先を振り分ける:

| 発見の性質 | 戻し先 |
|-----------|--------|
| **問題の発見**（適切さの条件自体の見落とし、新たな力・制約） | `/tdd-problem`（problem.md へ） |
| **設計の不備**（受け入れ条件が成り立たない、暫定値が不適切） | `/tdd-design`（design.md へ） |

この振り分けが肝心。問題の発見を設計に戻すと、問題と解決が再び混ざる。

```
実装で「問題の発見」 → /tdd-problem で problem.md 更新 → /tdd-design → /tdd-run
実装で「設計の不備」 → /tdd-design で design.md 更新 → /tdd-run
```

## 成果物の連鎖

```
problem.md  ──→  design.md  ──→  テスト + 実装  ──→  findings.md
（何が適切か）   （どう解くか・    （どう測り・      （使ってどうだったか）
                受け入れ条件・    実装するか）
                利用仮説・暫定値）       │
      ↑                                  │
      └──────（ズレ → findings → problem/design に戻り、次のループ）──┘
```

## ファイル構造

```
plans/
  <project>/
    problem.md               # 問題定義（/tdd-problem）
    design.md                # 設計・受け入れ条件・利用仮説・暫定値（/tdd-design）
    findings.md              # 実装・利用からの戻り（事実＋解釈・戻し先つき）。削除せず history/ へ
    history/
      problem-v1.md
      design-v1.md
      findings-design-v1.md  # 反映後にアーカイブ（対応バージョンを明記）

src/
  xxx.js                     # 実装（/tdd-run）
  xxx.test.js                # テスト（/tdd-run が設計・実装）
```

完成後:

```
docs/
  features/
    <feature>.md             # 機能ドキュメント（オプション）
plans/
  archives/
    <project>-2026-06-09/    # アーカイブ（オプション）
```

## 各成果物の要点

### problem.md（/tdd-problem）
- 問題（観察された症状と、その背後の問題を区別）
- 問題を構成する力・条件
- 範囲外
- 解決したと言える状態（定性的）

### design.md（/tdd-design）
- 問題の分解（独立に解ける小問題へ）
- 設計（概念と道具立て）と全体の組み立て
- 満たすべき性質（受け入れ条件）— 検証可能な主張
- 量的パラメータ（暫定値、最終判断は run）
- run への申し送り

### findings.md（/tdd-run・/tdd-feedback）
- 利用フェーズ: 「事実」の節に 操作の事実・利用仮説の照合・印象（ユーザーの言葉）
- 「解釈・原因の仮説」の節は、発見ごとに **戻し先**（/tdd-problem か /tdd-design か）を明記
- 実装フェーズ由来の受け入れ条件・暫定値の検証結果も含む
- **削除せず**、反映後は対応バージョンを明記して history/ へアーカイブ

## 制約まとめ

| | やること | やらないこと |
|---|---|---|
| /tdd-problem | 問題の言語化 | 解決手段・テスト・実装 |
| /tdd-design | 問題の分解・道具立て・受け入れ条件・暫定値 | テスト設計実装・実装コード・問題の再定義 |
| /tdd-run | テスト設計実装・TDD実装・利用可能な状態にする・検証 | 設計の作り替え（不備は findings へ）・大規模調査・本番デプロイ |
| /tdd-feedback | 利用事実・印象の収集・利用仮説の照合 | 設計の擁護・誘導質問・ユーザーの代わりの解釈・コード修正 |

共通: 時間見積もりはしない。

## トラブルシューティング

### design.md / problem.md が見つからない
前フェーズを先に実行する（problem → design → run の順）。

### テストが5回連続で失敗
1. 続ける 2. findings.md を生成して再設計／再定義へ 3. テスト自体を見直す

### 設計フェーズで問題が曖昧だと分かった
design では問題を定義し直さない。何が不明かを明示して /tdd-problem に差し戻す。

## このスキルを再設計するとき

スキルの構造そのものを変えるなら、まず「なぜこの形か」を読む（使うときは不要）:

- `CLAUDE.md` — 再設計時の入口（このリポジトリで作業すると自動ロード）
- `docs/rationale.md` — なぜこの形か（思考枠組み／深いモデルの本体）
- `docs/notes/` — そこに至った思考・対話の記録
