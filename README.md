# TDD支援スキル

Claude Code で、問題定義・語彙定義・実装を分離して開発を進めるためのスキルセットです。

## 概要

このワークフローは **問題 → 辞書 → テスト → 実装 → フィードバック** の5層構造を持ちます。中心の3層（辞書・テスト・実装）は AI が作業する場で、辞書とテストは実装の意味を定義するガードレールです。AI はこの意味空間の中で動きます。外側の2層（問題・フィードバック）は現実との接地点で、「何が適切か」と「使ったらどうだったか」を持ち込みます。

設計と実装を分離するアプローチとは異なり、TDD を参照源としつつ、AI が意味空間の中で構築と分解をできることを目指しています。

## ワークフロー

```
┌──────────┐   ┌───────────────────────────────────────┐   ┌────────────┐
│ problem  │──▶│                 run                   │──▶│  feedback  │
│          │   │  ┌────────┬────────────┬──────────┐   │   │            │
│          │   │  │  dict  │ test tree  │   impl   │   │   │            │
│          │   │  └────────┴────────────┴──────────┘   │   │            │
└──────────┘   └───────────────────────────────────────┘   └────────────┘
     ▲                                                           │
     └───────────────────────────────────────────────────────────┘
```

### 問題定義

問題とは「ユーザーがやりたいことがあり、それを阻む何かがある状態」です。問題定義では、「やりたいこと」と「それを阻むもの」を、解決手段を使わずに言語化します。ここで記述するのはユーザーが「できない」状態であり、次の実装セッションで「できる」のツリーを構成することと対応しています。

```
/tdd-problem
```

完了すると `plans/<project>/problem.md` が作成されます。あわせてユーザーストーリー（`user-story.md`）が自動生成され、次の実装セッションへの申し送りになります。

### 実装

問題定義をもとに、辞書の計画を立て、テストツリーを構成して実装するセッションです。

```
/tdd-run
```

セッション開始時に辞書計画（コンテキストの確定・既存語彙との関係整理）が行われます。その後「できる」のツリーを組み立て、骨格テストを生成してから、末端から実装して合成していきます。ルートが green になったら辞書・テスト・実装のウォークスルーで名前の一貫性を確認し、実アプリへの統合まで行います。

完了の定義はテストが緑になることではなく、**実際にアプリで動かせる状態**になることです。

### フィードバック

実装を実際に使ったあと、実装セッションとは別のセッションで起動します。実装の文脈から離れることで、設計を擁護せずに事実を集められます。

```
/tdd-feedback
```

語彙レビュー（wip 語彙の照合と昇格）、ソリューション構造のレビュー、利用インタビュー（何が起きたかの事実収集）の順に進みます。発見は `plans/<project>/findings.md` にまとめられ、性質によって次の問題定義セッションか実装セッションへ戻ります。ズレがあれば次のループが始まります。

このセッションはクロージングも兼ねています。ループを完了と見なした場合、`plans/<project>/` 以下のファイル一式がアーカイブされます。

辞書とテストがどのタイミングで作られどう参照されるかの設計意図は [`docs/rationale.md`](docs/rationale.md) を参照してください。

## コマンド一覧

通常のループで使うのは `/tdd-problem`・`/tdd-run`・`/tdd-feedback` の3つです。語彙管理・受け入れテスト生成などは各スキルが内部で呼び出します。

### メインループ

| コマンド | 役割 |
|---|---|
| `/tdd-problem` | 問題定義。`plans/<project>/problem.md` を作成する |
| `/tdd-run` | 実装。辞書計画・テストツリー・実装・統合まで一貫して行う |
| `/tdd-feedback` | フィードバック・クロージング。語彙昇格・インタビュー・findings 生成 |

### 補助コマンド

| コマンド | 役割 | タイミング |
|---|---|---|
| `/tdd-init` | 新規プロジェクト導入時の初期化（config・scaffold・vocab） | 初回のみ |
| `/tdd-cycle <project>` | `tdd-run` を自律実行し完了後に一括報告する auto モード | `problem.md` 作成後 |
| `/tdd-refactor` | 辞書・テスト・実装を横断して理解とコードのズレを転写する | 任意のタイミング |
| `/tdd-update` | `git pull` + 新規スキルのシンボリックリンク追加 | スキル更新時 |

### 内部で呼ばれるコマンド（直接呼ぶことも可）

| コマンド | 役割 |
|---|---|
| `/tdd-vocab init\|plan\|promote\|check` | 語彙管理。コンテキストごとに問題/解法の2層で管理 |
| `/tdd-userstory create\|run` | ユーザーストーリーの生成・受け入れテストの実行 |
| `/tdd-scaffold` | プロジェクト固有のスタブ生成スクリプトを作成 |

## ファイルストラクチャー

```
plans/<project>/
  problem.md        # 問題定義
  user-story.md     # BDD仕様（/tdd-userstory create）
  dictionary.md     # 作業中の語彙（promote で docs/ に昇格）
  findings.md       # フィードバックの記録（削除せず history/ へ）
  history/          # アーカイブ

docs/
  dictionary.md     # 安定した語彙ネットワーク

tests/acceptance/
  <project>.spec.ts # 受け入れテスト

.claude/tdd/
  config.json       # plans コミット設定
  scaffold.sh       # スタブ生成スクリプト
```

## セットアップ

### スキルのインストール

このリポジトリをクローンしたら:

```bash
bash bin/update.sh
```

`skill.md` を持つすべてのディレクトリを `~/.claude/skills/` にシンボリックリンクします。完了後、Claude Code を**再起動**してスキルを認識させてください。

以降の更新は `/tdd-update` で行えます（`git pull` + 新規スキルのリンク追加）。

### プロジェクトへの導入

TDD スキルを使いたいプロジェクトで、**初回のみ**実行します:

```
/tdd-init
```

プロジェクトの言語・フレームワーク・ディレクトリ構成を調査したうえで以下を行います:

- `plans/` のコミット設定を確認し `.claude/tdd/config.json` に保存する
- プロジェクト固有のスタブ生成スクリプト `.claude/tdd/scaffold.sh` を生成する
- コードベースを俯瞰してコンテキストを発見し、`docs/dictionary.md` の骨格を作る
