# /tdd-run - TDD実践スキル

あなたはTDD実践者です。
**問題とは「できない」状態であり、実装とはそれを「できる」に変換することです。**

「できる」のツリーがあなたの地図です。ツリーの各ノードが一本のテストになります。
ルートノードが green になったとき、問題が解けています。

---

## セッション開始時に開くもの

```bash
cat plans/<project>/problem.md
cat docs/dictionary.md 2>/dev/null
cat docs/wip/dictionary.md 2>/dev/null
```

problem.md が無い場合: `/tdd-problem` で問題を定義してください。

**docs/dictionary.md が存在しない場合（新規プロジェクト）:**
ツリーを構成する前に `/tdd-vocab init` を呼ぶ。
ただし、この時点での語彙定義は**作業仮説**なので `docs/wip/dictionary.md` に書く
（`docs/dictionary.md` には書かない）。
wip の語彙が仮のガードレールとして機能する。
実装・受け入れを経た後、`/tdd-vocab promote` で stable に昇格する。

同一概念が dictionary.md と wip/dictionary.md の両方にある場合、wip が優先です。
これらはセッション中、コードを書く行為の背景として常に存在します。

---

## 「できる」のツリー

```
X は xxx ができる  ← ルートノード（問題解決の仮説）
├── A は aaa ができる
│   ├── D は ddd ができる
│   └── E は eee ができる
└── B は bbb ができる
    └── C は ccc ができる
```

各ノードは「名前は〜ができる」の形をしています。
**名前が先にあり、その名前が何をできるかが続きます。**
「X とは xxx ができるような何かだ」——この問いに答えることが、ノードを置く行為です。

**ツリーはテストファイルです。**
E のテストに対応するコードを書くとき、X の振る舞いについて意識できていること——
これが近視眼的な実装を防ぎます。

ツリーはセミラティス状になってよい（ノードが複数の親を持つ）。
共有ノードは命名の契機です。

ツリーは理解の更新とともに変わります。
ただし、**上位ノードの性質が変質するときは problem.md との照合が必要**です（手順6）。

---

## 手順

### 1. 読み込み

セッション開始時に上記三ファイルを読む。

コードベースの調査（最小限）:
- 実装の差し込み位置
- 既存の関連実装・使えるもの

### 2. ルートXの定義と命名

problem.md を読んで問う: **「何ができれば、この問題は解けているか」**

**まず既存語彙で表現できないか確認する:**
docs/dictionary.md と docs/wip/dictionary.md を参照し、
「この問題は、既存の概念の組み合わせで表現できるか？」を問う。
既存語彙で表現できるなら、新しい名前は作らない。

既存語彙で表現できない場合、新しいXを定義する:
「X とは xxx ができるような何かだ」という形で表現する。
名前が定まったら **tdd-vocab に登録を依頼する**:

```
/tdd-vocab: 概念「X（名前）」を登録してください
定義: xxx ができるもの
docs/wip/dictionary.md に
```

### 3. ルートのテストを書いて red のまま置く

「X は xxx ができる」のテストを書く。
このテストはツリーが完成するまで green にしない。

### 4. ツリーを分解してユーザーに提示

「X が xxx できるためには何が必要か」を分解する。各ノードを一行で。
末端（直接実装できる粒度）まで続ける。

**分解が終わったらユーザーに提示し、確認を得てから次に進む。**

### 5. 末端から実装して上へ合成する

末端ノードから順に red → green → refactor を繰り返す。
green になったら一つ上のノードへ移り、合成する。

各ノードのテストの書き方:

```javascript
describe('A は aaa ができる', () => {
  it('xxx したとき yyy になる', () => {
    // 実装より先にインターフェースを定める
  })
})
```

**テスト実行コマンド:**

```bash
npm test -- <test-file-path>   # または bun test <file> 等、検出した方式で
```

**ループ防止（5回連続失敗時）:**

```
⚠️  5回のイテレーションで通りませんでした。

次のアクションを選んでください:
- 続ける
- 一時中断（findings.md を生成して問題定義へ）
- 分解の仕方を見直す
```

**命名の契機を逃さない:**
以下が見えたとき、立ち止まる:

- 複数のノードがまとまって「〜の流れ」として捉えられるとき
- セミラティスの共有ノードに名前がないとき
- 「この関数群は何をしているか」に一言で答えられるとき

立ち止まったら、**まず既存語彙で表現できないかを確認する**。
既存語彙で表現できなければ、新しい名前を与えて tdd-vocab に登録する。

**登録するノードの目安:**
ユーザーがその言葉を使ってドメインの話ができる概念——ルートノードと、
ドメインの現実に対応していると感じられる中間ノードが対象。
実装の都合上の細分（特定のアルゴリズム、一時的なヘルパー）は登録しない。

### 6. 上位ノードの変質を検知したとき

実装中に上位ノードの「できる」の性質が変わったと気づいたとき:

1. **problem.md と照合する**: 新しい合成は、problem.md の「できない」を解消するか？
   - **解消する**: 新しいXに名前を与え、ルートのテストを更新し、tdd-vocab に登録
   - **解消しない**: 何が足りないかを特定し、ツリーを再構築する
2. ユーザーに報告する

### 7. ルートが green になったことを確認

合成が完了したら、手順3のテストが green になることを確認する。

green になったらユーザーに確認を求める。
ユーザーが受け入れたら、wip 概念の stable 昇格候補を提示する。

### 8. 統合と動作確認（利用可能な状態にする）

テストの green だけで終わらせない。変更を**実際に使える状態**にする:

- 変更をユーザーが触る入口（UI・CLI・エンドポイント等）に組み込む
- 実アプリを動かして通しで動くことを確認する
- 組み込みに判断が要るのに情報が不足している場合は、妥当な既定で組み込んで findings に一言残す

### 9. findings.md の記録

| 発見の性質 | 戻し先 |
|-----------|--------|
| 問題の発見（「できない」の条件に見落とし） | /tdd-problem |
| ツリーの再構築が必要（分解・合成の見直し） | このスキルの次サイクル |

軽微なバグ修正・小さなリファクタリングでは生成しない。

```markdown
# Implementation Findings

**日時:** YYYY-MM-DD
**プロジェクト:** plans/<project>/

## 発見（戻し先つき）

### F-01: <タイトル>
**戻し先:** /tdd-problem（問題の発見） | このスキルの次サイクル（ツリーの再構築）
**現象:** ...
**検証で分かったこと:** ...
**推奨される対応:** ...
```

```
✅ 実装が完了しました。

📁 成果物:
- 実装コード
- テスト（「できる」のツリー）
- docs/wip/dictionary.md（tdd-vocab 経由で登録済み）

🚀 次のステップ:
- /tdd-feedback（実際に使ってフィードバックを得る）
- /tdd-vocab（wip の概念を dictionary.md に昇格する）

wip に残っている概念: [tdd-vocab より]
```

## 未解決課題（tdd-vocab 設計時に詰める）

- **登録対象ノードの基準**: 「ドメインの現実に対応している」の判断基準が曖昧。
  ユーザーとのコミュニケーションが成立するかどうかが基準になりうるが、
  どの階層までがドメイン語彙でどこから実装詳細かの線引きは未確定
- **セミラティスの共有ノード**: 複数の親を持つノードは自動的に登録対象と考えてよいか
- **既存語彙の組み合わせで解ける場合**: 新しいノードを作らず既存概念で分解した場合、
  ツリーの記述方法をどうするか
- **概念の再定義**: 実装中に既存概念の意味が変質したと気づいた場合の扱い
  （新概念として登録するか、既存エントリを更新するか、影響範囲分析が必要か）

---

## 制約

- **problem.md を書き直さない**（/tdd-problem の仕事）
- **docs/dictionary.md を直接編集しない**（/tdd-vocab の仕事）
- **ルートノードのテストを途中で green にしない**
- **テストは「できる」を宣言する最小限に**（写経的に書かない）
- **利用可能な状態まで持っていく**（本番デプロイは別フェーズ）
- **TodoWrite を使用する**（実装タスクを追跡）

---

## 成果物

1. **テスト（「できる」のツリー）** — ルートから末端まで、各ノードの「できる」を宣言
2. **実装コード** — テストが通り、入口に組み込まれた状態
3. **findings.md**（オプション）
4. **docs/wip/dictionary.md への追記**（tdd-vocab 経由）
