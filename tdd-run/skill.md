# /tdd-run - TDD実践スキル

あなたはTDD実践者です。
**問題とは「できない」状態であり、実装とはそれを「できる」に変換することです。**

---

## Auto モード

オーケストレーターから `mode: auto` を指定して呼ばれた場合、通常ユーザーに確認を求めるポイントで**自律的に決定して進む**。

変更されるポイント（各手順に `[auto]` で明記）:

| 手順 | 通常モード | auto モード |
|------|-----------|------------|
| 手順4 ツリー確定 | ユーザーに提示して承認を得る | 決定して進む |
| 手順4 利用仮説 | ユーザーと確認しながら書く | problem.md から自律的に作成する |
| 手順4 コンテキスト割り当て | ユーザーに提示して確認を得る | 決定して進む |
| 手順7 ルート green | ユーザーに確認を求める | 自動的に手順7.5へ進む |

auto モードで自律的に決定した事項は `plans/<project>/auto-decisions.md` に記録する。
オーケストレーターがセッション終了後にユーザーへ報告するための素材になる。

```markdown
# Auto Decisions: <プロジェクト名>

**日時:** YYYY-MM-DD

## ツリー構造
（決定したツリーと、なぜその分解にしたかの理由）

## モジュール境界
（各ノードの配置と選択の理由）

## 利用仮説
（problem.md から導出した仮説）
```

---

「できる」のツリーがあなたの地図です。ツリーの各ノードが一本のテストになります。
ルートノードが green になったとき、問題が解けています。

## worktree のセットアップ

セッション最初に行う（ファイル読み込みより前）。`<project>` は `plans/` 配下のディレクトリ名。

ブランチ `tdd/<project>` が存在するかで分岐する:

```bash
# 新規（初回）
git worktree add ./tdd/<project> -b tdd/<project>
mkdir -p ./tdd/<project>/plans
cp -r plans/<project> ./tdd/<project>/plans/

# 再利用（tdd-feedback の B「同一プランへの戻し」後）
git worktree add ./tdd/<project> tdd/<project>
```

再利用の場合、plans/ は前サイクルのコミットに含まれているためコピー不要。

以降の作業（ファイル読み込み・テスト・実装）はすべてこのworktree内で行う。

## セッション開始時に開くもの

```bash
cat plans/<project>/problem.md
cat plans/<project>/user-story.md 2>/dev/null
cat docs/dictionary.md 2>/dev/null
cat plans/<project>/dictionary.md 2>/dev/null
```

problem.md が無い場合: `/tdd-problem` で問題を定義してください。

**plans/<project>/dictionary.md が存在しない場合（新規 problem に取り組む場合）:**
ツリーを構成する前に `/tdd-vocab plan` を呼ぶ。
この時点での語彙定義は**作業仮説**なので `plans/<project>/dictionary.md` に書く
（`docs/dictionary.md` には書かない）。
wip の語彙が仮のガードレールとして機能する。
実装・受け入れを経た後、`/tdd-vocab promote` で stable に昇格する。

同一概念が docs/dictionary.md と plans/<project>/dictionary.md の両方にある場合、plans/ 側が優先です。
これらはセッション中、コードを書く行為の背景として常に存在します。

**.claude/tdd/scaffold.sh の確認:**
`.claude/tdd/scaffold.sh` が存在するかを確認する。存在しない場合は `/tdd-scaffold` を呼んで生成する。

## 「できる」のツリー

```
X は xxx ができる  ← ルートノード（問題解決の仮説）
├── A は aaa ができる
│   ├── D は ddd ができる
│   └── E は eee ができる
└── B は bbb ができる
    └── C は ccc ができる
```

各ノードは「名前は〜ができる」の形をしています。
**名前が先にあり、その名前が何をできるかが続きます。**
「X とは xxx ができるような何かだ」——この問いに答えることが、ノードを置く行為です。

**ノードの名前は日本語で書く。**
「X とは〜ができるような何かだ」という問いに答える概念名——まだ存在しない対象への宣言。
日本語で書くことで、実装の識別子（関数名・クラス名）がノード名に紛れ込むことを防ぐ。

**「できる」の述語は振る舞いを書く——手段は書かない。**

| NG | OK |
|----|-----|
| 展開状態マネージャーは localStorage でディレクトリの開閉を管理できる | 展開状態マネージャーはディレクトリの開閉状態を保持して管理できる |

ノードに技術名（API・ライブラリ・ストレージ種別・通信プロトコル等）が現れたら、
実装詳細がノードに漏れているシグナル。

手段の選択はドライバー（テストコード）の責務。
ノードが手段を先取りすると、ブラウザ外でのオンメモリ実装という選択肢が消える。

**ツリーはテストファイルです。**
E のテストに対応するコードを書くとき、X の振る舞いについて意識できていること——
これが近視眼的な実装を防ぎます。

ツリーはセミラティス状になってよい（ノードが複数の親を持つ）。
共有ノードは命名の契機です。

ツリーは理解の更新とともに変わります。
ただし、**上位ノードの性質が変質するときは problem.md との照合が必要**です（手順6）。

## 手順

### 1. 読み込み

セッション開始時に上記三ファイルを読む。

コードベースの調査（最小限）:
- 実装の差し込み位置
- 既存の関連実装・使えるもの

### 2. ルートXの定義と命名

problem.md を読んで問う: **「何ができれば、この問題は解けているか」**

**user-story.md が存在する場合:** 各シナリオの Then 節（期待される結果）を並べて見る。
ルートの「できる」は、これらの Then が成立するときに言えることの共通分母として現れる。
Then 節から逆算してルートを確認する——problem.md からの答えと一致しているかを照合する。
一致しない場合は、problem.md とユーザーストーリーのどちらかに見落としがある可能性を検討する。

**まず既存語彙で表現できないか確認する:**
docs/dictionary.md と plans/<project>/dictionary.md を参照し、
「この問題は、既存の概念の組み合わせで表現できるか？」を問う。
既存語彙で表現できるなら、新しい名前は作らない。

既存語彙で表現できない場合、新しいXを定義する:
「X とは xxx ができるような何かだ」という形で表現する。
名前は**日本語の概念名**で定める（例: `フロントマターテンプレートローダー`）。
名前が定まったら `plans/<project>/dictionary.md` のアプリケーションドメインに書く:

```markdown
### <概念名>

en: EnglishName
**定義:** xxx ができるもの
**関係:** （他の概念との関係を書く）
```

### 3. ルートのテストを書いて red のまま置く

「X は xxx ができる」のテストを書く。
このテストはツリーが完成するまで green にしない。

### 4. ツリーを分解し、骨格テストと語彙を確定する

「X が xxx できるためには何が必要か」を分解する。各ノードを一行で。
末端（直接実装できる粒度）まで続ける。

分解が終わったら、ツリーを確定する前に**技術語スキャン**を行う:

全ノードの述語（「は〜ができる」の〜の部分）を走査し、以下のカテゴリの語が含まれていないかを確認する:
- URL・エンドポイント・HTTPメソッド
- ライブラリ・フレームワーク名
- ストレージ種別（localStorage・DB・S3等）
- 通信プロトコル・ファイルフォーマット

含まれていれば「その述語が表す振る舞いは何か」を問い直し、技術語を排除した述語に書き直してからツリーを確定する。

次に**scaffolding可能性チェック**を行う:

全ノードの主語が日本語の概念名として成立するかを確認する。
URLパス・技術的な操作の記述・camelCase の識別子が紛れていれば、概念として語れる名前に問い直す。

このチェックは「テストが書けない」という判断が実装フェーズで起きる前に、その原因を構造的に排除する。

次に**ロード可能性確認**を行う:

各ノードの配置先モジュールが、テスト実行環境でロードできるかを確認する。
以下に該当するモジュールはロード不可とみなす:
- テスト実行環境にないランタイム依存（ブラウザ API・GUI フレームワークのコンテキスト等）を直接参照する
- ロード時に副作用（イベントリスナー登録・画面操作等）が走る

ロード不可のノードがある場合、以下の2択を決定してから次に進む:
1. **ロジックを別モジュールに抽出する** — テスト環境でロード可能なモジュールに純粋ロジックを移し、コンテキスト割り当てを変更する
2. **そのノードの単体テストを書かない** — `it()` を「手動確認のみ」と明示してスキップする

「テストファイル内にロジックをコピーして検証する」は選択肢に含めない。

**分解が終わったらユーザーに提示し、確認を得てから次に進む。**
**[auto] ユーザーへの提示なしに進む。決定したツリーを `auto-decisions.md` に記録する。**

確認が取れたら、まず `plans/<project>/test-tree.md` を生成する。

**test-tree.md の生成:**

ツリーの確定と同時に、このセッションが実現しようとしていることを利用仮説として書き留める。
利用仮説は「使ったらこうなるはず／こうなったら外れ」の形で、実装前にコミットしておく。

**user-story.md が存在する場合:** シナリオの Given/When/Then を素材にする。
各シナリオが「使ったらこうなるはず」の一項に対応する。シナリオが通らない条件が「こうなったら外れ」になる。
**user-story.md が存在しない場合:** problem.md の「解決したと言える状態」を素材にする。

ユーザーと確認しながら書く。
**[auto] ユーザーとの確認なしに上記素材から自律的に作成する。**

```markdown
# テストツリー: <プロジェクト名>

**作成:** YYYY-MM-DD

## できるのツリー

X は xxx ができる
├── A は aaa ができる
│   ├── D は ddd ができる
│   └── E は eee ができる
└── B は bbb ができる

## 利用仮説

- 使ったらこうなるはず: ...
- こうなったら外れ: ...
```

test-tree.md が作れたら、ツリーの名前を起点として2つを同時に行う。

**テスト骨格の生成:**

テストファイルは `tests/<dir>/` に置く（`<dir>` は tdd-vocab plan で確定したコンテキストの英語 dir 名）。
ツリーの各ノードに対応する `describe()` ブロックを、入れ子構造を保ったままテストファイルに書く。
`it()` の中身は空（TODO）のまま。

テストの `describe()` の主語には日本語のノード名をそのまま使う:

```javascript
describe('フロントマターテンプレートローダーは xxx ができる', () => {
  describe('テンプレートマッチャーは aaa ができる', () => {
    it('TODO', () => {})
  })
  describe('フロントマター文字列ビルダーは bbb ができる', () => {
    it('TODO', () => {})
  })
})
```

この時点でツリーとテストファイルは1対1に対応している。
**ノードが骨格に現れないまま実装に進むことはしない。**

**スタブの生成（scaffolding）:**

ツリーの各ノードに対して `.claude/tdd/scaffold.sh` を呼ぶ:

```bash
.claude/tdd/scaffold.sh <Subject_en> <verb_en> <context>
```

- `Subject_en`: ノードの S を語彙の `en:` フィールドで解決した英語識別子（PascalCase）
- `verb_en`: ノードの V を直訳した英語 camelCase 識別子
- `context`: tdd-vocab で確定したコンテキスト dir 名

各ノードは独立したファイルに対応する。同じ S を持つ複数ノードがある場合、スクリプトは既存ファイルに関数を追記する。

`describe()` の日本語名と実装の英語識別子は対応している——概念の宣言（日本語）と識別子の宣言（英語）が別フェーズで行われる。
スタブが存在することで、実装フェーズで「どの関数に書くか」を決める必要がなくなる。

**コンテキストの割り当て:**

すべてのノードは独立したファイルに配置する。確認するのはファイル配置ではなく**コンテキスト**のみ:

| ノード | コンテキスト |
|--------|------------|
| X | `shop` |
| A | `shop` |
| B | `payment` |

セミラティスの共有ノードも独立ファイルとして扱う。ファイルが増えすぎる懸念はコンテキストを狭く定義することで対処する。

確認が取れたら次に進む。
**[auto] ユーザーへの提示なしに進む。コンテキストの割り当てを `auto-decisions.md` に記録する。**

**ソリューションドメイン語彙の登録:**

ノード名のうちソリューションドメインに相当するもの（装置・変換・操作の名前）を
`plans/<project>/dictionary.md` のソリューションドメインに書く。
除外するもの: ルートノード、既にアプリドメイン語彙として登録済みの概念、実装詳細の細分。

```markdown
### <ノード名>

**定義:** <何をする装置か（暫定）>
**関係:** （ウォークスルーで補完）
```

アプリドメイン概念との対応関係はウォークスルー（手順7.5）で確定させる。

### 5. 末端から実装して上へ合成する

末端ノードから順に red → green → refactor を繰り返す。
green になったら一つ上のノードへ移り、合成する。

各ノードのテストの書き方:

```javascript
describe('A は aaa ができる', () => {
  it('xxx したとき yyy になる', () => {
    // 実装より先にインターフェースを定める
  })
})
```

**実装への参照:**

`@vocab` と `@test` アノテーションは手順4のscaffoldingで既に書いてある。
実装を埋める際に削除・変更しない。

**テスト実行コマンド:**

```bash
npm test -- <test-file-path>   # または bun test <file> 等、検出した方式で
```

**ループ防止（5回連続失敗時）:**

```
⚠️  5回のイテレーションで通りませんでした。

次のアクションを選んでください:
- 続ける
- 一時中断（findings.md を生成して問題定義へ）
- 分解の仕方を見直す
```

**命名の契機を逃さない:**
以下が見えたとき、立ち止まる:

- 複数のノードがまとまって「〜の流れ」として捉えられるとき
- セミラティスの共有ノードに名前がないとき
- 「この関数群は何をしているか」に一言で答えられるとき

立ち止まったら、**まず既存語彙で表現できないかを確認する**。
既存語彙で表現できなければ、新しい名前を与えて `plans/<project>/dictionary.md` に書く。

**登録するノードの目安:**
ユーザーがその言葉を使ってドメインの話ができる概念——ルートノードと、
ドメインの現実に対応していると感じられる中間ノードが対象。
実装の都合上の細分（特定のアルゴリズム、一時的なヘルパー）は登録しない。

### 6. 上位ノードの変質を検知したとき

実装中に上位ノードの「できる」の性質が変わったと気づいたとき:

1. **problem.md と照合する**: 新しい合成は、problem.md の「できない」を解消するか？
   - **解消する**: 新しいXに名前を与え、ルートのテストを更新し、`plans/<project>/dictionary.md` に書く
   - **解消しない**: 何が足りないかを特定し、ツリーを再構築する
2. ユーザーに報告する

### 7. ルートが green になったことを確認

合成が完了したら、手順3のテストが green になることを確認する。

green になったらユーザーに確認を求める。
ユーザーが受け入れたら、手順7.5 に進む。
**[auto] ユーザーへの確認なしに手順7.5へ進む。**

### 7.5. 辞書・テスト・実装のウォークスルー

名前がフェーズをまたいで一貫しているかを確認し、`plans/<project>/dictionary.md` を更新する。

まず照合表を作る:

```
語彙（plans/dictionary）  | テスト describe()  | 実装（関数・モジュール名）
--------------------------|-------------------|---------------------------
ツリービルダー             | なし               | buildTree()
ネスト変換                 | ✓                 | buildTree() 内部
ツリーレンダラー           | ✓                 | renderTreeHtml()
```

照合する3点:

- **語彙 → テスト**: 語彙に登録された名前が `describe()` に現れているか。
  消えた名前は「静かに解消された」ので findings に記録する。
- **テスト → 実装**: `describe()` の主語名が実装の関数・モジュール・クラス名と一致しているか。
  手順4のscaffoldingで名前は固定されているはずなので、ここで乖離があれば手順4の段階で変換が起きている。
  乖離を発見したら findings に記録し、実装を正しい名前に修正する。
  一致しているなら語彙エントリの「関係」フィールドに実装参照を書き足す。
- **実装 → 語彙**: 実装に現れた装置名のうち語彙未登録のものがあれば追加を検討する。
- **語彙 → src**: 確定した実装パスを各語彙エントリの `**src:**` フィールドに記入する。
  手順 4 のモジュール境界確認で決めたファイル配置と一致しているかも確認する。

### 8. 統合と動作確認（利用可能な状態にする）

テストの green だけで終わらせない。変更を**実際に使える状態**にする:

- 変更をユーザーが触る入口（UI・CLI・エンドポイント等）に組み込む
- 実アプリを動かして通しで動くことを確認する
- 組み込みに判断が要るのに情報が不足している場合は、妥当な既定で組み込んで findings に一言残す

### 8.5. ユーザーストーリーテストの実行

統合と動作確認が完了したら、`/tdd-userstory run <project>` を実行する。
`plans/<project>/user-story.md` が存在しない場合はスキップしてよい。

**結果に応じた対応:**

| 結果 | 対応 |
|------|------|
| `pass` / `pending` | 手順9へ進む |
| `fail`（アサーション失敗） | 失敗内容を findings.md に記録して手順9へ進む（後述） |
| `execution-error` | エラー対応ループへ（後述） |

**`execution-error` のとき — 対応ループ（最大3回）:**

1. エラー内容を分析し、対応策を考える
2. 通常モード: 分析結果と対応策をユーザーに提示し、承認を得てから実施する
   **[auto]**: 確認をスキップし、対応策を自動的に実施する
3. `/tdd-userstory run <project>` を再実行する
4. `execution-error` が続く場合は 1〜3 を繰り返す（最大3回）

3回試みても `execution-error` が解消されない場合:

```
⚠️ ユーザーストーリーテストを3回試みましたが実行できませんでした。

エラー種別: <種別>
最後のエラー: <エラーメッセージ>
試みた対応策:
  1回目: <対応内容>
  2回目: <対応内容>
  3回目: <対応内容>

ユーザーストーリーテストをスキップして次のステップへ進みます。
```

この内容を findings.md に記録して手順9へ進む。

**findings.md への記録フォーマット:**

`fail` の場合:

```markdown
### F-XX: ユーザーストーリー受け入れテストの失敗
**戻し先:** /tdd-problem（問題定義の見直しが必要な場合）
```

## 制約

- **problem.md を書き直さない**（/tdd-problem の仕事）
- **docs/dictionary.md を直接編集しない**（/tdd-vocab promote 経由のみ）
- **ルートノードのテストを途中で green にしない**
- **テストは「できる」を宣言する最小限に**（写経的に書かない）
- **テストはロジックの実装をロードして検証する** — テストファイル内に定義したローカル関数を検証するテストは書かない
- **利用可能な状態まで持っていく**（本番デプロイは別フェーズ）
- **TodoWrite を使用する**（実装タスクを追跡）
- **ツリーのすべてのノードを骨格テストに書いてから実装に入る**
- **ノードを省略する判断は、骨格テストの `it()` を書こうとした後に行う**（書く前に省略しない）
