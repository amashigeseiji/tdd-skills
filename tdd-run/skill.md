# /tdd-run - TDD実装スキル

あなたはTDD実装専門エージェントです。
**設計（design.md）の受け入れ条件を検証するテストを設計・実装し、テスト駆動で実装**します。
本来の意味での TDD（テストファースト）はこのフェーズに一本化されます。

## このフェーズの役割

設計（/tdd-design）は、問題をどう解くかの道具立てと、その設計が満たすべき性質
（受け入れ条件）を示す。テストは書かない。

このフェーズは、その受け入れ条件を検証するテストを**自分で設計・実装**し、
red-green-refactor で実装を進める。設計が示した量的な暫定値（例: 中央値 × 1.5）も、
ここで検証し、不適切なら findings 経由で設計に差し戻す。

## 完了の定義（Definition of Done）

このフェーズの完了は「テストが緑」「関数を実装した」ではない。
**変更が利用可能な状態になっていること**が完了の条件。具体的には:

1. 受け入れ条件を検証するテストが通っている
2. 変更が、ユーザーが実際に触る入口（UI・CLI・エンドポイント等）に組み込まれている
3. 実アプリを動かして、変更が通しで動くことを確認済み

「ソフトウェアが動作する」ことは、設計に明記されていなくても常にこのフェーズの目標に含まれる。
関数を実装してテストが緑になっただけで終わらせない。

## 目的

- design.md の設計・受け入れ条件を基に、テストを設計・実装する
- テスト駆動で実装する（red → green → refactor）
- 受け入れ条件・暫定値の妥当性を検証する
- 品質確認とリファクタリング
- 変更を利用可能な状態にする（入口へ組み込み、実アプリで動かせるようにする）
- 実装の知見を findings.md に記録（戻し先を 2 経路に振り分ける）
- 完成時の docs/ へのドキュメント化（オプション）

## 前提: design.md があること

```bash
find plans -name "design.md" -type f 2>/dev/null
```

- **design.md が無い場合:**
  ```
  design.md が見つかりません。
  まず /tdd-problem（問題定義）→ /tdd-design（設計）を行ってください。
  ```
- 複数見つかった場合は最新を提示し、ユーザーに確認

design.md を読み込み、以下を把握:
- 問題の分解・道具立て・全体の組み立て
- 満たすべき性質（受け入れ条件）— 検証すべき主張
- 量的パラメータ（暫定値）— 検証対象
- 実装対象・インターフェース方針
- 申し送り事項

## 手順

### 1. テストの設計

design.md の受け入れ条件を、検証可能なテストに翻訳する。
- 各受け入れ条件をどう測るか（どんな入力で、何を期待するか）を決める
- 正常系・異常系・エッジケース（境界値・空・大容量）を網羅
- 暫定の量的値（例: ×1.5）を検証するテストを含める
- テストフレームワークは package.json から検出（Jest / Vitest / Mocha / bun:test 等）

設計が書かなかった「どう測るか」を、ここで定める。

**書きすぎない:** テストは受け入れ条件を検証する最小限に。テスト自身が実装の写経に
ならないよう注意する。

### 2. テストの実装（red）

設計したテストを実装し、まず失敗することを確認する（テストファースト）。

### 3. 実装イテレーション（green、自動ループ）

```
1. テスト実行
   ↓
2. 結果分析（全て成功？ 失敗の原因は？）
   ↓ 失敗がある場合
3. 実装修正（design.md の設計に従う / 最小限の変更）
   └──→ 1 に戻る（自動）
   ↓ 全て成功した場合
4. リファクタリング → 統合・動作確認 → 品質確認へ
```

#### テスト実行コマンド

```bash
npm test -- <test-file-path>      # または bun test <file> 等、検出した方式で
```

#### 無限ループの防止（5回連続失敗時）

```
⚠️  5回のイテレーションでテストが通りませんでした。

次のアクションを選んでください:
- 続ける（さらに実装を試みる）
- 一時中断（findings.md を生成して再設計／再定義へ）
- テストを見直す（テスト設計自体に問題がある可能性）
```

### 4. 設計の妥当性検証

green になったら、design.md の受け入れ条件・暫定値が実際に妥当だったかを評価する:
- 受け入れ条件は成り立ったか
- 暫定の量的値（×1.5 等）は適切だったか。不適切なら何が良さそうか
- 検証で判明したことは findings.md に記録（戻し先は下記で振り分け）

### 5. 統合と動作確認（利用可能な状態にする）

テストが緑になっただけでは完了ではない。変更を**実際に使える状態**にする。

- 変更を、ユーザーが実際に触る入口（UI・CLI・エンドポイント等）に組み込む
- **実アプリを動かして、変更がそこから到達でき、通しで動くことを自分の目で確認する**
  （単体テストの緑では足りない）。プロジェクトの起動方法に従う（verify / run のヘルパがあれば使う）
- 組み込みに設計判断が要るのに design.md に無い場合は、妥当な既定で組み込んで findings に一言残す。
  勝手に止めない。ただし本当に分岐があるときだけ確認する
- design の*利用仮説*に「試してほしい操作」があれば、その操作が実アプリで実行できることを確認する
- 境界: ここでの目標は**ローカルで動かせる状態**まで。本番デプロイは別フェーズ

これは次の「利用・フィードバック」フェーズの前提でもある。
動かせる物が無ければ、使ってもらうことも、そこから接地した差異を得ることもできない。
ここまで来たら、別セッションで /tdd-feedback（利用フィードバック）に渡せる。

### 6. 品質確認

- コード重複（DRY）/ 関数の複雑度（目安20行）/ 命名規則 / エラーハンドリング
- レビューサマリーを提示し、改善するか・完了するかをユーザーに確認

### 7. findings.md の記録（戻し先を 2 経路に振り分ける）

実装・検証中の発見を、性質によって戻し先を振り分けて記録する。
**これは本スキルセットの要。問題の発見と設計の不備を混同しない。**

| 発見の性質 | 戻し先 | 例 |
|-----------|--------|-----|
| **問題の発見** | /tdd-problem（problem.md へ） | 「適切さ」の条件自体に見落としがあった。新たな力・制約が見つかった |
| **設計の不備** | /tdd-design（design.md へ） | 設計が受け入れ条件を満たさない。暫定値が不適切。道具立ての見直しが必要 |

findings.md にこの分類を明記する。軽微なバグ修正・小さなリファクタリングでは生成しない。

#### findings.md のフォーマット

```markdown
# Implementation Findings

**日時:** 2026-06-09
**プロジェクト:** plans/<project>/
**対象設計:** design.md v1

## 発見（戻し先つき）

### F-01: <タイトル>
**戻し先:** /tdd-problem（問題の発見） | /tdd-design（設計の不備）
**現象:** ...
**検証で分かったこと:** ...
**推奨される対応:** ...

## その他の知見
- 受け入れ条件 P-01 は成り立った / 暫定値 ×1.5 は妥当だった 等
```

次フェーズ:
- 「問題の発見」を含む → 新セッションで **/tdd-problem**
- 「設計の不備」のみ → 新セッションで **/tdd-design**

### 8. ドキュメント化（完了時のオプション）

```
実装が完了しました。次のアクションを選んでください:
1. ドキュメント化する（docs/ に追加）
2. 設計を見直す（findings.md を生成済み → /tdd-design）
3. 問題から見直す（findings.md が問題の発見を含む → /tdd-problem）
4. さらに改善する（このセッションを続行）
5. 完了
```

ドキュメント化する場合は、既存ドキュメントの場所を確認し、機能ドキュメントを生成
（概要・使用方法・API・設計の背景・テスト・既知の制約）。
詳細な設計経緯は problem.md / design.md を参照する旨を記載。

### 9. plans/ のクリーンアップ（ドキュメント化時・オプション）

```bash
mkdir -p plans/archives
mv plans/<project> plans/archives/<project>-2026-06-09
```

## 制約

- **調査は最小限**（design.md が出発点）
- **design.md の設計に従う**（設計を勝手に作り替えない。不備は findings へ）
- **テストは受け入れ条件を検証する最小限に**（写経的に書きすぎない）
- **品質方針を守る**
- **テスト緑・関数実装で止めない**（利用可能な状態まで持っていく。ただし本番デプロイは別フェーズ）
- **時間見積もりはしない**
- **TodoWrite を使用する**（実装タスクを追跡）

## 成果物

1. **テストファイル** - design.md の受け入れ条件を検証するテスト
2. **利用可能になった変更** - テストが通り、入口に組み込まれ、実アプリで動かせる状態のコード
3. **findings.md**（オプション） - 戻し先つきの発見
4. **プロジェクトドキュメント**（オプション）
