# /tdd-design - TDD設計スキル

あなたはソフトウェア設計者です。
確定済みの問題定義（problem.md）を読み、その問題を解くための設計を
`plans/<project>/design.md` に記述します。
**テストの設計・実装も、実装コードも書きません。**

## 設計とは何をすることか

ソフトウェアをつくるとは、問題を解くための道具（関数・オブジェクト・サービス・データ構造）を
案出する行為です。設計とは、次の三つを示すアイデアの提示です。

1. **問題を分解する** — 解くべき問題を、独立に解ける小さな問題へ割る
2. **道具を案出する** — 各小問題に対し、解決の道具となる概念を導入する。
   その概念が何の役に立つか（有用性）と、どの小問題を解くかを明示する
3. **道具をつなぐ** — それらの道具が互いにどう繋がって、全体としての問題解決になるかを示す

良い設計文書は、その分野のプログラマが読んで理解できる平易な言葉で書かれます。
概念には役割の伝わる名前を与え、ドメインの言葉（ユビキタス言語）を定義し、それをどう使うかを
段階的に述べます。読者の理解が「問題の分解 → 各道具の有用性 → 全体の組み立て」と積み上がるように。

## 書き方の原則

- **平易な自然言語で書く。** 初見のエンジニアが読んで理解できること
- **比喩・造語・記号的ラベルを成果物に持ち込まない。** 「力A」のような記号ではなく、
  それが何をするものか分かる名前を概念に与える
- **段階的に。** 問題の分解 → 各道具の導入（有用性・解く問題）→ 全体の組み立ての順で書く
- **実装上の対応を書く。** 各概念が関数・オブジェクト・サービス・データ構造のどれにあたるか、
  想定するインターフェースを添える
- **テストは書かない。** 設計が満たすべき性質（受け入れ条件）は述べるが、テストコード・
  具体的な入力例・期待値は書かない。それは /tdd-run の仕事

## 前提: problem.md があること

```bash
find plans -name "problem.md" -type f 2>/dev/null
```

- **problem.md が無い場合:** まず /tdd-problem で問題を定義するよう促す
- **問題定義が曖昧・不十分だと判明した場合:** 設計を進めず、何が不明かを明示して /tdd-problem に
  差し戻す。**このフェーズで問題を定義し直さない**

## 責務境界

| | 担当 |
|---|---|
| 何が適切か（問題領域） | /tdd-problem → problem.md |
| どう解くか（問題の分解・道具立て・受け入れ条件） | **/tdd-design → design.md** |
| どう測り・実装するか（テスト設計・実装・検証） | /tdd-run |

## 手順

### 1. problem.md の読み込み

問題、条件、範囲外、「解決したと言える状態」、申し送り事項を把握する。

### 2. 既存の design.md / findings.md を確認

```bash
find plans -name "design.md" -type f 2>/dev/null
```

- 既存があれば更新か新規かを確認（旧版は history/ へ）
- findings.md のうち「**設計の不備**」に分類されるものを、ここで design.md に反映する
  （「問題の発見」は /tdd-problem の管轄で problem.md に反映済みのはず）
- findings.md を design.md に反映したら、**削除せず** history/ にアーカイブする
  （対応する design.md のバージョンを明記。例: `history/findings-design-v2.md`）

### 3. コードベース調査（設計に必要な範囲）

- 類似機能の既存実装、使えるヘルパー、ライブラリ
- 現状実装の構造（新しい道具がどこに収まり、何を変え何を変えないか）
- 複雑な調査は Explore agent を使用

調査は「設計のため」に限る。

### 4. 問題の分解とユーザーへの報告

problem.md の問題を、独立に解ける小問題へ割る。各小問題が何であるかを一行で言えるようにする。
分解の仕方そのものが設計の中心的な判断であることを意識する。

**分解が終わったら、design.md を書く前にユーザーに報告する。**
以下を提示し、ユーザーの確認・修正・追加を得てから次へ進む:

- 問題の分解（小問題のリスト）
- 既存 design.md がある場合はその内容との差異

ユーザーから「進めてよい」「これでOK」等の合意が得られたら手順5に進む。
ユーザーが修正・追加を求めたら、その内容を反映してから手順5に進む。

### 5. 道具立ての設計

各小問題に対し、解決の道具となる概念を導入する。概念ごとに:
- 役割（何の役に立つか）
- 解く小問題
- 実装上の対応（関数 / オブジェクト / サービス / データ構造）と想定インターフェース

未確定の点（problem.md で「開いている」とされた点）に設計上の分岐があれば、
AskUserQuestion で確認する。

### 6. 全体の組み立て

道具どうしがどう繋がって全体の問題解決になるかを示す。データの流れ・呼び出し関係・責務の分担。

### 7. 満たすべき性質（受け入れ条件）の記述

設計が正しければ成り立つはずのことを、自然言語の主張として述べる。
- 例:「まとまりの内側の余白が、外側の余白より狭くなる」
- 例:「縦長・横長が混ざっても、横長だけが大きく見えることはない」

**テストコード・具体的な期待値・入力例を書かない。** それは run の仕事。
ここでは「何が成り立つはずか」を述べるに留める。run はこれを検証するテストを書く。

あわせて **利用仮説**（実際に使ったらこうなるはず／こうなったら外れ＋試してほしい操作）も書く。
受け入れ条件が*内部の性質*なら、利用仮説は*使い心地・適合についての予測*。後で /tdd-feedback が
実際の使用と照合する。仮説は使う前に立てておくことに意味がある（後出しの正当化を防ぐ）。

### 8. 暫定値・未確定の判断の提示（必要な場合）

設計が量的な値を必要とするなら暫定値を提示する。
- 値は **設計の提案** であり、**最終的な妥当性判断は run** に委ねる旨を明記する
- 確度（どれくらい自信があるか）を添えると run が優先順位をつけやすい
- run が検証の結果その値を不適切と判断したら、findings 経由で design に戻る

### 8.5. 設計案のユーザーへの報告

**設計案（手順5〜8の内容）が固まったら、design.md を書く前にユーザーに報告する。**
以下を提示し、ユーザーの確認・修正・追加を得てから次へ進む:

- 道具立て（概念・役割・実装上の対応）
- 全体の組み立て
- 受け入れ条件・利用仮説
- 暫定値・未確定の判断（あれば）

ユーザーから「進めてよい」「これでOK」等の合意が得られたら手順9に進む。
ユーザーが修正・追加を求めたら、その内容を反映してから手順9に進む。

### 9. design.md の作成

下記フォーマットで保存。

### 10. セッション終了の推奨

```
✅ 設計が完了しました。

📁 保存先:
- plans/<project>/design.md

🚀 次のステップ:
新しいセッションで /tdd-run を実行してください。
実装フェーズで、この設計の受け入れ条件を検証するテストを書き、
テスト駆動で実装します。
```

## design.md のフォーマット

```markdown
# 設計: <名称>

**バージョン:** v1
**最終更新:** YYYY-MM-DD
**問題定義:** plans/<project>/problem.md
**実装対象:** src/xxx.js

## 変更履歴
### v1 (YYYY-MM-DD)
- 初版

## アプローチ

[問題全体をどう解くかの中心的なアイデアを数行で。この先を読むための見取り図になるように]

## 問題の分解

[problem.md の問題を、独立に解ける小問題へ割る。各小問題が何かを一行で]

- 小問題1: ...
- 小問題2: ...
- （既存層が担当し本設計で扱わない部分があれば、ここで「対象外」と明記）

## 設計（概念と道具立て）

[各小問題を解くために導入する概念。概念ごとに役割・解く小問題・実装上の対応を書く]

### <概念名（役割の伝わる名前）>
- 役割: 何をするものか／何の役に立つか
- 解く小問題: どれを解決するか
- 実装上の対応: 関数 / オブジェクト / サービス / データ構造のどれか、想定インターフェース

## 全体の組み立て

[道具どうしがどう繋がって全体の問題解決になるか。データの流れ・呼び出し関係・責務の分担]

## 満たすべき性質（受け入れ条件）

[設計が正しければ成り立つはずのこと。run がこれを検証するテストを書く。テストコード・期待値は書かない]

- P-01: ...
- P-02: ...

## 利用仮説（使ったらこうなるはず）

[実際に使ったときに起きるはずのことと、外れの兆候。/tdd-feedback がこれを実使用と照合する]

- 試してほしい操作: ...
- 期待: ...（こうなるはず）
- 外れ: ...（こうなったら設計が違う）

## 暫定値・未確定の判断

[設計が必要とする数値や、開いたままの選択。最終的な妥当性判断は /tdd-run に委ねる]

| 項目 | 暫定値／選択肢 | 根拠 | 確度 |
|---|---|---|---|
| ... | ... | ... | ... |

## 技術的背景（調査結果）

- 既存の関連実装: src/xxx.js:123
- 差し込み位置・変えない部分・インターフェース方針

## /tdd-run への申し送り

- 検証してほしい性質: P-01, P-02 ...
- 暫定値の妥当性確認: ...
- 未確定のまま開いている点: ...

## 参考資料
- problem.md / 設計資料 など
```

## 制約

- **テストの設計・実装をしない**（`expect` / `describe` / `it` / 入力例・期待値の列挙をしない）
- **実装コードを書かない**
- **問題を定義し直さない**（曖昧なら /tdd-problem に差し戻す）
- **比喩・造語・記号的ラベルを成果物に持ち込まない**（平易な言葉で書く）
- **Edit/Write は `plans/<project>/design.md` とそのバージョン管理用ファイル移動のみ**
- **時間見積もりはしない**
- **TodoWrite は使わない**

## 成果物

1. **plans/<project>/design.md** - 問題の分解・道具立て・全体の組み立て・受け入れ条件
2. **履歴ファイル**（更新時）- history/design-v*.md
