# 辞書設計とパターン層についての雑考

**日付:** 2026-06-30

まだ結論を出していない問いと、途中で固まった判断の記録。

---

## 固まった判断

### pattern エントリーは通常エントリーと同じ形（Option B）

`{"context": [], "entries": [], "pattern": []}` のように外部構造として分けるOption A と、`domain: "pattern"` として通常エントリーに混ぜるOption B を検討し、B に決定。

理由：辞書の自由度（`definition` や `relations` で任意の構造を表現できる）がそのまま活きる。Ruby における「クラスもまたオブジェクトである」と同型の整理。既存の `dict-search.js` 資産がそのまま使える。

Option A の利点（投影関係の明示性、パターン定義の厳密化）は認めつつも、B の柔軟性が上回ると判断した。

---

## 未解決・後回しにした問い

### application / solution ドメインの区別は本当に意味があるか

コプリエン的に言えば、ソリューションドメインとはパラダイムのことである。OO か FP かといった言語の構造から来るパラダイムが、問題を表現可能な形に変換する枠組みを提供する。この観点では、パターンこそがソリューションドメインの語彙である。

さらに言えば、ドメインモデル（DDDが語るもの）はすでにある観点から見た有用な抽象であり、ドメインを直接見ているわけではない。ユビキタス言語は問題を直接語る言語ではなく、それを通じてドメインを見るための眼鏡である。git と svn で「コミット」の意味が異なるように、概念の意味付けがそのソフトウェアの問題解釈そのものを表している。

こうなると application と solution を分けることの意味が揺らぐ——すべては何らかのソリューション語彙に他ならないのだから。

この問いはいったん保留。現在の三値（application / solution / pattern）はそのまま使いつつ、後で設計を見直す可能性がある。

### pattern の「投影関係」をどこに記述するか

「この語彙はステートマシンにおける State です」という情報——どのパターンのどの役割を担うか——は現在の設計では明示されない。`references: ステートマシン` はパターンの採用を記録するが、役割（role）は記録しない。

既知パターンはAIの知識で補完できるが、独自パターンではこの情報が要るかもしれない。Option B の枠組み内では `references: { pattern: "ステートマシン", as: "状態" }` のような拡張が考えられるが、まだ設計していない。

### Shadow DOM / Light DOM アナロジー

投影関係を整理するために持ち出したアナロジー。パターンが Shadow DOM（テンプレート）、エントリーが Light DOM（具象化）。クラスとインスタンスの関係とも言える。

このアナロジー自体は有効だが、最終的に Option B を選んだことで、この区別はスキーマではなく意味的な区別として残ることになった。
