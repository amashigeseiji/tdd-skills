# 2026-06-10 設計は対話である — TDDスキル再設計の対話

このメモは、`docs/rationale.md` を生んだ対話の記録（探索の沈殿）。結論だけでなく、
*どの訂正がどの転換を生んだか*を残す。要約では落ちる「対話のテクスチャ」がここにある。

> このスキル群は、ある画像レイアウト生成ツールの開発を題材に練られた。以下の「layout」
> 「画像」等はその題材を指す一般化した呼び方。

## きっかけ

`/tdd-design` で layout の design.md を書いたら、難解だった（「力A への対応」「占有率ρ／凝集比γ」
など）。発端は tdd-design スキルの「理論家／実験家」という科学メタファーが、思考の枠ではなく
*出力の文体*にまで漏れていたこと。

## 対話の流れと、転換を生んだ訂正

1. **メタファー撤去 → 設計観の言語化。** 「あなたはソフトウェア設計者」に。設計＝問題を小問題へ
   分解し、道具（関数・オブジェクト）を案出して繋ぐこと、を平易に書く方針へ。DDD のユビキタス
   言語、ダイクストラの問題分解が背景。スキル（design/run/README）を全面改修。

2. **「力・条件」は残す。** ユーザーがこの Alexander 的な表現を好む（問題空間を力で語る）。
   → 問題空間＝力、解決空間＝道具、という使い分けに整理。

3. **アレグザンダー前期の輸入は致命的か。** ユーザーが鋭い問いを出す: design の「問題ツリー＋各
   ノードに解」は、Alexander が「都市はツリーではない」で否認した前期そのものでは。
   → 合意: 分解は*診断であって生成ではない*。実際の design.md も targetHeight 結合や閉/開フォーム
   分岐で semilattice になっており、純粋なツリーではない。留保が成立。

4. **積み上がらない問題。** 分解-解決はパラダイム内の問題解決で、枠を更新しない。deep model
   （Evans の share モデル）は再編成＝パラダイム転換で、分解は生めない。
   → 鍵: deep model を*生成する*手続きは作れないが、*試みる契機*は儀式化できる（後期 Alexander
   の unfolding、Schön の reflective conversation）。記録の蓄積 ≠ 理解の蓄積。

5. **接地と、AI を差異の源にする危険（Claude が point 3 を訂正）。** 当初 Claude は「AI を対話相手に
   するのは固有の失敗モード」と書いた。ユーザーが反論: AI=Evans、ユーザー=ドメインエキスパートで、
   構造は同じでは。→ Claude が撤回・精密化: 同じ構造で、残るのは*程度と傾向*の差（賭けを事実の
   文法で出す／抵抗を出し惜しむ）。**AI は差異の源でなく引き出し役**に置く、が正しい言い直し。

6. **TDD は知識獲得。** ユーザー: red/green の適合が TDD の核ではなく、Beck の triangulation・
   Cunningham の負債のように*失敗して知識を得る*ことが核。AI 文脈では文書化が要。findings はその器。
   → コードは使用で清算される仮説。**緑 ≠ 完了**。

7. **利用フェーズが最大の欠落。** 使ってみる工程がスキルに無い。→ tdd-run の責務を「利用可能な
   状態にする」まで拡張（完了の定義・統合と動作確認を追加）。

8. **インタビューと役割分離。** ユーザーが書くのは面倒 → Claude がインタビューする。役は UX
   リサーチャー（開発者と別）。別セッションで、設計に肩入れしない立場。

9. **一人シミュレーションという正しい枠（ユーザーが訂正）。** Claude の「あなたと、実際の雑誌編集者
   は別物」は*正しいが射程外*。このスキルは実地適合ではなく*開発フローのシミュレーション*。
   ユーザーが一人で回し、Claude に役を割り当てる。→ 役割分離は「市場代表性」でなく「視点の独立」、
   利用仮説は自作自演の共謀を止める「使用版テストファースト」として*必須*に。

10. **インタビューの具体像。** まず操作の事実を集める。利用仮説に「試す操作」を添えると楽。
    「デカい」の*解釈はユーザー*でよい（自分の印象に特権的アクセス）。Claude の役は*解像度を
    上げる質問*。良いインタビューは生の印象から problem.md を析出させる。

11. **言葉と運用の詰め。**「賭け」→「仮説（利用仮説）」。feedback.md と findings.md は1ファイルに
    （事実の節／解釈の節を分ける）。findings は削除せず history へアーカイブ。tdd-feedback の
    呼称取り違え（私／あなた）を修正（Claude＝あなた、人間＝ユーザー）。

## この対話で作った／変えたもの

- 新スキル `tdd-feedback`（利用フィードバック）。4フェーズ化。
- tdd-design: 利用仮説を追加。tdd-run: 完了の定義＝利用可能な状態。
- 全スキルから科学メタファーを撤去、平易な語彙へ。
- リポジトリ化（`~/dev/tdd-skills`）、`CLAUDE.md`／`docs/rationale.md`／`docs/notes/`。
