# 2026-06-16 tdd-refactor の設計

## 発端

tdd-run においてリファクタリングの契機がほぼ発生しない問題があった。
ステップ5の `red → green → refactor` がほぼ動かない理由:

- テストツリーの記述がすでに解を形成しており、それを直接記述すれば適切なコードになる
- コード管理方針が記述されていないため、書き直しになることがほぼない

→ tdd-run の micro-refactor（ノード1個の緑直後）は維持しつつ、
  セッション全体を通したマクロリファクタを別スキルとして分離することになった。

---

## 設計の核心

### 別セッション原則

tdd-feedback と同じ論理。実装者の文脈に汚染されると見えないものがある。
tdd-run では一つのコンテキストの内側しか見えていない。
tdd-refactor では複数コンテキストのツリーを同時に観察できる——この同時観察が重要。

### AI にとって「理解が深まる」とはどういうことか（開かれた問い）

人間の「理解が深まる」は累積的（過去の経験が現在の知覚を変える）。
AI はセッションをまたいで記憶を持たないため、同じ意味では深まらない。

ただし、tdd-refactor の文脈では機能的に類似した状態を作れる:
複数の tdd-run が完了した後のコードベース全体を一度に観察すること。
個別セッションでは不可能だった構造観察が可能になる。

これが「理解が深まる」ことと機能的に対応するかは開いたまま置いておく。

### B の同一性

最も重要な洞察:

X = {A, B, C}, Y = {B, D} のとき、
X の tdd-run では B は X の内部要素として定義される。
Y の tdd-run では B は Y の内部要素として再定義（または再実装）される。
二つの B の同一性は、二つのツリーを同時に見て初めて観察できる。

個別の tdd-run では Y を作るときに X の B を流用しようというフローにはなりにくい。
なぜなら、流用するためには B を独立したものとして扱う必要が生じるから。
B の同一性はシグナル——独立したサブドメインの可能性を示す。

シグナルの形:
- 異なるコンテキストのテストに同じ shape の `describe()` ノードが独立して存在する
- Y のコードが X の `src/` ディレクトリから import している
- 異なる `src/` に意味的に同一の関数が別々に書かれている

### crosscut の二分類

- **技術的な横断的関心事（インフラ）**: エラーログ・認証チェック——ドメインを持たない → 共通モジュール抽出
- **ドメインを示す crosscut（サブドメインの萌芽）**: フォーム受理機構・HTTP ハンドリング——
  それ自体が独立した問題領域を持つ可能性 → 辞書・ツリー・問題定義の再構成

### 「名前のドリフト」の実際の意味

当初、辞書の日本語名と英語の関数名の対応ズレを「名前のドリフト」と呼ぼうとしたが、
このシステムではそのような直接の翻訳関係は前提にしていない。

辞書と実装の橋は `@vocab` アノテーション。
実際にチェックできるのは:
- `@vocab` のリンク切れ（参照先エントリが消えた）
- 逆引きの欠落（stable エントリに対応する `@vocab` がない）

意味的な整合（辞書の定義と関数の実装が合っているか）はコードを読んで判断する——自動化できない。

---

## 観察の5視点

(a) 意味的な整合: 辞書定義と @vocab が指す実装の読み比べ（判断）
(b) 構造的な正確さ: check-vocab.js による機械的チェック
(c) 理解可能性: 責務の混在・describe() の形の確認
(d) 未使用コードの削除
(e) コンテキスト横断の構造パターン: 技術的 crosscut / ドメインを示す crosscut / B の同一性

  人間の「累積的な理解の深まり」と機能的に対応するかは開かれたまま
