# ロード可能性確認の追加（tdd-run 手順4）

**日付:** 2026-06-17

## 発端

`trigger.test.js`（hero-image-generator / layout-trigger）に、プロダクションコードを一切 import しないテストが生成されていた。`src/main.js` がブラウザ DOM に直接依存しているためロードできず、auto-decisions の判断として「純粋ロジックをテスト内で検証する」方針が採られた。その実態は、`main.js` のロジックをテストファイル内にコピーして検証するものであり、13件全て常に green になる偽陽性のテスト群だった。

## 診断

問題は3か所で形成・許容されていた。

**手順4（コンテキスト割り当て）で形成:** `main.js` に DOM 依存があるという事実は auto-decisions に記録されたが、スキルに「ロード不可のモジュールにノードを割り当てた場合の判断基準」がなかった。記録して先に進める構造になっていた。

**手順5（テスト記述）で偽陽性が書かれた:** ロードできないため、コピーでしのぐという判断がここで起きた。スキルの制約に「写経的に書かない」とあるが、コピー検証という逆の行為を明示的に禁じていなかった。

**手順7.5（ウォークスルー）で見落とされた:** 語彙・テスト・実装の名前対応を確認するが、「テストが実装をロードしているか」を確認しない。名前が揃っていれば通過できる。

## 概念的な理解

rationale §12「辞書・テスト・実装の指示連鎖」の観点で整理すると:

テストがモジュールをロードしていない場合、「テスト → 実装」の節が切断されている。
ただしこれは見かけ上は切断に見えない——describe() の名前は対応していて、テストは green になる。
これが「偽の連鎖」である。連鎖が切断されているのではなく、存在しないのに存在するように見える。

コピーによる「純粋ロジックのテスト」が最も悪い理由: ロジックのコピーが main.js の実際の実装と乖離しても、テストは気づかない。かつ「テストがある」という誤った安心感を生む。

## 修正

**tdd-run 手順4 の scaffolding可能性チェックに「ロード可能性確認」を追加した。**

コンテキスト割り当てを確定した直後に:
- 配置先モジュールがテスト実行環境でロードできるかを確認する
- ロード不可の場合は「ロジックを別モジュールに抽出する」か「そのノードの単体テストを書かない」かを手順4で決定する
- 「テストファイル内にロジックをコピーして検証する」は選択肢に含めない

**制約セクションに1行追加した:**

「テストはロジックの実装をロードして検証する——テストファイル内に定義したローカル関数を検証するテストは書かない」

## 残る問い

今回のケースでは「抽出」を選ばなかった（DOM 依存モジュールは構造上 main.js から切り離しにくいという判断）。
その判断が妥当だったかどうか——つまり本来は抽出すべき設計だったか——は tdd-refactor のフェーズで観察する対象になる。
ロード不可を「テスト不可明示」で処理したケースは、後続の feedback ループで実用上問題が出たときに再評価される。
