# UI語彙ドメインについての対話記録

**日付:** 2026-07-17

`docs/references/ui-interaction-description.md`（2026-07-07 作成の検討メモ）を具体化する
対話の中で固まった判断と、保留にしたアイデアの記録。

---

## 固まった判断

### ui は「ハンドル」と「器」の両方を同じドメインに入れる

きっかけは「domain: ui は部品(カード等)、domain: ui-pattern は一連の操作を支えるもの
(master-detail等)」という当初の切り分け案だった。ここに「UIは、solutionドメインの装置に
取り付けられたハンドルである」という見方が加わった——「ファイルリスト解決器」があり、
それをクリック・編集できるとき、その機構がUIである、という説明。

この見方を押し進めると、ヘッダーやサイドバーのような「他のUI部品をまとめる器」は
特定の概念へのハンドルになっていない（ページ全体の構造を支えているだけで、何か一つの
概念を操作するわけではない）ことに気づく。ハンドルの定義そのままでは器を含められない。

最終的には、器かハンドルかは `relations` の使い分け（`references` で概念を指すか
`contains` で子部品を持つか）で区別すれば十分で、ドメインを分ける理由がないと判断した。
概念とその表示、というMVC的な構造——「記事」と「記事UI」があり、「記事UI」を通じて
概念的対象である「記事」を操作している、UIコンポーネントは複数ありうる——という発想が
このハンドルの定義の出どころ。

### ui-pattern を pattern から分ける理由は「照合対象の違い」

domain を増やしすぎない、という基準（CLAUDE.md の「孤立した概念はアドホックな追加の兆候」
という原則）に照らすと、`ui-pattern` は
`pattern` と全く同じフィールド形（`heuristic` + `components`）を持つため、本当に
分ける必要があるのか、という問いが立った。

結論は「分ける」。理由は形ではなく、tdd-run のどのステップでどの入力と照合するかが違う
——`pattern` はテストツリーの形、`ui-pattern` は `ui` エントリー同士の関係の形。
`dict-search.js --filter domain=X` で照合対象を絞れることに実利があり、混在させると
コード構造のパターン照合とUI構造のパターン照合が互いを汚染する。

### domain: "layout" は作らない

「あるページはヘッダー・サイドバー・入力エリアからなり、操作部品はヘッダー右側にある」
という要求から、空間配置を表現する `domain: "layout"` を検討したが、却下した。

決め手になった問い直し：「マスターペインが左にあろうが上にあろうが master-detail は
成立する」。必要なのは配置ではなく役割の区別であり、それは `ui-pattern` の `components`
フィールドがすでに担える。ページは「`ui` エントリーの集合」として表現すれば足り、
「ヘッダー」のような空間的ニュアンスを持つ言葉自体は `ui` エントリーの名前として
使ってよいが、辞書が配置を規定しているわけではない、と整理した。

### states は概念エントリーに一元化する

「記事が非公開のときは『公開する』UIがあり、公開中のときは『非公開にする』UIがある」
という状態依存のUIをどう表現するかを検討した。UI側に `states` を持たせる案（手軽）と、
状態の本来の持ち主である概念に `states` を持たせ、UI側は参照するだけにする案
（表記ゆれを防げる）の二択で、後者を採用した。

## 保留にしたアイデア

### コンテキストをモデル単位（状態の全域）として再定義する

states を概念に一元化する議論の中で、「あるコンテキストとは、一つのモデルとそれが取りうる
状態の全域である」という、より大きな再定義のアイデアが出た（MVCのモデル単位でコンテキストを
切る、という発想）。これが成立すると、状態・UI・語彙境界（BC）が一つの構造に統合される
可能性がある。

現状のコンテキストの粒度（BC単位）を再定義する大きな変更になるため、今回は着手していない。
`docs/references/ui-interaction-description.md` の開いた論点にも記録済み。次に辞書の
コンテキスト設計を見直す機会があれば、ここから始めるとよい。

### フォーカス・時間の記述

`docs/references/ui-interaction-description.md`（2026-07-07 初版）がすでに指摘していた
「4語彙（状態・できる・遷移・効果）に収まらない可能性が高い」という論点のうち、フォーカスと
時間（アニメーション、タイムアウト）は今回のUI語彙ドメイン化でも扱わなかった。引き続き
対象外。

### 検証可能性

UIに面するツリーノードは既存の「できる」のツリーの中でテストされる想定で、UI語彙
（`ui`/`ui-pattern`）から新しいテスト導出の仕組みを作る必要はないと考えているが、
実際に運用してみないと確かめようがない。
