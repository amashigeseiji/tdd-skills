## 「できる」のツリー

web 版 Claude との事前対話（`tdd-run-draft.md`）で提案されていた枠組み。
本セッションで採用・洗練された。

**ツリーはテストファイルそのもの。**

```
X は xxx ができる   ← ルートノード（問題解決の仮説）
├── A は aaa ができる
│   ├── D は ddd ができる
│   └── E は eee ができる
└── B は bbb ができる
    └── C は ccc ができる
```

各ノードの表現は「X ができる」ではなく「**X は aaa ができる**」。
「X ができる」は能力の主張で X が何かを問わない。
「X は aaa ができる」は X を定義する行為——名前が先にあり、定義がそれに続く。
「X とは aaa ができるような何かだ」という問いに答えることがノードを置く行為。

**ポランニーの近位項・遠位項との対応:**
E のテストに対応するコードを書くとき、ツリーの中での位置——E が A を支え A が X を支える——
が常に文脈として存在する。E（近位項）を操作しながら X（遠位項）を了解している状態。
これが包括的理解（comprehend）であり、近視眼的な実装を防ぐ仕掛け。

**セミラティス:**
ツリーはセミラティス状になってよい（ノードが複数の親を持つ）。
共有ノードは命名の契機——その共有に名前が与えられることで、合成の意味が明示される。

## 受け入れテスト＝ルートノードのテスト

「外側ループ（受け入れテスト）」と「内側ループ（振る舞い宣言テスト）」という二層は、
ツリーの同じ構造の「位置の違い」にすぎない。

- ルートノードのテスト = 受け入れテスト（問題解決の仮説）
- 中間・末端ノードのテスト = 振る舞い宣言テスト

どのノードも「○○は△△ができる」という同じ形をしている。
最初にルートのテストを書いて red のまま置くのは「ツリーの頂点から始める」の帰結。
GOOS の二重ループは、このツリーで統一される。

## 問題は「できない」状態

問題＝「できない」状態、実装＝「できない」を「できる」に変換すること、という再フレーミング。

ツリーの不変量はルートX ではなく **problem.md にある「できない」の記述**。
X は問題を解くための仮説であり、実装中に更新されることがある。

**上位ノードの変質を検知したとき:**
1. problem.md と照合する——新しい合成は「できない」を解消するか？
2. 解消するなら新しいXに名前を与え、ルートのテストを更新し tdd-vocab に登録
3. 解消しないなら何が足りないかを特定してツリーを再構築

下位ノードの実装から上位ノードへの逆行（E + C → Y という新しい合成の発見）を認めることは、
「テストを書き換えてよい」ではなく「problem.md を照合した上でルートを更新する」こと。

## tdd-vocab の設計

### 四つの呼ばれ方

| 呼び出し | タイミング | 目的 |
|---------|-----------|------|
| `init` | プロジェクト開始時（tdd-run から） | 問題領域の語彙ネットワークを一式作る |
| `add` | tdd-run・tdd-feedback から随時 | 新しい概念を wip に登録する |
| `promote` | ユーザーの任意タイミング | wip → stable に昇格する |
| `check` | 整合性が気になるとき | 孤立概念・リンク切れ・矛盾を確認する |

### 語彙セットは行動可能空間のガードレール

語彙セットにある概念がツリーのノードの候補になる。
語彙にない概念でノードを埋めることを抑制し、AI がこのプロジェクト固有の言葉で考えるよう導く。
これは禁止ではなく方向づけ——語彙にない概念が必要なら wip に登録して使う。

### 語彙は互いにリンクしてネットワークをなす

各エントリの定義は他の概念を参照する（自然言語のリンク）。
孤立した概念（どこからも参照されず、何も参照しない）はアドホックな追加の兆候。
体系性はこのリンク構造が担保する。

**キャンバスの例:**
```
キャンバス: 画像を並べる支持体。グリッドを持てる。
グリッド: キャンバスを分割して得られるセルの集まり。行と列から成る。
セル: 行と列の交点にある単位。画像を配置できる。
行: グリッドを横に走る単位。複数のセルを含む。
列: グリッドを縦に走る単位。複数のセルを含む。
```

概念が孤立せず互いを参照している——これが体系性。

### wip は作業仮説（provisional guardrail）

tdd-run が tdd-vocab init を呼ぶとき、アウトプットは `docs/dictionary.md`（stable）ではなく
`docs/wip/dictionary.md` に置く。この時点での語彙定義はまだ作業仮説だから。

wip の語彙が仮のガードレールとして機能し、実装・受け入れを経た後に promote で stable に昇格する。

## Coplien のドメイン区別

Jim Coplien（*Multi-Paradigm Design*, 1998）の枠組み：

- **アプリケーションドメイン**: ユーザーの関心が寄せられている知識の体系。
  過去のドメイン分析技法では problem domain と呼ばれていたもの。
- **ソリューションドメイン**: 実装者の関心。アプリケーションドメインの概念を表現するための素
  （OOPのClass・RDB・関数型の関数など）。複数のソリューションドメインに同時に関わることがある。

重要な洞察：アプリケーションドメインとソリューションドメインを同時並行で調査し、
ソリューションドメインの語彙でアプリケーションドメインが**無理なく**表現できるようにすることが
マルチパラダイムデザインの目標のひとつ。

**OrderStore はアプリケーションドメインの概念。**
注文を記録する行為は現実世界に存在する（ノートへの記帳でも成立する）。
それを PostgreSQL で実現するか MySQL で実現するかがソリューションドメインの関心。

tdd-vocab が管理するのは主にアプリケーションドメインの語彙。
ソリューションドメインの語彙（React・PostgreSQL・OOPパターン）は原則エントリ不要。

## ワークフローの確定

```
tdd-problem → tdd-run（内部で tdd-vocab init を呼ぶ）→ tdd-feedback
```

tdd-vocab init を tdd-run の内部で呼ぶ理由:
- problem.md を読んだ直後に語彙を定義するのが情報の流れとして自然
- ガードレールのない状態でツリーを構成し始めるとアドホックな概念でノードを埋めてしまう
- 辞書とツリーを同時に考えられる
