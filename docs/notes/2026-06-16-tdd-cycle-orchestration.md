# 2026-06-16 tdd-cycle の設計（オーケストレーション）

## 発端

tdd-run / tdd-vocab / tdd-refactor がある程度揃ってきたところで、
「これらをオーケストレーションできないか」という問いが出た。

- tdd-problem: ユーザーとの対話が前提 → 自動化対象外
- tdd-feedback: ユーザーの経験を入力する → 自動化対象外
- tdd-run, tdd-vocab, tdd-refactor: 自動化の余地がある

---

## 設計の核心

### 承認ポイントの分類

tdd-run にある承認ポイントを「なぜ存在するか」で分類した:

| ポイント | 目的 | auto モードでの扱い |
|---------|------|-------------------|
| ツリー分解の確認 | 基準チェック + ドメイン知識注入 | 自律判断（基準を自己適用） |
| 利用仮説の確認 | problem.md との照合 | problem.md から自律作成 |
| モジュール境界の確認 | 基準チェック | 自律判断（基準を自己適用） |
| ルート green の受け入れ | 完了の確認 | 自動的に次フェーズへ |
| tdd-vocab promote | 語彙ネットワーク全体への影響判断 | ユーザー承認を維持 |
| refactor 実施 | 優先度・リスク判断 | ユーザー承認を維持 |

ツリーとモジュールの承認が自律化できる理由: スキル内に明示された基準（述語に技術名を入れない等）
が自己チェック可能な形で書かれているため。基準が明示的であれば、AI は自分の出力を照合できる。

### オーケストレーターが別エージェントとして「レビュー」することの価値

同じ情報源を持つ別エージェントのレビューに意味があるか、という問いに対して:

- 意味はある（生成文脈の汚染を避けられる、フレーミング効果）
- ただし限定的（ドメインの妥当性については同じ problem.md を読んでいる）
- 現時点ではコスト対効果が不明なので実装せず、実績を見てから判断

→ auto モードで進め、後からレビュー機能を追加する判断をした

### アーキテクチャ

オーケストレーター（tdd-cycle）が Plan フェーズを別途持つ必要があるか、という問いに対して:

- tdd-run が自分でツリー生成から実装まで担える（auto モード）
- オーケストレーターの役割は「何を・いつ・どの順で動かすか」の制御であり、内容のレビューではない
- Plan を分離する必要はない

最終的な構造:
```
tdd-cycle
├── Phase 1: 起動確認（problem.md + 権限の事前確認）
├── Phase 2: tdd-run サブエージェント（auto mode）
├── Phase 3: tdd-refactor サブエージェント（観察+計画まで）
├── Phase 4: 一括報告
└── Phase 5: promote と refactor の実施（ユーザー承認後）
```

### ツールの事前承認と実行中断の問題

auto モードで問題になるのは、Bash コマンドの権限プロンプトによる実行中断。

対策:
1. **Phase 1 で settings.json を読んで不足する allowlist を事前チェックする**
   → 不足があれば実行前に止まってユーザーに伝える
2. **実行中に権限プロンプトが出た場合は即座に停止してブロックされたコマンドを報告する**

mid-run で止まるよりも、起動前に止まる方がユーザーにとって扱いやすい。

## auto-decisions.md の役割

対話なき実行でも決定が消えない形を作るためのファイル。

tdd-run の auto モードで自律的に決定したこと（ツリー構造・モジュール境界・利用仮説）を
記録しておき、Phase 4 の一括報告でユーザーが確認できるようにする。
これは「沈殿」（rationale section 5）のメカニズムであり、
「対話なき実行」を後から inspect するための素材になる。
