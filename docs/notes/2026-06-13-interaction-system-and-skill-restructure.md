## 発端

`ai_coding_design_discussion.md`（Claude web版との対話記録）を起点に、
tdd-\* スキルの設計について議論した。

---

## ポランニーの近位項・遠位項とAIコーディング

ポランニーの暗黙知の枠組み：
- **近位項**（proximal）：「から知る」もの。意識の焦点から外れて身体に吸収される
- **遠位項**（distal）：「として知る」もの。近位項を通じて了解される対象
- **包括的理解（comprehend）**：近位項を通じて遠位項を了解すること

AIコーディングでの問題：コードを書くとき、個々の関数（近位項）しか見えておらず、
それを通じて了解されるべき「何か」（遠位項）が見えていない。

**合成レイヤーへの命名が鍵：**
関数群の合成が「何をするか」に名前がない状態では、合成は前景から消えず、
コードは「関数の平べったいリスト」として認識されつづける。
名前を与えることで近位項になれる——それを通じて遠位項（インタラクション全体の意味）を了解できる。

## インタラクションの系

**定義：** 複数の要素が協調してひとつのことを成し遂げるとき、その協調全体に与えられる名前。
「グリッド要素の相対配置の決定」「入力の受付から保存までの流れ」のように、
要素の羅列ではなく、要素が参加するインタラクション全体をひとつとして捉えたもの。

**レイヤー構造：** 全体と部分の関係はレイヤーになっている。
ある階層の系（全体）が、上の階層では要素として機能する。
設計とはこのレイヤー構造を意識的に構築すること。

**仮設計として：** 実装前の最善の想定として名前を置く。
実装中に合成の実際の形が見えてきたとき更新される。

## TDDにおける分解と合成

**重要な気づき：** TDDで与えられるのはX（ユーザーストーリー・受け入れ条件）だけで、
「X = {A, B, C}」という分解をするのはTDD実践者自身の仕事。
分解は設計フェーズが事前に行うものではなく、TDDを回す中で出てくるもの。

**実装の責務は分解と合成：**
- 設計の責務：全体に名前を与えること（インタラクションの系の命名）
- 実装の責務：全体を分解し、合成することで実現すること

```
設計：X = {A, B, C}（全体と要素の宣言）
実装：「Aはxxxができる」→ A = {D, E}（さらなる分解）→ D, E を実装 → A を合成 → X を合成
```

この分解と合成の過程でこそ「逆流」（設計への発見の還元）が起きる。

## TDDにおけるテストの二種類

**振る舞いの宣言テスト（TDDの本体）：**
「Aはxxxができる」という形で要素のインターフェースを実装より先に定める。
効果はインターフェースと実装の分離。
内側ループ（GOOSのunit test loop）に対応。

**性質の検証テスト（受け入れ条件）：**
設計が「系が実現されたとき P-01 が成り立つ」と述べたことを、合成されたシステム全体に対して確かめる。
外側ループ（GOOSのacceptance test loop）に対応。最初から書いておいてredのまま置く。

現在のtdd-runの記述は後者（性質の検証）だけを指しており、TDDの中心である前者（振る舞いの宣言）が
抜けている。

## 設計文書の役割の再定義

**従来の理解：** コード構造の事前仕様、tdd-runが実装のために読む仕様書

**新しい理解：** 機械と現実をつなぐグルー。語彙と文脈の定義。

- **ユビキタス言語の辞書：** Xとは何か、Aとは何か、をドメインの言葉で定義
- **コンテキスト境界：** この語彙が適用される範囲の明示
- **インタラクションの系：** そのコンテキスト内で何が起きるか

設計文書がない、あるいは「コード変換指示」になっていると、
近位項（コード）から遠位項（ドメインの現実）への経路が断たれる。

## 文書の二層構造

**揮発層（`plans/`）：** フィーチャー固有、サイクル内で消費される
- `problem.md`：このフィーチャーの問題定義
- `findings.md`：このサイクルのフィードバック

**永続層（`docs/`）：** リポジトリ全体の設計知識として蓄積される
- `docs/dictionary.md`：安定した語彙（受け入れ済みの概念）
- `docs/wip/dictionary.md`：コミット前の概念（実装中に発見された、未受け入れ）

**wip のライフサイクル：**
1. tdd-run が概念を発見 → `docs/wip/dictionary.md` に登録
2. tdd-feedback で利用受け入れ確認
3. ユーザーの任意タイミングで `docs/dictionary.md` に昇格
4. プランアーカイブ時に wip があれば通知

**優先順位：** tdd-run は両方を読む。同一概念があれば wip が優先（新解釈が旧解釈をオーバーライド）。

**再定義の場合は追加と区別：** 既存概念の書き換えは影響範囲の明記が必要。

## スキル体系の再構成案

### 廃止
- **tdd-design**：独立フェーズとして不要。内容は tdd-run に吸収。
  「分解は実装者の仕事」「語彙定義は tdd-vocab の仕事」であれば存在理由がなくなる。

### 残る・変わる
- **tdd-problem**：変わらず。問題定義を `plans/` に置く
- **tdd-run**（大きく変わる）：
  - 設計と実装の両方の実践者
  - problem.md と `docs/dictionary.md` を読んでXを定義
  - 分解・振る舞い宣言・実装・合成を行う
  - 概念発見時は tdd-vocab を呼んで登録を依頼
  - プランアーカイブ時は wip があれば通知
- **tdd-feedback**（概ね変わらない）：
  - 利用フィードバック収集
  - 概念の再解釈が生まれたら tdd-vocab を呼ぶ
  - アーカイブ時に wip 通知

### 新規
- **tdd-vocab**：
  - `docs/dictionary.md` と `docs/wip/` の唯一の管理者
  - tdd-run / tdd-feedback から「概念を登録して」と呼ばれる
  - 整合性チェック（重複・矛盾・包含関係）
  - 再定義時の影響範囲分析
  - wip → stable への昇格
  - ユーザーの明示的な呼び出し（`/tdd-vocab`）で動く
