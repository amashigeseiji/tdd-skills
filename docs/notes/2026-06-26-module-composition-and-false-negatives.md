# モジュール合成・偽陰性・エントリーポイント宣言（tdd-run 追記）

**日付:** 2026-06-26

## 発端

`editor-realtime-preview` の実装で2種類の問題が発生した（`plans/editor-realtime-preview/tdd-skills-feedback.md` 参照）。

1. `debouncer.js` と `autoPreviewInitializer.js` を ESM モジュールとして作成したが、`editor.js` が非モジュールスクリプトだったため import できず、同等の関数をインラインで実装した。モジュールのテストは通っていたが、本番コードはそのモジュールを使っていなかった。
2. `editor.js` の `type="module"` 化は技術的に可能だったが、AIは制約を「所与の前提」として扱い、迂回策を選んだ。

## 分析

### テスト対象と本番コードの分離

人間にとっては「テストするモジュール = 本番で動くコード」は自明だが、AIはこの同一性を自動的に担保しない。前回（2026-06-17）のロード可能性確認追加は「テスト → モジュール」の接続を担保した。今回の問題は「モジュール → 本番コード（エントリーポイント）」の接続が欠けていた。

### 偽陰性

Khorikov「単体テストの考え方/使い方」の枠組みでは、今回の問題は**偽陰性**に分類される——実装が間違っているのにテストが合格し続ける状態。モジュールが依存グラフから孤立している場合、そのモジュールを使わない別の実装に変わってもテストは通り続ける。

### フレーム問題

AIが「自分が作ったモジュールが後の行為に何を含意するか」を追跡できないことは、古典的なフレーム問題として整理できる。完全な解決はないが、明示的なチェックポイントによって緩和できる。

### 制約を「所与」として扱う傾向

AIは既存の構造を変えずにタスクを局所的に解こうとする。`editor.js` が非モジュールであることを固定された制約として扱い、`type="module"` 化という解消策を考えなかった。「制約を迂回すること」が複数の評価基準（テスト容易性・依存追跡可能性・偽陰性リスク）でスコアを下げるという評価軸がなかった。

## tdd-run への追記内容

### 新規セクション「テストとモジュール設計」

- ソフトウェアはモジュールの合成体であり、依存グラフとして形成される
- 孤立ノードは合成に参加していない——そのテストが通っても動作を保証しない
- テストの4象限（Khorikov）: 偽陰性（実装が誤っているのにテストが合格）を特に注意する
- 実装選択肢の評価基準: テスト容易性・依存追跡可能性・偽陰性リスク・変更コスト

### 手順4 コンテキスト割り当て

コンテキスト表の後に、ルートノード X のエントリーポイント宣言を必須化した。

```
エントリーポイント: app.js → X
```

ツリー内部の依存関係はツリー自体が表現しているため、必要な宣言はルートとエントリーポイントの接続のみ。これにより、ツリーごと孤立しているケースも検出できる。

### 手順8 統合確認

宣言したエントリーポイントからルートモジュールが実際に参照されていることを確認する手順を追加した。「未定」が残っている場合は統合先を決定してから次に進む。

## 残る問い

- エントリーポイント宣言が「未定」のまま手順8を迎えた場合、AIはどう判断するか（統合か削除か）の基準がまだ明示されていない
- 言語によっては依存関係の静的解析が困難なケース（`<script src>` 等）があり、そうした構造自体を評価基準でマイナスとして扱うことで対処する設計になっている
