# 言語依存の扱いと、中継引数から情報隠蔽へ

**日付:** 2026-07-07

## 発端

「システム全体の関数呼び出しをグラフ化できるか」という技術的な質問から始まった。
モジュール依存（import/export）は dependency-cruiser 等で機械的に取れるが、
関数レベルの呼び出し関係は静的解析では原理的に近似になる（コールバック・動的ディスパッチ・高階関数）。
TypeScript Compiler API や LSP の Call Hierarchy を使えば実用上十分な近似は作れる。

ここから二つの問いが派生した。

1. こういう「言語次第」の知識をこのリポジトリでどう扱うか。
2. 関数レベルの事実が要る具体的なリファクタリング——中継引数の隠蔽——で、AIに何を考えさせるか。

---

## 観察1：「言語次第」問題はアダプタパターンで扱える

言語依存がワークフローのどこに現れるかを調べると、局所的だった。

- problem / feedback / vocab — 言語フリー。問題の言葉・使用経験・概念のネットワークに言語は入らない。
- scaffold / run / refactor — ここに集中する。テストランナーの書き方、モジュール＝ファイルという前提、依存グラフの取り方。

そして tdd-scaffold の depgraph アダプタがすでに解法の原型を持っている。
**スキル側は「調査の手順」と「呼び出し規約＋正規化 JSON の形」だけを固定し、
言語ごとの実現はプロジェクト側に生成するアダプタスクリプトに委ねる。**

rationale の語彙で言い直すと：「テストツリー」「依存グラフ」「利用可能な状態」といった
概念は遠位項であり、スキルが固定すべきもの。言語固有の実現（Jest の書き方、pytest の流儀）は
近位項であり、AI がすでに棲みついていて透明に操作できる領域。モデルは各言語のツールを
知っているから、スキルが近位項の知識を書き並べるのは冗長で、陳腐化の負債になる。
どの言語・どのツールかはスキルの知識からではなく、プロジェクトを調査して発見し、
アダプタを実行して検証する——接地は調査と実行から取る。

原則として一行にすると：
**スキルは概念と契約を固定し、実現はプロジェクト側アダプタに委ね、接地は調査と実行で取る。**

### 付随する観察：JS 前提の漏れ

tdd-run に describe/it が18箇所、tdd-refactor に `grep "describe("`・`.js` パス・`npm test` の
直書きがある。これが「JS を参照実現として例示している」のか「JS 前提が無自覚に漏れている」のかは
区別しておく必要がある。前者なら「以下は JS/Jest での実現例」と一行宣言すれば済む。
後者なら、describe のネストがない言語（Go のサブテスト等）でツリーの表現自体が変わるため、
契約（何がツリーの本体か）を言語非依存に言い直す必要がある。

また rationale §5 の「ファイル＝モジュール、ファイルパス＝住所」は辞書→モジュール境界の対応を
支える前提だが、Java/C#/PHP の namespace 圏では弱まる。言語依存として一番深いのはここかもしれない。

## 観察2：中継引数の隠蔽に必要なのは呼び出しグラフより細かい事実

具体例：モジュールAの関数 a, b, c がどれもモジュールBの関数 d を引数に取る。
実体は a が b を呼び、b が c を呼び、d を*使う*のは c だけ。
このとき c を独立モジュール C に抽出すれば、A のインターフェースから d が消える。

必要な事実は呼び出しグラフ（a→b→c）ではなく **d という値の流れ**——
「d は a→b→c と運ばれ、適用されるのは c の中だけ」。
静的解析なら use-def 連鎖、リファクタリングの言葉なら中継引数（pass-through parameter）。

AIに考えさせることは事実の層と解釈の層に分かれる。

**事実の層（機械的に確定できる・ツール化すべき）**

1. 各引数を「使っている」か「運んでいるだけ」かに分類する。
2. 使用点の局在を確認する（d の使用が c に局在しているか）。
3. 呼び出し側が d に束縛している実引数を列挙する。常に一種類なら配管であり隠蔽できる。
   複数（テストのモック等）なら本物の可変点であり、隠蔽は設計変更になる。
4. 波及範囲を出す（a・b のシグネチャ変更が呼び出し側とテストに波及する）。

この層は depgraph アダプタの延長（引数の使用局在を足した正規化 JSON）としてツール化できる。

**解釈の層（語彙と接地が要る）**

5. その局在に名前があるか。C という概念が辞書の語彙として立つか。
   「共有ノードは命名の契機」と同型の兆候。名前が出ないなら構造いじりでありアドホック。
6. 注入は意図か偶然か。コードから確定できないため人間への確認事項として出す。
7. 依存は「解消」ではなく「局所化」。A→B が A→C→B に付け替わり、B の知識が C に閉じる。
8. 変換の選択肢は複数ある。(a) モジュール C 抽出、(b) c が B を直接 import、
   (c) 合成ルートで部分適用 `c = makeC(d)`。選択は 6 の答えに依存する。

## 観察3：Parnas の情報隠蔽が問いを一本化する

観察2の 5〜7 は、Parnas (1972) の情報隠蔽——モジュール分解の基準は処理の工程ではなく
隠すべき設計決定（secret）である——で束ねられる。

- a→b→c の d 中継は**工程で切った分解の症状**。処理の流れに沿って切ると、
  下流だけが必要とする知識を上流が運搬させられる。中継引数はその運搬の痕跡。
- 事実の層（1〜3）は「使いもしない知識を持っている箇所」の機械的検出だった。
- 判断は「**d の選択は誰の秘密か**」に集約される。c の秘密なら C に隠す。
  呼び出し側の秘密（差し替えの可変点）なら、インターフェース上の d は漏洩ではなく
  公開すべき決定であり、消してはいけない。

チェックリストの代わりに、生成的な問いを3段で置ける：

> **このモジュールは何を知っているか。そのうち、知らなくてよいものはどれか。それは誰の秘密か。**

中継引数の検出も、抽出の是非も、変換の選択も、この問いの下流に落ちる。

辞書との接続：辞書エントリがドメイン境界＝ファイルパスを宣言するとき、そのモジュールが
*何を隠すか*も同時に宣言されていると読める。現状の辞書は「概念が何であるか（definition）」を
持つが、モジュール境界と対応する概念の「何を隠すか（secret）」は暗黙のまま。

留保：Parnas の基準は「変わりやすい決定を隠せ」であり、変更の見込みという時間軸が入る。
これはコードから読めず、接地（人間の見立て、feedback での実使用）から取るしかない。
ワークフローの役割分担とちょうど噛み合う。

## 未解決の問い

1. 「スキルは概念と契約を固定し、実現はアダプタに委ねる」を rationale か CLAUDE.md の
   判断軸に昇格させるか。昇格させるなら、run/refactor の JS 直書きを例示宣言に整理する作業が伴う。
2. tdd-refactor に「中継引数の局在は境界の兆候」という検出項目を足すか。
   足す場合、事実の層をアダプタ（引数フロー解析）としてツール化するか、当面は AI の読解に任せるか。
3. 辞書エントリに secret（そのモジュールが隠す決定）を持たせるか。
   definition との重複や記述コストと、構造的決定の安定化（pattern 層と同じ役割）との釣り合い。
