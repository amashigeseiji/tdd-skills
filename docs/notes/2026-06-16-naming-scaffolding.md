# 命名・Scaffolding・抽象レベル（2026-06-16）

このノートは `tdd-run` スキルへの3つの変更を通じて明らかになった考え方の記録。
出発点は `editor-frontmatter-template` と `editor-sidebar` の findings.md に繰り返し現れたパターン。

---

## 繰り返したパターン：ツリーの名前が実装に現れない

`editor-sidebar`（2026-06-14）: 「展開状態マネージャー」がツリーに宣言されたが、実装では
`initSidebarTree()` の内部に混在し、その名前を持つ関数が存在しなかった。

`editor-frontmatter-template`（2026-06-16）: 「フロントマターテンプレートローダー」がルートノードに
宣言されたが、実装では `fetchData()` の内部処理として埋め込まれた。ルートのテストは green になったので
問題として認識されなかった。

共通構造：ツリーに名前がある → テストが green になる → 名前を持つ実装が存在しない。

## なぜ起きるか

スキルは「振る舞い」を振り付けるが、名前を固定しない。
ツリーで宣言された名前をAIが役割の記述として読み、役割を果たす何かを実装フェーズで新たに命名する。
「ローダーという役割を果たすなら何でもよい」という読みが、名前の消失を許す。

さらに、動詞への変換が起きる。`frontmatterTemplateLoader`（名詞、entity）が
`loadFrontmatterTemplate`（動詞、action）になる。これは同じ振る舞いに見えるが概念の変質であり、
ツリーが「Xという何かが存在する」と宣言したことと、実装が「loadという操作がある」と言うことは違う。

## 命名は宣言である

名前を付けることはその対象を存在させる宣言に等しい——これは対話の中で繰り返し現れてきた前提。
ツリーのノードを書くことは「X とは xxx ができるような何かである」という宣言であり、
その名前は実装に引き継がれる約束でなければならない。

この連続性を担保するために、ノード名を英語識別子で定義することにした。日本語のカタカナ名は
実装の識別子として直接使えないため、「名前を変換せずに引き継ぐ」という制約を実現できない。
英語識別子で書けば変換不要になり、`describe()` の主語と実装の関数・クラス名が一致する。

`frontmatterTemplateLoader` をカタカナにすれば「フロントマターテンプレートローダー」であり、
それを `loadFrontmatterTemplate` にする積極的な理由はない——単に動詞先行の命名慣習に
引っ張られただけだった。

## Scaffolding が構造的に強制する

宣言を実装に引き継ぐことを「確認ステップ」に頼ると、見落とされる（実証済み）。
ツリー確定と同時にノード名をスタブとして実装ファイルに書く（scaffolding）ことで、
実装フェーズに入った時点で名前は既に存在する。「どの関数に書くか」を決める余地がなくなる。

スタブにはこの時点で `@vocab` と `@test` アノテーションも書く。
語彙エントリとテストファイルパスはツリー確定時に確定しているので、後で書く必要がない。

## 「テストが書けない」は抽象レベルのシグナル

`editor-frontmatter-template` の F-02：describe に `/get_frontmatter_templates エンドポイントから
JSON 設定を取得できる` という記述が現れ、「ブラウザの fetch に依存するためテストできない」という
判断につながった。実際にテストすべき振る舞いは `getTemplates(config)` という純粋関数として
容易にテストできるものだった。

「テストが書けない」と感じるのは、テストを構想する時点で具体的な実装手段（fetch、localStorage等）が
念頭にあるからであり、それは述語が既に手段に引っ張られているサインである。
テストを省略するのではなく describe を問い直す——「この振る舞いを、実装手段を決めずに表現するとどうなるか」。

この問い直しをツリー確定時に構造的に促す仕掛けとして2つのチェックを追加した:

1. **技術語スキャン**: 全ノードの述語を走査し、URL・ライブラリ名・ストレージ種別等が含まれていれば書き直す
2. **Scaffolding可能性チェック**: 全ノードの主語が実装の識別子として成立するかを確認する。
   成立しないノード（URLパス等）はツリーに含めない

2 は「URLパス `/get_frontmatter_templates`」のような主語を機械的に却下できる。
1 では捕まえられない（述語に技術語がないが主語が識別子でない）ケースを補完する。

両チェックとも完全ではない（`localStorage は ...` のように主語が技術名でもスタブは形式的に生成できる）が、
「テストが書けない」という判断が実装フェーズで起きる前に、その原因の多くを構造的に排除できる。
