# Scaffolding・指示関係・パラダイム依存（2026-06-17）

`tdd-run` の scaffolding 設計をめぐる対話から浮上した概念的整理の記録。

---

## 出発点：「S は verb できる」と関数命名

テストを「S は verb できる」で書く方式について、関数コードでの困難が論点になった。

OOP では S がクラスに対応するため `Cart#calculateTotal()` と自然に変換される。
関数では S がクラスを持たず、`calculateTotal(cart)` になる——このとき verb が関数名になる。
S（Cart）は引数の型として現れ、テストツリー上の概念の起点としての役割を失いやすい。

「テストが空欄のまま実装だけ行われる」問題との接続：S を命名の起点にしようとして行き詰まると、
テストの記述力が落ち、空欄が生まれる。verb を関数名にすることでこの詰まりが解消できる。

S がクラスに対応しやすいのはパラダイムの相性ではなく、クラスが「主体」という概念を持つからであり、
自然言語の S（主語）が元々持つ主体性と一致するからだ。
関数に主体という概念がないことは、命名規則の問題ではなくパラダイムの問題。

## 定義域としての S

`calculateTotal(Cart)` の Cart は、数学的な意味での定義域（domain）でもある。
DDD の「ドメイン辞書」に出てくる概念は「問題領域で定義されているもの」の集合——
両方の意味での domain が重なる、という接続が浮上した。

ただし、この設計は数学的操作よりも自然言語との近さを優先している。
problem.md という自然言語入力が基礎にある以上、プログラミングパラダイムの中に
自然言語的特性を持ち込む方が、現在の枠組みとして扱いやすい。

## 自然言語のアンカーと意味・シンボルの関係

「辞書 | テスト | コード」の関係を「意味 | シンボル」として配置している——
シンボルを通じて意味を志向する、という設計の目的がある。

- 辞書 = 意味の層（現実との指示関係を持つ）
- テスト = 辞書を指し示すシンボル
- 実装 = テストを指し示すシンボル

この階層で「辞書 → テスト」の節を S が担い、「テスト → 実装」の節を scaffolding が担う。

## 指示の連鎖と scaffolding の役割

テストが空欄になると連鎖の中間が抜け、実装が辞書・現実と切り離される。
これが「AI がコードの意味を把握できない」状態の構造的な説明になる。

scaffold.sh による自動スタブ生成は「テスト → 実装」の節を構造的に保守する。
S は `describe()` ブロックに現れ続けることで、「辞書 → テスト」の節を保守する。

変換規則（S → ファイル名、verb → 関数名）が明示されていなかったことが、
連鎖を壊す原因の一つだった——AI が自由に解釈できる余地があった。

## 設計決定

### すべてのノードを独立ファイルに

従来、一部のノードを親ファイルに「閉じ込める」ルールがあった。
セミラティス（共有ノード）を考えると、独立している方が後から共有が起きても何も変えなくていい。
ファイルが増えすぎる懸念は、コンテキストの粒度を狭くすることで対処する。

### S → ファイル名、verb → 関数名

変換規則をスクリプトに書くことで、AI の解釈の余地がなくなる。
同じ S を持つ複数ノードがある場合（同一主語・複数 verb）、スクリプトは既存ファイルに関数を追記する。

### tdd-scaffold を独立スキルに

scaffold.sh の生成（初期化）は tdd-run の毎回の実行に必要ではない。
skill.md に埋め込むとトークン消費が増える。
tdd-scaffold として独立させ、プロジェクト初期化時に一度呼ぶ構成にした。

tdd-run には `.claude/tdd/scaffold.sh` がなければ `/tdd-scaffold` を呼ぶ、という1行だけが残る。

### src/tests ルートをプロジェクトから検出

scaffold.sh に `src/` `tests/` とハードコードするのではなく、
tdd-scaffold がプロジェクト構造を調査して実際のパスをスクリプトに焼き込む。
Rails なら `rails generate` を活用した wrapper を生成するなど、パラダイムに応じた実装を持つ。
