# 問題と解決の共進化——2026-06-11 の対話から

## 発端

tdd-problem の「力A・力B」という問いかけが戸惑いを生む、という指摘。
ユーザーの代替提案：問題とは「やりたいのにできない状態」であり、
障害を取り除くのが解決だ、という整理。

→ 同意。「力」は Alexander の forces 由来だが、問題を分析した後の語彙であり、
ヒアリング時に押しつけると枠の先渡しになる。問題は「やりたいこと（目標）＋
それを阻む障害・制約」という方向で問う方が自然。

## ジャクソンとアジャイルの対立構造

Michael Jackson のアプローチ（Problem Frames / World and the Machine）：
- 問題領域（世界）と解決領域（機械）を厳密に分離する
- 要件は世界の記述、仕様は機械のインターフェース記述
- これは「tame problems」に対して有効なアプローチ

アジャイル・DDD のアプローチ：
- 投機的な設計行為（解を先に動かす）によって問題の見え方が変わる
- 問題を正確に定義してから解くのではなく、解を試みながら問題を理解していく
- これはデザイン思考的なアプローチであり、wicked problems に対応している

Coplien の指摘（Multi-Paradigm Design）：設計パラダイムの選択が問題の中に
何が「見える」かを変える。OO で設計しようとすると問題の中にオブジェクトらしき
構造が見えてくる。それが問題の本質的構造なのか、設計パラダイムの投影なのかは
事前に判断できない。

## ソフトウェア外のデザイン理論との接続

### Wicked Problems（Rittel & Webber, 1973）
- 計画・政策・デザインの問題は "wicked" — 解を試みることでしか理解できない
- これがフィードバックループの「必然性」を説明する
- ジャクソン的な「先に定義」が機能するのは tame な場合に限られる

### 共進化（Dorst & Cross, 2001）
- 設計者の実際の思考観察：問題空間と解空間は共進化する
- どちらが先ということがなく、両者が交互に更新されながら前進する
- Lawson の観察（建築家 vs 科学者）とも一致

### アブダクション（Dorst, Frame Innovation, 2015）
- 演繹「事実＋法則 → 結論」
- 帰納「事例 → 法則」
- 設計「望ましい結果＋作動原理 → 問題の再定義」← アブダクション
- 「この解が良い答えになるような問題とは何か」という方向の推論

## rationale.md および skill への含意

### 変えないこと
- problem → design → run → feedback のフェーズ分離は有効
- 上記の考えは既にループ構造に含まれている

### 明確にすること（rationale.md に追加）
- problem.md は「スナップショット（バージョン付きの仮説）」であって確定文書ではない
- ループは例外的修正ではなく予測される正常な軌道
- Jackson の world/machine が問題/解決分離の直接的根拠
- Wicked problems の概念がループ必然性の理論的根拠

### tdd-problem のトーンへの含意
- 「問題の真相を確定する」ではなく「現在の理解を最も正直に言語化する」
- 改版は失敗ではなく、理解が深まったことの証拠
- ヒアリング時は「力A・力B」ではなく「やりたいこと」「何が阻んでいるか」で問う
