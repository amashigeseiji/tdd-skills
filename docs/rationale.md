# なぜこの形か（設計の考え方）

**初版:** 2026-06-10  
**改訂:** 2026-06-19  

この文書はスキルを**再設計するときに読む**。「どう使うか」ではなく「なぜこの形なのか」を残す。結論だけでなく、その背後の思考枠組みを更新するための起点。

---

## 1. ワークフローの設計意図

5層構造（problem → [辞書 / テストツリー / 実装] → feedback）の設計意図は、**AI が意味空間の中で自律的に構成できる**ことにある。

内側の3層（辞書・テストツリー・実装）が AI の作業場で、辞書とテストツリーが「緩やかなガードレール」として機能する。「緩やかな」のは二つの意味がある:

- **実装の手段を縛らない** — 辞書とテストは「何がどう連携してできるか」を宣言するが、どう実装するかは AI が決める
- **AI が構成する** — 辞書とテストツリーは人間が一方的に設定するのではなく、AI が問題定義をもとに構成し、実装中に更新する

外側の2層（problem / feedback）は現実との接地点。problem は「何が適切か」を持ち込み、feedback は「使ったらどうだったか」を持ち込む。意味の正しさの判断は人間と実行から取る（→「接地の非対称」節）。

---

## 2. AI コーディングの特徴

このワークフローは AI をコーディングの**主体**として扱う。人間が指示を与え AI が部分的に補助するのではなく、AI が辞書・テストツリー・実装を一貫して構築する。人間は問題定義とフィードバックによって現実との接地を担う。

この設計が成立するためには、AI の認知のあり方を踏まえる必要がある。

### AI の認知の特徴

AI は記号から記号を生む装置だ。入力テキストをもとに続く記号を生成する。これは強力だが、二つの構造的な特徴がある。

**接地の欠如**: AI それ自体は現実に接地していない。確信に満ちた記号を、接地の有無に関わらず同じ文法で出す。AI が「この実装は正しい」と書いても、それは記号的な整合性の産物であり、現実での検証の産物ではない。

**セッションスコープ**: AI はセッションをまたいで記憶を持たない。一つのセッションの中では文脈を保持するが、累積的な理解の深まりが起きない。また、対話相手の枠組みに合わせる傾向があり、設計を前進させる「抵抗」を出し惜しむリスクを持つ。

### コードへの現れ方

これらの特徴はコードの構造に現れる。AI がコードを書くとき、典型的には「問題を解くための道具（関数）をつくり、道具を組み合わせて問題を解く」という流れになる。関数に名前は付くが、それらを組み合わせた全体には名前が付かない。問題を直接解こうとするため、全体を先に構想しないまま部品が積み上がる。結果として、コードは関数の平べったいリストとして存在し、各関数が全体の中でどう協調するかがコードの外から見えにくい。

### この設計が取る方針

**接地は人間と実行から取る。** 問題定義（problem.md）と実際の使用（feedback）が現実との接触点になる。走るテストが最も直接的な接地点だ。

**全体・部分の関係をコードに持ち込む。** AI の作業に先立って全体の構造（テストツリー）と意味空間（辞書）を用意する。AI はこの構造の中でコードを構成するため、生成される関数は全体の中での位置を持つ。

この補い方の理論的根拠が、次節のポランニーの枠組みだ。

## 3. ポランニーの暗黙知

Michael Polanyi（*The Tacit Dimension*, 1966）の暗黙知の枠組みを使う。

- **近位項（proximal）**: 「から知る」もの。意識の焦点から外れて身体に吸収される。
- **遠位項（distal）**: 「として知る」もの。近位項を通じて了解される対象。
- **包括（comprehension）**: 近位項を通じて遠位項を了解する働き。

具体例で言えば、車の運転がこれにあたる。運転に習熟した人は、ハンドルやペダルそのものには意識を向けていない。手がハンドルをどう握っているか、足がどれだけペダルを踏み込んでいるか——こうした操作（近位項）は意識の焦点から外れ、身体に吸収されている。意識が向いているのは、フロントガラスの向こうに広がる道と、その先へ続く進路（遠位項）である。運転者はハンドル操作という近位項を「通じて」、車を道に沿って走らせることを了解している。この「通じて了解する」働きが包括である。

ここで近位項そのものに意識の焦点を移すとどうなるか。ハンドルの握り方に注意を向けた瞬間、道の流れを見失う。近位項は焦点を当てるものではなく、焦点を当てる先（遠位項）を支えるために背景へ退いていなければならない。焦点を近位項へ引き戻すと、包括的な全体が崩れる。

AI のコーディングが近視眼に陥るとき、起きているのはこれと同じことである。個々の関数の振る舞い——ハンドル単体の動き（近位項）——だけに意識の焦点が当たり、それらを通じて了解されるべき全体（遠位項）が視野から落ちる。関数は動くが、それが何のための部品なのか、全体の中でどう協調するのかという遠位項がない。前節で述べた「コードが関数の平べったいリストとして積み上がる」状態は、近位項しか見えておらず、包括的な全体を欠いた姿である。

## 4. テストツリー

テストツリーは、性質の異なる二つの部分・全体関係を同時に担う。一つは客体（機械・コード）の側、もう一つは主体（AI エージェントの認知）の側である。両者は別のことを語っているが、ともに部分・全体という同じ形を持ち、互いに照応する。テストツリーは、この照応を一つの構造として成り立たせる装置である。

### 客体の側：分解と合成

テストをツリーとして捉えるとは、**全体の分解と要素の合成という逆操作の対**として見ることだ。

このワークフローでは、まず「問題を解く機械 X」を構想し、X に名前を付ける（「X は xxx ができる」）。X の名前は problem.md の「できない」に対応する——X が存在すれば問題が解消する。次に、X が備えるべき要件を段階的に分解してツリーを構築する。このトップダウンの操作がテストツリーの構築だ。実装はその逆を辿る——末端の要素から実装し、合成して X に至る。

```
X は xxx ができる   ← ルートノード（問題解決の仮説）
├── A は aaa ができる
│   ├── D は ddd ができる
│   └── E は eee ができる
└── B は bbb ができる
```

実装中に合成の形が変わったとき、ルートを更新する。ツリーの不変量はルートではなく problem.md にある問題の記述。ツリーはセミラティス状になる（ノードが複数の親を持つ）。共有ノードは命名の契機——その共有に名前が与えられることで、合成の意味が明示される。

名前を付けることがこの操作の出発点になる理由はポランニーの枠組みで説明できる。X に名前が与えられると、X は「掴める対象」として扱えるようになる——分解の出発点として、また合成の目標として。名前がなければ X はぼんやりした問題意識のままで、段階的に分解する対象にならない。

こうして構築されたテストツリーは、テストファイルそのものになる。`describe()` のネストがノードの階層を、`it()` がノードの葉を表す。このフローで実装されるコードは、はじめから全体・部分の関係を持ったものになる——作られる関数は X のための部品であり、部品の意味は X との関係から与えられる。

### 主体の側：近位項・遠位項・包括

同じツリーは、それを書く主体の認知の側からも見える。E のコードを書くとき、ツリーの中での位置——E が A を支え、A が X を支える——が常に文脈として存在する。E（近位項）を操作しながら X（遠位項）を了解している状態。これが包括である。

近位項と遠位項は固定した役ではなく、焦点の位置によって入れ替わる。各ノードは、子に対しては遠位項（子を通じて了解される対象）であり、親に対しては近位項（それを通じて親を了解する手がかり）である。ツリーはこの入れ子になった「から知る／として知る」の連鎖を構造として持つ。

### 照応

客体の側の部分・全体関係（分解・合成）と、主体の側の部分・全体関係（近位項・遠位項・包括）は、テストツリーという一つの構造の上で重なる。コードが部分・全体の構造を持つと同時に、それを書く認知の側にも対応する部分・全体の構造がある。テストツリーが近位項と遠位項の関係を構造として持ち、辞書がその「了解」の語彙を固定する。これが内側3層の設計の根拠である。

### ノードの二層

- **ルートノード（受け入れ確認）**: 系全体が problem.md の「できない」を解消するかを確かめる。最初に書いて red のまま置き、全ノードの実装後に green になることを確認する。
- **中間・末端ノード（振る舞いの宣言）**: 「A は aaa ができる」という形で、要素のインターフェースを実装より先に定める。インターフェースと実装の分離がここで起きる。これが TDD の駆動力の本体。

### 命名と振る舞いの客体化

名前のない振る舞いはコードの中に埋もれる。処理は走っているが、その「何か」を指し示す手段がない。AI にとってこれは「どう書いてあるか」の違いに過ぎず、処理を記述するのではなく、その処理が体現している概念について語ることができない。

名前を付けると事態が変わる。名前を持つ概念は、指し示せるものになる——分解の対象として、要件を問う対象として、他の概念との関係を語る対象として。セクション3の言葉で言えば、名前を持った概念が近位項として操作できるようになる。埋もれていた振る舞いが客体化され、扱える対象になる。

名詞と動詞の非対称性がここに現れる。動詞（`matchTemplate`）は操作に貼ったラベルで、等価な操作に置き換えられる余地がある。名詞（`テンプレートマッチャー`）は概念の存在宣言で、操作への還元を遮断する。概念をコードから分離して語れるのは、名詞として立っているからだ。

ノード名を日本語で書くのは、概念名と実装識別子（関数名・クラス名）を混同しないためだ。言語が異なれば、AI が問題定義から実装を先読みしてそのまま識別子をノード名に使うことを防げる。

### 「テストが書けない」はシグナル

「テストが書けない」が実装フェーズで出てきたとき、それは `describe()` の抽象レベルが実装手段に引っ張られているシグナルだ。省略ではなく `describe()` を問い直す起点にする。述語に技術名（localStorage, fetch, SQL 等）が入った瞬間、そのノードは概念ではなく実装手段の記述になっている。

## 5. 辞書

辞書は AI の行動可能空間を定義するガードレールだ。事前に定義された語彙の中でテストツリーを構成することで、AI はプロジェクト固有の言葉でノードを埋める。

辞書は人間が一方的に決めるのではなく、AI がコードベースの調査と実装の中で構成・更新する。語彙にない概念が必要になったときは wip に登録してから使う——禁止ではなく制御された拡張。

### 指示の連鎖

辞書は指示の連鎖の起点でもある:

```
外部の現実 ← 辞書（概念） ← テスト（describe/it） ← 実装コード
```

各層は下の層の指示対象。辞書はドメインの現実を指し示し、テストは辞書の概念を指し示し、実装はテストの「できる」宣言を指し示す。ルートノードが辞書の概念と対応している限り、末端のコードは現実と接続されている。

### ネットワーク構造

語彙は互いにリンクしてネットワークをなす。定義が他の概念を参照することでリンクが生まれる。孤立した概念（何も参照せず、どこからも参照されない）はアドホックな追加の兆候。体系性はこのリンク構造が担保する。

辞書のネットワーク構造はセクション4のセミラティスとも対応している——辞書の概念がテストツリーのノードとして現れ、ひとつの概念が複数の「できる」の支えになるとき、その概念は独立したサブドメインの候補になる。

### 二層管理

- `docs/dictionary.md`（安定）: ユーザーが受け入れた概念
- `plans/<project>/dictionary.md`（作業仮説）: 実装中に発見された概念。フィードバックで照合・昇格

## 6. その他の設計判断

### フェーズ分離（problem / run / refactor / feedback）

問題定義と解決案を同じ場で書くと、解決手段が問題の記述に紛れ込む（「正規化が必要」は既に解決の言葉）。使用の事実とその解釈を同じ場で書くと、接地した事実と作り手の作文が混ざる。

- **problem** — 「できない」状態の記述。解決手段は書かない。
- **run** — 辞書・テストツリー・実装を一貫して行い、利用可能な状態にする。
- **refactor** — 複数の run の後、全コンテキストを横断して「理解とコードのズレ」を観察し転写する。個別の run では見えなかった構造が見える。
- **feedback** — 実際に使い、事実と印象を集め、利用仮説と照合する。

なぜ design を独立フェーズにしないか: 分解と合成は実装の責務であり、実装しながら合成の形が見えてくる。design フェーズを分離すると、実装から設計への逆流（実装中の発見が設計を更新する）が起きる場所が失われる。

### 問題と解の共進化

Horst Rittel と Melvin Webber（1973）が示したように、設計の問題の多くは wicked problems であり、**解を試みることによってしか問題は理解できない**。

problem.md はスナップショット——現時点の問題理解を言語化したバージョン付きの仮説。run / feedback が「問題の見方が変わった」と示したとき problem.md は改版される。これが正常な軌道。

### 接地の非対称（AI は差異の引き出し役）

AI は記号から記号を生む装置で、それ自体は現実に接地していない。接地は、人間が実際に使うことと、走るテスト・コードから来る。

AI 固有のリスク:
- 確信に満ちた記号を、接地の有無に関わらず同じ文法で出す
- 相手の枠に合わせる傾向があり、対話を生産的にする抵抗を出し惜しむ

帰結: **接地と抵抗は人間と実行から取り、AI は差異の源ではなく引き出し役に置く。**「できる」のツリーの red は、状況が反論している姿であり、最も直接的な接地点。

### 設計は対話（差異・抵抗がエンジン）

Evans の deep model への breakthrough は、ドメインエキスパートとの対話で自分の理解のズレが露呈したことが契機だった。Donald Schön の reflective conversation with the situation: 設計者が一手を打つと状況が talk back し、枠を reframe する。**失敗するテストは、状況が反論している姿。**

対話は持続しない（終われば蒸発する）。効くのは暫定的な言語化と、それへの抵抗の摩擦。対話には必ず沈殿（持続する文書への書き戻し）を伴わせる。辞書はこの沈殿の一形態。

deep model は既存概念の細分化ではなく再編成——小さなパラダイム転換。手続き化できるのは「再編成を試みる契機」だけ。このスキル群は、直線フローに加えて契機（feedback / vocab による再定義）を持つ。

### 緑 ≠ 完了

TDD の核は red/green の適合ではなく、知識を獲得するプロセス。コードは使用で清算される仮説であり、green は局所チェックポイントにすぎない。run の完了は「テスト緑・関数実装」ではなく、**利用可能な状態**（実アプリで動く）。

### フィードバック（一人シミュレーション）

このスキル群は実地ユーザーへの適合を測る装置ではなく、開発フロー全体をシミュレートする装置。ユーザーが一人で回し、その中で AI に役（run では実装者、feedback ではインタビュアー）を割り当てる。

実装者は設計にコミットしている。インタビュアーはコミットしていない。同じ人が続けると作った本人が自分の仕事をかばうため、別セッションで役を分ける。利用仮説を使う前に立てること（甘い採点を止める）、事実を先に集めること（何をしたか・出力）が作法の骨格。
