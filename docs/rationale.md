# なぜこの形か（設計の考え方）

**初版:** 2026-06-10
**出典:** `docs/notes/2026-06-10-design-as-dialogue.md`（この文書を生んだ対話）

この文書は「どう使うか」ではなく「**なぜこの形なのか**」を残す。スキルを*使う*ときは要らない。
スキルを**再設計する**ときに読む。結論だけでなく、その背後の思考枠組みを更新するための起点。

---

## 0. 前提となる立場

ソフトウェアをつくるとは、問題を解くための道具（関数・オブジェクト・サービス・データ構造）を
案出する行為。設計とは、問題を小問題へ分解し、各小問題を解く道具を導入し、道具どうしを繋いで
全体の解決にする、というアイデアの提示。これを*プログラマに通じる平易な言葉*で書く。

## 1. フェーズ分離（problem / design / run / feedback）

問題定義と解決案を同じ場で書くと、解決手段が問題の記述に紛れ込む（「正規化が必要」は既に解決の
言葉）。設計でテストまで書くと、設計が「どう測るか」を先回りしてテストを書きすぎる。使用の事実と
その解釈を同じ場で書くと、接地した事実と作り手の作文が混ざる。だから責務を分ける:

- **problem** — 何が適切か。問題の言葉だけ。
- **design** — どう解くか。小問題への分解・道具立て・受け入れ条件・利用仮説。テストは書かない。
- **run** — 受け入れ条件を検証するテストを設計・実装し、テスト駆動で実装。利用可能な状態まで。
- **feedback** — 実際に使い、事実と印象を集め、利用仮説と照合。ズレが次のループを開く。

分けるのは「混同を防ぐ」だけでなく、**コミットと保留の緊張**を扱うため。分解-解決は前に進むため
*枠へのコミット*を要求し、深いモデルの発見は*正しい概念が現れるまで保留*を要求する。両方が要る。

## 2. 問題は「力」、解決は「道具」（Alexander の前期と、その批判）

problem.md は問題を「そこに働く独立した力・条件（forces）」として書く。これは Christopher
Alexander が生涯手放さなかった見方（パターンも「文脈における力の系を解決する配置」）。

design.md は問題を小問題へ分解し、各ノードに道具を当てる。これは Alexander 前期
（*Notes on the Synthesis of Form*, 1964）の、misfit 変数をツリーに分解して各部分を解く方法に近い。
Alexander はこれを後に否認した（*A City is Not a Tree*, 1965 の semilattice 批判、1971年版序文の
方法そのものの放棄）。理由: 良い形は問題ツリーを解いて*合成*されるわけではない。

ではなぜ design の分解を採るのか。**分解は緊張がどこにあるかを写し取る診断であって、解を含意
しない**から。やかんの「ハンドルだけ低熱伝導材」という解は要求ツリーから生成されない——全体を
見たときの創造的な一手。分解は*生成の方法*ではなく*診断と記述*として使う。実際、良い design は
ノードと解が一対一にならず、共有変数や文脈分岐で結合した semilattice になる（それでよい）。

**深いモデル（Evans の deep model）は、既存概念の細分化ではなく再編成**＝小さなパラダイム転換。
分解は定義上これを生めない。そして再編成は手続き化できない。**手続き化できるのは「再編成を試みる
契機」だけ**（後期 Alexander の unfolding も、アルゴリズムではなく「全体を見て次の一手を選ぶ」
規律）。だからこのスキル群は、分解-解決の直線フローに加えて、契機（feedback／再設計）を持つ。

## 3. 設計は対話（差異・抵抗がエンジン）

概念の洗練は、直線的な分解からではなく、**対話の中で生まれる**。Evans のシンジケートローンの
breakthrough は、ドメインエキスパートとの対話で*自分の理解のズレ*が露呈したことが契機だった。
エンジンは事例の帰納ではなく、**視点どうしの不一致の検出と探索**（Bateson「差異を生む差異」）。

Donald Schön の *reflective conversation with the situation*: 設計者が一手を打つと状況が talk back し、
それを受けて枠を reframe する。対話相手は人間とは限らず、*状況・素材*が反論してくる。
**失敗するテストは、状況が反論している姿**。Evans のエキスパート、赤いテスト、feedback の不一致は、
どれも「異なる理解を突きつける他者」という点で同じ。

ただし、対話は媒体としては最も持続しない（終われば蒸発する）。効くのは「対話という形式」ではなく
*暫定的な言語化と、それへの抵抗の摩擦*。だから対話には必ず**沈殿**（持続する文書への書き戻し）を
伴わせる。さもないと「積み上がらない感覚」が会話の中へ移動するだけになる。

## 4. 接地の非対称（AI は差異の引き出し役）

AI は記号から記号を生む装置で、それ自体は現実に接地していない（記号接地問題は未解決）。接地は、
**人間が実際に使い・検証すること**と、**走るコード／テスト**から来る。しかも接地は*委譲*であって
*解決*ではなく、人間が実際に触ったスライスだけが接地し、触っていない部分は未接地のまま*接地済みと
見分けがつかない*まま残る。

ここから、AI 固有のリスクは「未接地であること」ではない（Evans も対象には未接地だった）。残るのは
*程度と傾向*の差: (1) AI は確信に満ちた記号を、接地の有無に関わらず同じ文法で出す（賭けを事実の
文法で出す）。(2) 相手の枠に合わせる傾向があり、対話を生産的にする抵抗を出し惜しむ／捏造しやすい。

帰結: **接地と抵抗は人間と実行から取り、AI は差異の*源*ではなく*引き出し役*に置く**。
AI が出す reframe は*事実ではなく検証すべき仮説*として出し、人間かテストが接地する。
（AI↔AI の対話は未接地どうしで偽の差異が出やすい。候補出しには使えるが接地源にはならない。）

## 5. TDD は知識獲得（緑 ≠ 完了）

TDD の核は red/green の適合そのものではなく、**知識を獲得するプロセス**。Kent Beck の
*triangulation*（例が2つ揃うまで抽象を確定しない）は、抽象＝モデルを具体例の蓄積から*析出*させる
帰納の装置で、深いモデル発見の小さな機構。Ward Cunningham の技術的負債も原義では「いまの理解を
反映したコードを出荷し、理解が深まったらリファクタで*新しい理解をコードへ転写*する」——利子は
「理解とコードのズレ」の摩擦。

だから **コードは使用で清算される仮説**であり、緑は局所チェックポイントにすぎない。run の完了は
「テスト緑」「関数実装」ではなく、**利用可能な状態**（入口に組み込み、実アプリで動く）。
これは設計に明記がなくても常に run の目標に含まれる（「動くべき」は最低ライン）。

## 6. 利用フィードバック（一人シミュレーション）

このスキル群は実地ユーザーへの適合を測る装置ではなく、**開発フロー全体をシミュレートする**装置。
ユーザーが一人で回し、その中で Claude に役（run では開発者、feedback ではインタビュアー）を割り
当てる。デプロイを含まないので、ユーザーはほぼ本人で固定。

役割分離は飾りではなく**構造上の仕掛け**: 開発者役は設計にコミットしている。インタビュアー役は
コミットしていない（むしろ壊しにいく）。同じ人が続けてやると作った本人が自分の仕事をかばうので、
**別セッション**で役を分け、設計に肩入れしない視点を取る。一人二役でも、役さえ守れば差異は本物。
ただし出せるのは「本人の使用と判断に対する不適合」まで（別の人の知識が要るズレは原理的に出せない）。

このフェーズの作法:
- **利用仮説を使う前に立てる**（design が「使ったらこうなるはず／外れ＋試す操作」を残す）。
  使う前にコミットした仮説が、自作自演の共謀（甘い採点）を止める。use レベルの「テストファースト」。
- **事実を先に集める**（何点・どのテンプレ・アスペクト比・出力）。印象だけでは再現も解釈もできない。
- **印象の解像度を上げる**。「デカい」を、場所特定・軸を複数差し出す・対比・結果接地、で
  *ユーザー自身に*ほどいてもらう。解釈の主体はユーザー（自分の印象に特権的にアクセスできる）。
  Claude は解釈を代行せず、解像度を上げる質問に徹する。
- 出力は `findings.md` 一枚に、**事実の節**と**解釈・原因の仮説の節**を分けて書く。
  findings は削除せず、反映後は対応バージョンを明記して `history/` にアーカイブする。

良いインタビューは、生の印象（「全部デカい」）から problem.md の力（図と地／相対サイズ）をそのまま
析出させる。これが feedback → findings →（必要なら）problem/design の問い直しという loop の実体。

## 7. 二モード（探索 ⇄ 沈殿）と、このリポジトリ自身

直すべきは「文書を非線形にする」ことではない。文書は本来、線形な*沈殿物*。必要なのは:
- **発散的な対話（探索）** — 概念を発見する。手続き化できない。
- **収束的な沈殿（固定）** — 記録し検証する。problem/design/test。

の二つを明示的に**交替**させること（一回こっきりではなく再帰的に回す）。

このリポジトリの文書構造自体が、それを実体化している:
- `docs/rationale.md` ＝ 持続する概念層
- `docs/notes/` ＝ 探索の沈殿
- `CLAUDE.md` ＝ 再設計時にそれを読み直させる契機
