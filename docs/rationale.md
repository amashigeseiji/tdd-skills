# なぜこの形か（設計の考え方）

**初版:** 2026-06-10
**改訂:** 2026-06-14
**出典:** `docs/notes/2026-06-10-design-as-dialogue.md`, `docs/notes/2026-06-13-interaction-system-and-skill-restructure.md`

この文書は「どう使うか」ではなく「**なぜこの形なのか**」を残す。スキルを*使う*ときは要らない。
スキルを**再設計する**ときに読む。結論だけでなく、その背後の思考枠組みを更新するための起点。

## 0. 前提となる立場

ソフトウェアをつくるとは、**全体と部分の関係を持った構造を生み出す**行為。
関数・オブジェクト・サービス・データ構造は要素（部品）であり、要素を列挙することが目的ではない。
要素が協調してひとつのことを成し遂げるとき、その協調全体に名前を与えることが設計の核心。

この「複数の要素が協調してひとつのことを成し遂げる全体」を**インタラクションの系**と呼ぶ。
設計とは系の名前を与え、系を支える要素を案出し、要素どうしの協調を示すこと。
そして問題とは「できない」状態であり、実装とはその「できない」を「できる」に変換すること——
この観点では、系の名前は「何が『できる』ようになるか」という問いへの答えになる。

## 1. フェーズ分離（problem / run / feedback + vocab）

問題定義と解決案を同じ場で書くと、解決手段が問題の記述に紛れ込む（「正規化が必要」は既に解決の
言葉）。使用の事実とその解釈を同じ場で書くと、接地した事実と作り手の作文が混ざる。
だから責務を分ける:

- **problem** — 何が適切か。問題の言葉だけ。解決手段は一切書かない。
- **run** — 何を実現するか（系の定義）と、どう実装するか（分解・テスト・合成）を一貫して行う。
  語彙定義・テストツリーの構成・実装・統合まで、利用可能な状態にする。
- **feedback** — 実際に使い、事実と印象を集め、利用仮説と照合。ズレが次のループを開く。
- **vocab** — 語彙ネットワークの管理。他のフェーズから呼ばれ、語彙の追加・整合性確認・昇格を行う。
  フェーズではなくインフラ。

分けるのは「混同を防ぐ」だけでなく、**コミットと保留の緊張**を扱うため。分解-実装は前に進むため
*枠へのコミット*を要求し、深いモデルの発見は*正しい概念が現れるまで保留*を要求する。

**なぜ design を独立フェーズにしないか。**
「設計」と「実装」を分けると、分解はすでに設計フェーズの成果物として存在し、実装者はそれを受け取る
だけになる。しかし分解と合成は実装の責務であり、実装しながら合成の形が見えてきて初めて系の真の
姿が分かる——これが逆流（実装から設計への更新）の実体。分解を事前に固定すると、この逆流が起きる
場所が失われる。だから tdd-run が系の定義から分解・合成まで一貫して行う。

## 2. ポランニーの近位項・遠位項と「できる」のツリー

Michael Polanyi（*The Tacit Dimension*, 1966）の暗黙知の枠組み：
- **近位項（proximal）**: 「から知る」もの。意識の焦点から外れて身体に吸収される。
- **遠位項（distal）**: 「として知る」もの。近位項を通じて了解される対象。
- **包括的理解（comprehend）**: 近位項を通じて遠位項を了解すること。

AIコーディングの問題: コードを書くとき、個々の関数（近位項）しか見えておらず、それを通じて了解
されるべき「何か」（遠位項）が見えていない。コードは「関数の平べったいリスト」として認識され続ける。

**「できる」のツリーがこれを解く。**

```
X は xxx ができる   ← ルートノード（問題解決の仮説）
├── A は aaa ができる
│   ├── D は ddd ができる
│   └── E は eee ができる
└── B は bbb ができる
```

このツリーはテストファイルそのものである。E のテストに対応するコードを書くとき、
ツリーの中での位置——E が A を支え、A が X を支える——が常に文脈として存在する。
E（近位項）を操作しながら X（遠位項）を了解している状態、これが包括的理解。

ツリーはセミラティス状になる（ノードが複数の親を持つ）。共有ノードは命名の契機——
その共有に名前が与えられることで、合成の意味が明示される。

ルートノードの「できる」が問題（「できない」）の解消に対応している。
実装中に合成の形が当初の想定と異なると気づいたとき、ルートを更新する。
その際、**problem.md の「できない」が解消されているかを照合する**。
ツリーの不変量はルートではなく problem.md にある問題の記述。

## 3. 問題定義の限界と共進化

Michael Jackson の "World and the Machine"（*Problem Frames*, 2001）は、問題と解決の分離に
最も明晰な形式を与える。「世界（world）」= ソフトウェアが置かれる環境、「機械（machine）」=
ソフトウェア本体。要件は世界の記述であり、仕様は機械のインターフェース記述。
これが problem.md と run の分離の直接的な根拠になる。

しかし、Horst Rittel と Melvin Webber（1973）が示したように、設計の問題の多くは "wicked problems"
であり、数学的な "tame problems" と本質的に異なる。**解を試みることによってしか問題は理解できない。**
解の候補を動かすたびに問題の見え方が変わる。

これは problem → run の一方向フローへの問いに見えるが、逆に言えば feedback が problem.md を
改版するループの**必然性**を説明する。ループは例外的な修正ではなく、予測される軌道だ。

Dorst と Cross（2001）の観察では、設計者の実際の思考は**問題空間と解空間の共進化（co-evolution）**。
問題を理解してから解を探すのでも、解から問題を定義するのでもなく、両者が交互に更新されながら前進する。
推論の形は**アブダクション**（Dorst）: 「望ましい結果 ＋ 作動原理 → 問題の再定義」という向き。

帰結: **problem.md はスナップショット**。現時点の問題理解を言語化したバージョン付きの仮説。
run / feedback が「問題の見方が変わった」と示したとき problem.md は改版される——これが正常な軌道。

## 4. 語彙と行動可能空間

Coplien（*Multi-Paradigm Design*, 1998）はドメインを二つに分ける:
- **アプリケーションドメイン**: ユーザーの関心が寄せられている知識の体系。現実世界に根ざす。
- **ソリューションドメイン**: 実装者の関心。アプリケーションドメインの概念を表現するための素
  （OOPのClass、RDB、関数型の関数など）。

設計の目標のひとつは、ソリューションドメインの語彙でアプリケーションドメインが**無理なく**
表現できるようにすること。「無理なく」という感覚のズレが、設計の不具合のシグナルになる。

**語彙セットはAIの行動可能空間を定義するガードレール。**
事前に定義された語彙の中で「できる」のツリーを構成することで、AIは学習データ由来の汎用的な
概念ではなく、このプロジェクト固有の言葉でノードを埋める。
語彙にない概念が必要になったときは wip に登録してから使う——これは禁止ではなく制御された拡張。

語彙は互いにリンクして**ネットワーク**をなす。定義が他の概念を参照することでリンクが生まれる。
孤立した概念（何も参照せず、どこからも参照されない）はネットワークに統合されていない兆候で、
アドホックな追加の可能性がある。体系性はこのリンク構造が担保する。

語彙は二層で管理する:
- `docs/dictionary.md`（安定）: ユーザーが受け入れた概念
- `docs/wip/dictionary.md`（作業仮説）: 実装中に発見された概念。仮のガードレールとして機能し、
  受け入れ後に stable に昇格する

## 5. 設計は対話（差異・抵抗がエンジン）

概念の洗練は、直線的な分解からではなく、**対話の中で生まれる**。Evans のシンジケートローンの
breakthrough は、ドメインエキスパートとの対話で*自分の理解のズレ*が露呈したことが契機だった。
エンジンは事例の帰納ではなく、**視点どうしの不一致の検出と探索**（Bateson「差異を生む差異」）。

Donald Schön の *reflective conversation with the situation*: 設計者が一手を打つと状況が talk back し、
それを受けて枠を reframe する。**失敗するテストは、状況が反論している姿。**
Evans のエキスパート、赤いテスト、feedback の不一致は、どれも「異なる理解を突きつける他者」。

ただし、対話は媒体としては最も持続しない（終われば蒸発する）。
効くのは「対話という形式」ではなく*暫定的な言語化と、それへの抵抗の摩擦*。
だから対話には必ず**沈殿**（持続する文書への書き戻し）を伴わせる。
vocabulary.md はこの沈殿の一形態——対話から析出した概念を固定する。

**深いモデル（Evans の deep model）は、既存概念の細分化ではなく再編成**＝小さなパラダイム転換。
分解は定義上これを生めない。手続き化できるのは「再編成を試みる契機」だけ
（後期 Alexander の unfolding も、アルゴリズムではなく「全体を見て次の一手を選ぶ」規律）。
このスキル群は、直線フローに加えて契機（feedback / vocab による再定義）を持つ。

## 6. 接地の非対称（AI は差異の引き出し役）

AI は記号から記号を生む装置で、それ自体は現実に接地していない。接地は、
**人間が実際に使い・検証すること**と、**走るコード／テスト**から来る。

AI 固有のリスク: (1) 確信に満ちた記号を、接地の有無に関わらず同じ文法で出す。
(2) 相手の枠に合わせる傾向があり、対話を生産的にする抵抗を出し惜しむ。

帰結: **接地と抵抗は人間と実行から取り、AI は差異の*源*ではなく*引き出し役*に置く。**
「できる」のツリーの red は、状況が反論している姿であり、最も直接的な接地点。

## 7. TDD は知識獲得（緑 ≠ 完了）

TDD の核は red/green の適合そのものではなく、**知識を獲得するプロセス**。

「できる」のツリーのノードには二つの層がある:
- **ルートノード（受け入れ確認）**: 系全体が問題（「できない」）を解消するかを確かめる。最初に
  書いて red のまま置き、全ノードの実装が終わったとき green になることを確認する。
- **中間・末端ノード（振る舞いの宣言）**: 「A は aaa ができる」という形で、要素のインターフェースを
  実装より先に定める。インターフェースと実装の分離がここで起きる。これがTDDの駆動力の本体。

これは GOOS（*Growing Object-Oriented Software, Guided by Tests*, Freeman & Pryce, 2009）の
二重ループ——外側のacceptance test loopと内側のunit test loop——に対応するが、
「できる」のツリーでは両者が同じ構造の異なる階層として統一される。

Ward Cunningham の技術的負債の原義: 「いまの理解を反映したコードを出荷し、理解が深まったら
リファクタで*新しい理解をコードへ転写*する」——利子は「理解とコードのズレ」の摩擦。
**コードは使用で清算される仮説**であり、緑は局所チェックポイントにすぎない。
run の完了は「テスト緑」「関数実装」ではなく、**利用可能な状態**（入口に組み込み、実アプリで動く）。

## 8. 利用フィードバック（一人シミュレーション）

このスキル群は実地ユーザーへの適合を測る装置ではなく、**開発フロー全体をシミュレートする**装置。
ユーザーが一人で回し、その中で Claude に役（run では実装者、feedback ではインタビュアー）を割り当てる。

役割分離は構造上の仕掛け: 実装者は設計にコミットしている。インタビュアーはコミットしていない。
同じ人が続けてやると作った本人が自分の仕事をかばうので、**別セッション**で役を分け、設計に
肩入れしない視点を取る。一人二役でも、役さえ守れば差異は本物。

このフェーズの作法:
- **利用仮説を使う前に立てる**（problem.md の「解決したと言える状態」がその素材）。
  使う前にコミットした仮説が自作自演の共謀（甘い採点）を止める。
- **事実を先に集める**（何をしたか・出力）。印象だけでは再現も解釈もできない。
- **印象の解像度を上げる**。「デカい」を、場所特定・軸を複数差し出す・対比・結果接地で
  *ユーザー自身に*ほどいてもらう。Claude は解釈を代行せず、解像度を上げる質問に徹する。
- 出力は `findings.md` 一枚に、**事実の節**と**解釈・原因の仮説の節**を分けて書く。

## 9. 二モード（探索 ⇄ 沈殿）と、このリポジトリ自身

- **発散的な対話（探索）** — 概念を発見する。手続き化できない。
- **収束的な沈殿（固定）** — 記録し検証する。problem / test / dictionary。

二つを明示的に**交替**させること（一回こっきりではなく再帰的に回す）。

このリポジトリの文書構造自体が、それを実体化している:
- `docs/rationale.md` ＝ 持続する概念層
- `docs/dictionary.md` ＝ 持続する語彙層（対話から析出した概念の固定）
- `docs/notes/` ＝ 探索の沈殿
- `CLAUDE.md` ＝ 再設計時にそれを読み直させる契機
