# tdd-run の analysis / compose 分割案

**日付:** 2026-06-30

## 提案の概要

tdd-run を分解・分析フェーズ（analysis）と実装フェーズ（compose）に分割し、タスクの複雑さに応じて処理を変える構想。

```
tdd-run analysis → レポートファイル出力 → 人間の確認 → tdd-run compose
```

### 複雑さの3段階

- **軽い**: 既存のテストに乗っかるだけ。新規テストツリーは作らない
- **普通**: 現状の tdd-run と同様。ツリーを1つ構成する
- **重い**: 複数の観点（OO中心・イベント中心・データ中心など）でツリーを構成し、比較して結論を出す。サブエージェントを並列起動して実行

---

## 評価

### 良い点

**分解と合成の分離が認知的に正しい**
何を作るか（analysis）とどう作るか（compose）は別の操作。現在の tdd-run は両者が混在しており、分離することで各フェーズを深く考えられる。

**レポートファイルが承認ポイントになる**
現状は分解から実装まで一気に流れるため、分解の誤りが後で発覚する。レポートを挟むことで人間が内容を確認してから実装フェーズに入れる。これは rationale.md が述べる「固定点としての人間の役割」とも整合する。

**重いケースでサブエージェントが機能する**
複数のツリーを1つのAIが保持するとコンテキスト消費が大きい。エージェントを並列起動すれば各エージェントが独立してツリーを構築でき、合成エージェントが比較・統合する。Workflow ツールとの相性が良い。

### 懸念点

**通常ケースの摩擦**
80%の問題は「普通」の複雑さなのに、analysis と compose の2コマンドを叩く必要があると負担が増す。`tdd-run` が内部で complexity を評価して分岐し、重い場合だけ2フェーズが明示されるという設計にすれば摩擦を抑えられる。

**「重い」の判断基準が未定義**
analysis の冒頭でこの判定ができないと詰まる。候補となる基準：
- 複数の bounded context をまたぐ
- 辞書に語彙がほぼない新規ドメイン
- 問題定義が曖昧・競合している
- アーキテクチャの決定が長期的な影響を持つ

**compose の複雑化**
重いケースでは compose 自体も複雑になりうる。analysis の複雑さが compose に引き継がれる設計が必要。

## 現状との関係

現在の tdd-run に追加したパターン照合ステップ（Step4末尾）は「分析の強化」の第一歩として位置付けられる。この分割案はその先にある本来の形として整合している。

## 今後の方針

設計コストが大きいため、現行のパターン照合ありの tdd-run を実際に使ってみて「ここが足りない」という具体的なフィードバックが出てからリファクタリングする。実装して気づくことが多い類の設計。

未知のパターンへの取り組みとしては tdd-refactor が適切な場所になりうるという観点とも連動する（`docs/notes/2026-06-30-dictionary-pattern-design-rough.md` 参照）。
