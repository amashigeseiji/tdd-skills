# tdd-vocab のトークン使用量調査

**日付:** 2026-07-01

## 経緯

週次の usage で `tdd-vocab` が11%（1日では14%）を占めていた。ユーザーは `/tdd-vocab` を直接呼んでおらず、主に呼んでいるのは `/tdd-run`（週4%）。直接呼んでいないスキルが呼んでいるスキルより大きい割合を占める理由を調べた。

## 調査方法

セッションのトランスクリプト（`~/.claude/projects/*/*.jsonl`）には各 assistant メッセージに `attributionSkill` フィールドが付いており、そのメッセージがどのスキルの実行コンテキストで生成されたかが分かる。これを集計すればスキル単位のトークン内訳が再現できる。

さらに、`tdd-vocab` に帰属するメッセージについては、同一セッション内で直前に走っていた**別の**スキル（`tdd-vocab` 以外で直近に見えた `attributionSkill`）を記録し、「どのスキルから呼ばれた `tdd-vocab` か」を推定した。

対象はプロジェクト群全体、過去8日分のログ。

## 結果

スキル別トークン使用量（`input + output + cache_creation + cache_read` の合計）:

```
tdd-run       11,526,499 tok (26.8%)  msgs=218
tdd-vocab      8,270,863 tok (19.2%)  msgs=129
tdd-problem    5,065,677 tok (11.8%)  msgs=76
tdd-userstory  4,708,869 tok (11.0%)  msgs=44
tdd-feedback   2,744,824 tok ( 6.4%)  msgs=72
tdd-scaffold   2,623,118 tok ( 6.1%)  msgs=47
```

`tdd-vocab` に帰属するトークンを、直前に走っていたスキルで内訳分解:

```
tdd-run の直後      3,792,004 tok (82.2%)
tdd-scaffold の直後   436,660 tok ( 9.5%)
tdd-feedback の直後   386,756 tok ( 8.4%)
```

**`tdd-vocab` の使用量の8割強は `tdd-run` から呼ばれた `/tdd-vocab plan` に由来し、`tdd-feedback` から呼ばれる `/tdd-vocab promote` は1割弱にとどまる。**

`tdd-feedback` 自体の使用率が1%程度と低いのは、(1) `tdd-run` に比べて起動回数自体が少ない、(2) その少ない起動の中でも `promote` が占める部分はさらに一部、という二重の理由による。

## 原因分析: plan と promote の非対称性

`tdd-vocab/subcmds/plan.md` と `promote.md` を比較すると、手続きの重さに構造的な差がある。

| | `plan`（tdd-run から） | `promote`（tdd-feedback から） |
|---|---|---|
| 作業の性質 | 新しい概念空間をゼロから対話で作る | wip 一覧を機械的にチェックして承認/却下 |
| 手順 | コンテキスト名確定→既存BCとの関係探索→problem.md から概念抽出→**対話して確定**→関係明示（5ステップ） | wip 一覧提示→矛盾確認→承認された分を移動（3ステップ） |
| dict-search 呼び出し | 既存BCとの関係探索、概念ごとの重複確認で複数回 | wip 概念ごとに1回程度 |
| 起動頻度 | **新規プランを始めるたび**に毎回 | プランを閉じるたびに1回 |

`plan` は「概念リストをユーザーに提示し、対話して確定する」（`plan.md` 手順4）という自由度の高い対話ループを持ち、`promote` にはこれに相当する重い対話がない。加えて `plan` は run のたびに（新規プランであれば）走るため、起動回数でも `promote` を上回りやすい。

## 提案

### 1. 「既存への追加」パスを軽量化する

`plan.md` 手順1でユーザーに「新設」か「既存への追加」かを確認している。**既存への追加の場合、既存BCとの関係探索（手順2）の大部分は不要**なはず（すでに属するBCが分かっているため）。現状はこの分岐後も同じ重さの手順2〜5を通っている。「既存への追加」を選んだ場合は手順2をスキップし、「このコンテキストに新しく増える概念だけ」を手順3〜4で拾う軽量パスに分ける。

### 2. dict-search の呼び出しをバッチ化する運用を明記する

`dict-search.js` は複数キーワードの OR 検索に対応しているが、`plan.md` の記述は「キーワードで検索する」としか書かれておらず、逐次呼び出しになりやすい。手順2・手順4の概念確認は、逐一 dict-search を打つのではなく「候補キーワードを洗い出してから一括で1回叩く」ことを明記する。

### 3. plan と promote の重複確認を削減する

`tdd-feedback/skill.md` に既に自己言及の NOTE がある通り、`promote` の「`docs/dictionary.json` の既存概念と矛盾しないか」チェックは、`plan` 時点で書いた `relations` フィールドと本質的に同じ情報を再確認している可能性がある。`plan` で関係を確定した時点の dict-search 結果を `wip` フィールドに記録しておき、`promote` はその記録に差分がないかだけを確認する（ゼロから re-search しない）設計にできないか検討する。

### 4. 効果測定用のスクリプトを残す

今回使った「`attributionSkill` を集計してスキル単位のトークン内訳を出す」手法は、他のスキルの再設計でも「軽くなったか」を検証するのに使える。使い捨てにせず `bin/` 以下にスクリプト化して残すことを検討する（対象プロジェクトパス・期間を引数化する程度でよい）。

## 対応（2026-07-01）

problem.md のフォーマット見直しで、上記の提案とは別に2つの効きどころが見つかり対応した:

- **`コンテキスト:` フィールドの欠落**: `plan.md` 手順1は problem.md の `コンテキスト:` フィールドを
  読む前提だったが、problem.md のフォーマットにこのフィールドがなく、毎回「なければユーザーに問う」
  分岐に落ちていた。problem.md フォーマットに `**コンテキスト:**` フィールドを追加し、ヒアリング中に
  スコープの呼び名が自然に出てきた場合のみ書く運用にした（`tdd-problem/skill.md`）。BC への位置づけ
  判断自体は引き続き plan の仕事。
- **申し送りセクションの未活用**: problem.md の「`## /tdd-run への申し送り`」には既に「語彙定義の種」
  「新規語彙の候補」という抽出済みリストがあるが、`plan.md` 手順3はこれを使わず本文をゼロから解析する
  指示だった。手順3をこのリストを起点にする形に変更した（`tdd-vocab/subcmds/plan.md`）。

提案1（既存への追加パスの軽量化）・2（dict-search バッチ化の明記）・4（効果測定スクリプト）は未対応。

## 残課題

- `tdd-scaffold` の直後に `tdd-vocab` が呼ばれているケース（9.5%）の内実は未調査。`tdd-scaffold/skill.md` には `tdd-vocab` への直接の呼び出し記述がなく、経路が不明。
- 今回の集計は「直前に見えた別スキル」という単純な推定であり、複数の `tdd-vocab` 呼び出しが同一セッション内で連続する場合の帰属精度は検証していない。
- 測定範囲はプロジェクト群の過去8日分のみ。他プロジェクトや長期間での傾向は未確認。
