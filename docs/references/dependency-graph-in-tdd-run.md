# 実装の依存グラフをスキルに組み込む構想

**日付:** 2026-06-30

## 現状の整理

スキルにおいて「実装はグラフである」という認識はあるが、実体としてグラフを持っているのは辞書だけ。

| 対象 | 実体 |
|------|------|
| 辞書グラフ | `dictionary.json` の `relations` フィールド |
| テストツリー | `test-tree.md`（ただし木構造、グラフではない） |
| 実装の依存グラフ | **なし** |

現状の 7.5 ウォークスルーは語彙と実装の対応を `src` フィールドで**点として**記録しているだけで、グラフとしての構造検証ができていない。

## 偽陰性との関係

`tdd-run/skill.md` の "Test and module design" セクションに記述がある：

> Tests for modules isolated from the dependency graph — nodes not participating in composition — produce false negatives.

依存グラフの実体がないため、この記述は概念説明にとどまり、機械的に検出する手段がない。

**孤立ノード**とは、テスト用に作成されたモジュールがテストされているが実際の構成（composition）に組み込まれていないケース。テストは通るが、エントリーポイントから到達できない。

## 対応方針

### ツールの扱い

依存グラフの取得は言語別ツールに委ねる：

- JS/TS: `madge`、`dependency-cruiser`
- Python: `pydeps`
- Ruby など: 対応ツールがあれば使う

**フォールバックとして grep は使わない。** Rails・Nuxt などフレームワークが暗黙に依存解決するプロジェクトでは import を正規表現で拾っても偽の結果になる。ツールがなければこのチェックはスキップする。

### スキルへの組み込み場所

Step 7.5（ウォークスルー）の冒頭に組み込む：

1. 依存グラフツールが使えるか確認（プロジェクトの言語を見てツール存在確認）
2. 使える場合 → グラフを生成し、`src` フィールドに記録された実装ファイルが孤立していないか確認
3. 使えない場合 → スキップ（機械的チェックなし）

### 三つのグラフが揃う状態

- 辞書グラフ（`relations`）
- テストツリー（`test-tree.md`）
- 実装依存グラフ（ツール生成）

この三つが揃って初めて 7.5 で「語彙・テスト・実装」の対応をグラフとして検証できる。

## 残課題

- 対応する言語・ツールの一覧をどこに管理するか
- ツール検出の手順をスキルに書く粒度
- エントリーポイントの定義を明示化（現状「グラフのルート」として暗黙に使っている）
