# 構想: 辞書管理の汎用ツール化（切り出し・配布）

**日付:** 2026-07-07
**出どころ:** dict-write.js（PR #11）実装時の対話。関係タイプ（contains/belongs_to/references）が
スクリプトで暗黙に固定されたことへの拡張性の疑問から、「辞書管理だけでも配布価値があるのでは」
という構想に発展した。
**状態:** 構想のみ。いますぐ着手しない。

## 構想

辞書管理（フォーマット + dict-search.js + dict-write.js + 二層管理）を TDD ワークフローから
切り出し、単体で配布・別ワークフローへ組み込み可能な汎用ツールにする。

核になる価値は二層管理の規律そのもの:
**AI が語彙を提案し（wip）、人間の承認だけが安定化させる（promote → 安定層）**。
ユビキタス言語の管理装置として、要件定義・ドキュメント・他のエージェントワークフローでも使える。

## 現状の結合点の棚卸し

汎用的な核:

- dictionary.json フォーマット（contexts + entries + relations + `#参照` のネットワーク）
- dict-search.js / dict-write.js の読み取り・書き込み分離、検証つき書き込み
- 二層管理と promote の承認規律

TDD 体系への結合点（切り出し時に外部化が必要なもの）:

1. パス規約 — `.claude/tdd/config.json` によるメタレポ探索、安定層 = `docs/dictionary.json`、
   wip 層 = `plans/<プラン名>/dictionary.json`
2. `domain: application | solution | pattern` の3値（TDD ワークフローの理解の三分法）
3. pattern エントリの `heuristic` 必須（tdd-run のパターン照合ステップとの契約）
4. 関係タイプ3種の固定と `contains` の双方向性解釈（dict-write.js にハードコード）
5. `src` フィールド / `@vocab` アノテーション（コードとの接続規約）

## 設計の方向: メタ語彙のスキーマ化

関係タイプの拡張性の議論と、この配布の構想は同じ一手に収束する:

**メタ語彙（domain の列挙・関係タイプとその双方向性期待・層のパス・必須フィールド）を
スクリプトのハードコードからスキーマ/プリセットとしてデータ化し、tdd-skills は
「プリセットの一つ」になる。** 汎用ツール側はスキーマを読んで検証するだけ。

- 関係タイプの拡張は、プロジェクト単位の辞書宣言ではなくスキーマ層で行うのが正しい粒度
  （体系の思考枠組みに属する語彙なので、変更には人間の明示的な行為を要求する点は保たれる）
- format.json（エントリのテンプレート）とスキーマ（検証規則）が一体化する余地がある

## 着手のトリガー / それまでの守り

トリガー候補（どちらかが起きたら着手を検討）:

- 配布・別ワークフロー組み込みの具体的な話が出る
- 4つ目の関係タイプ（または domain 値）が必要になる実例がプロジェクトの feedback から上がる

それまでの守り:

- **dict-search.js / dict-write.js に新しい TDD 結合を増やさない**（パス規約・列挙値の追加は
  スキーマ化を意識して、スクリプト先頭の定数に集約しておく）
- 「使用より先に拡張機構を作らない」（コードは使用で清算される仮説、という体系自身のテーゼに従う）
