# UI相互作用の記述語彙の検討

**日付:** 2026-07-07
**追記:** 2026-07-17
**状態:** 検討がおおむね収束。語彙ドメインとしての実装（tdd-vocab, tdd-run）に進む段階。フォーカス・時間表現は引き続き対象外。

## 背景 / 問題意識

UI を記述するための言語のようなものが欲しい、という問題意識から始まった検討。

きっかけは2つある:

1. **実装側の不満**: Web だと index.html にベタっと書く形になり、コンポーネント化や、コンポーネント間の関係の記述ができない。
2. **より根本の欲求**: 実装以前に、「入力フォームがあり、文字列が入力でき、ボタンを押したらサーバーに送信される」ということを **UI の問題として記述したい**。

2 が本題。1 の不満から React 等のコンポーネント指向の道具を探しても、2 は手に入らない。層が違うため。

## どの層の記述か

「入力フォームがあり、文字列が入力でき、ボタンを押したらサーバーに送信される」という文には、3つの層が混ざっている:

1. **問題の層** — ユーザーは文字列（コメント、検索語など）をシステムに伝えたい。伝えたことが受け取られたと分かってほしい。
2. **相互作用の層** — 「入力できる場がある」「送信という行為ができる」「行為の結果が外界（サーバー）に及ぶ」。
3. **実装の層** — `<input>`、`<button>`、fetch、コンポーネント分割。

記述したいのは 2 の層。HCI ではタスクモデルとアーティファクトの中間にあたり、CAMELEON フレームワークでは「抽象UI（Abstract UI）」と呼ばれる層（タスクモデル → 抽象UI → 具象UI → 最終UI という4段階の2段目）。

### problem.md との関係（注意点）

このリポジトリの鉄則では「入力フォーム」はすでに解決手段の語彙。ただし UI の場合、ユーザーの世界にはアーティファクト自体が含まれる（Norman の「実行の淵・評価の淵」はインターフェースそのものについての問題）。したがって「UI相互作用の記述」は problem.md の層ではなく、**problem と run の中間に一枚ある**と考えるのが整合的:

- problem.md には「文字列を伝えたい・届いたと分かりたい」を書く
- その下に「状態・できる・遷移・効果」の相互作用記述が来る

## 記述語彙の案: 4種類

UI の相互作用は、突き詰めるとほぼ4種類の語で書けるのではないか、という仮説。

| 語彙 | 意味 | 例 |
|---|---|---|
| **状態**（見えること） | 観測可能な状態 | 入力欄は空である／送信中である／エラーを表示している |
| **できる**（アフォーダンス） | ユーザーが取れる行為。条件つき | 送信できる。ただし空でないとき、送信中でないときだけ |
| **遷移**（起きること） | イベント → 状態変化 | 送信すると「送信中」になり、成功で「完了」、失敗で「エラー」 |
| **効果**（外界への作用） | UI の外の世界の変化 | 送信すると、サーバーに文字列が届く |

冒頭の例をこの語彙で書き直すと:

```
要素: テキスト入力
  状態: 空 | 入力あり
  できる: 文字列を入力する（→ 入力あり）

要素: 送信
  できる: 送信する
    条件: テキスト入力が「入力あり」かつ 送信中でない
    効果: サーバーに文字列が届く
  状態: 待機 | 送信中 | 完了 | 失敗
```

構造としては **状態機械（statechart）+ アフォーダンス + 効果**。

## 先行事例

車輪はある程度存在する。発明すべきは形式言語そのものより「問題の言葉で書くための語彙と規約」の方、というのが現時点の見立て。

- **Statecharts（Harel, 1987）** — UI の振る舞いを実装から独立に記述する形式。階層状態・並行状態を持つ。現代の実装が XState。
- **Elm アーキテクチャ（Model / Msg / Update / View）** — 同じ構造の実装側への射影。宣言的 UI（React, SwiftUI）の源流。
- **CAMELEON / UsiXML / ConcurTaskTrees** — HCI 研究側の抽象UI記述の系譜。タスクモデルから最終UIまでの段階を分ける。
- **Gherkin（Given/When/Then）** — シナリオ（一本の線）の記述。構造（網）は書けない。
- **7GUIs** — カウンタ〜スプレッドシートまで7つの定番UI課題集。記述語彙のベンチマークとして手頃。

## このリポジトリとの接続

4語彙は既存構造とほぼ同型:

| UI相互作用の語彙 | 既存の対応物 |
|---|---|
| できる（アフォーダンス） | tdd-run の「できる」のツリー |
| 遷移（イベント→状態変化） | tdd-userstory の Gherkin |
| 効果（外界への作用） | feedback の利用仮説（「使ったらこうなるはず」） |
| 状態（見えること） | 辞書の概念エントリ（application/solution）が持つ `states` フィールド（2026-07-17 追記） |
| 語彙のネットワーク | docs/dictionary.json（`ui` / `ui-pattern` / `design-token` の3ドメインを追加、2026-07-17 追記） |

以前は「要素間の構造的関係」（「このフォームはこの一覧の中にある」「送信が成功すると一覧が更新される」等）が欠けていると考えていたが、2026-07-17 の追記で `ui-pattern` ドメインによって埋まった。詳細は次のセクション。

## 2026-07-17 追記: 語彙ドメインとしての具体化

新しい形式言語や新しいファイルを設計する必要はなく、既存の辞書（`docs/dictionary.json`）にドメインを3つ追加するだけで表現できることが分かった。中間層の置き場所は「新ファイル」ではなく「辞書 + tdd-run のチェックポイント」で決着した。

### 追加する3ドメイン

| ドメイン | 役割 | context | 照合対象 |
|---|---|---|---|
| `ui` | 概念に取り付けられたハンドル、または他の `ui` エントリをまとめる構造的な単位（ページ・領域・タブなど） | あり（属するBCに紐づく） | — |
| `ui-pattern` | 一連の操作を支える構造的パターン（master-detail等）。役割名は既存の `pattern_entry.components`（`{name, role}`）で表現する | なし（横断的） | `ui` エントリ同士の関係。既存の `pattern` ドメインがテストツリーの形と照合するのと同じ機構を、`ui` エントリの関係の形に適用する |
| `design-token` | 「削除系は警告色」のような、意味づけられたトンマナ規約。生の色数値・px値は含めない | なし（横断的） | — |

`ui` は「ハンドル」（概念を操作する窓口。例: 削除ボタンが記事を `references` する）と「器」（他のUI部品をまとめる構造。例: ヘッダーが操作部品を `contains` する）の2つの役割を持つが、同じドメインとして扱う。両方とも「特定のコンテキストに属する名前付きのUI要素」という点で共通しており、器かハンドルかは既存の `relations` の使い分け（`references` で概念を指すか `contains` で子部品を持つか）で区別できるため、ドメインを分ける理由がない。

### 状態（states）

概念エントリ（`application`/`solution` ドメイン）に `states` フィールドを追加できる。状態はその概念が唯一のマスターとして持ち、UI側はその概念を `references` した上で、どの状態のときに関係するかを `relations` の `note` に書く。

```json
{"name": "記事", "domain": "application", "states": ["公開中", "非公開"]}
{"name": "非公開にするボタン", "domain": "ui",
 "relations": [{"type": "references", "target": "記事", "note": "記事が公開中のときのみ表示"}]}
```

UI側に `states` を複製しないのは、同じ状態名の表記ゆれを防ぐため。状態は概念に属するものであり、UIはそれを参照するだけ、という非対称を保つ。

### レイアウト

`domain: "layout"` は不要と判断した。理由:

- ページは「`ui` エントリの集合」として表現できれば十分（ページ自体を `ui` エントリとし、`contains` で子部品を持つ）
- 空間的な配置（左右・上下）は辞書に含めない。master-detail のようなパターンで本当に必要なのは配置そのものではなく**役割の区別**（マスターペイン／ディテイルペインという名前を持つこと）であり、これは `ui-pattern` の `components` フィールドがすでに担える。マスターペインが左にあろうが上にあろうが master-detail は成立する
- 「ヘッダー」「サイドバー」「タブ」のような、空間的なニュアンスを持つ言葉自体を `ui` エントリの名前として使うことは妨げない。ただしそれは辞書が配置を規定しているのではなく、単なる命名の結果

### design-token の初期抽出

`design-token` は問題ごとに発見されるのではなく、プロジェクト全体で一度決めて使い回す性質を持つ。`/tdd-vocab init`（既存コードベースを俯瞰して語彙を抽出する既存の手順）に、CSS/テーマファイルとその使用箇所から意味づけを拾うステップを追加する。まだ存在しない一般的な分類（primary/danger等）をAIが発明することはせず、既存コードにある対応づけを拾い上げ、明らかな欠落だけをユーザーに提示する。

## 開いた論点

- **中間層の置き場所**: 解決。新しいファイルではなく、辞書のドメイン追加（`ui`/`ui-pattern`/`design-token`）と tdd-run ステップ4のチェックポイントで足りる。
- **4語彙で足りるか**: フォーカス・時間（アニメーション、タイムアウト）は依然として対象外。レイアウトは「配置」としては辞書に含めないと決めた（上記参照）。
- **記述の粒度**: `ui` エントリは部品・ハンドル粒度で扱うと決めた（「文字列を与える手段」のような過度な抽象化はしない）。
- **検証可能性**: 未検証のまま。UIに面するノードも既存の「できる」のツリーの中でテストされる想定で、新しい検証機構は今のところ不要と考えている。実装を通じて確かめる。
- **（2026-07-17 追加・保留）コンテキストの再定義**: 「あるコンテキストとは、一つのモデルとそれが取りうる状態の全域である」という考え方（MVCのモデル単位でコンテキストを切る）を検討する余地がある。これが成立すると、状態・UI・語彙境界が一つの構造に統合される可能性がある。ただし現状のコンテキストの粒度（BC単位）を再定義する大きな変更になるため、今回は着手しない。アイデアとして記録するに留める。
